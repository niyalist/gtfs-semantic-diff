# gtfs-semantic-diff の内部構造（前編）：生差分・同定・イベント — 説明台帳のコア

gtfs-semantic-diff は、同一フィードの新旧2世代の GTFS を比較し、変化を
「路線廃止・減便・停留所改称」といった人間が認識できる意味のイベントとして
抽出するツールである。本稿はその内部構造を、実装コードと実データに基づいて
解説する技術記事の**前編**で、コア——生差分の全列挙（L0）、世代間同定（L1）、
便の照合（trip matching）、イベント抽出と説明台帳（L2）——を扱う。
**後編**は、このコアの出力から人が読める HTML レポートを合成する
プレゼンテーション層を扱う。

対象読者は GTFS のデータ仕様（stops / trips / stop_times / calendar の構造、
各種 ID の役割）を把握している技術者。実例は断りのない限り、富山地方鉄道バス
令和8年4月1日改正の実データ（RawDiff 30,700件 → イベント282件、
explained_ratio 1.0）から引いている。

## 1. 全体像：4つの層と、責務の徹底した分離

```
GTFS フィード（旧世代・新世代の zip）
   ↓  読み込み・正規化（全 .txt 保持、day_type 正規化）
L0  生差分の全列挙 ──────────── RawDiff（説明台帳の分母）
   ↓
L1  世代間同定 ────────────── MatchGraph（4種エンティティの対応仮説）
   ↓
    便の照合 ─────────────── TripDelta（便同士の割当結果）
   ↓
L2  イベント抽出 ───────────── ChangeEvent + evidence + 残差
   ↓
    （後編）プレゼンテーション層 → HTML レポート
```

この設計を理解する鍵は、各層が**何をしないか**にある。

- **L0 は解釈をしない**。生差分は主キーによる素朴・決定的な突合であり、
  「同じ停留所か」「同じ便か」という意味的判断を一切含まない。
- **L1・trip 照合は意味を割り当てない**。「何と何が同じものか」だけを内容から
  復元し、それが廃止なのか改称なのかは言わない。
- **L2 は対応付けをやり直さない**。L1 と trip 照合の結果を前提に、変化の意味を
  **ChangeEvent** というデータ構造として宣言し、その物証として L0 の生差分を
  引き取る。ChangeEvent はコアの最終出力であり、後編で扱うレポート生成は
  すべてその消費者である。

一見冗長なこの分業は、「網羅性の保証」という一点から導かれている。全差分を
列挙する台帳（分母）に解釈を混ぜると、解釈の変更で分母自体が動き、
「どこまで説明できたか」という被覆率の意味が失われる。だから最下層は
機械的に、賢さは上の層に、という構造になっている。

## 2. L0：RawDiff — 無解釈の全記録

### 2.1 データ形式

RawDiff 1件は7フィールドからなる（`model/rawdiff.py`）：

| フィールド | 内容 |
|---|---|
| `rawdiff_id` | `rawdiff_000001` 形式の連番。同じ世代ペアからは常に同じ ID が付く |
| `file` | `stop_times.txt` などファイル名 |
| `kind` | 差分の種別（下表） |
| `key` | 行を同定する主キー値の列 |
| `column` | セル変化・カラム増減のときの対象カラム名 |
| `old_value` / `new_value` | 新旧の値（生文字列） |

kind は基本7種＋集約3種：

| kind | 意味 | 粒度の規約 |
|---|---|---|
| `file_added` / `file_removed` | ファイルが片側にのみ存在 | **各1件**（中の行は列挙しない） |
| `column_added` / `column_removed` | カラムが片側にのみ存在 | カラムごとに1件 |
| `row_added` / `row_removed` | 主キー突合で片側にのみ存在する行 | 行ごとに1件 |
| `field_changed` | 両側にある行の値の相違 | **セル単位**（共通カラムのみ比較） |
| `rows_*_bulk` | 巨大ファイルの行差分の集約 | §2.5 |

「ファイル追加は1件」「カラム追加は1件」という粒度の規約は、説明台帳の
**分母の定義**そのものである。粒度をどう切っても、規約が決定的である限り
台帳の一貫性は保たれる。

前提となる読み込みの正規化も L0 の性格を決めている。zip 内の**全** `.txt` を
読む（既知の GTFS ファイルに限定しないため、事業者独自ファイルの差分も台帳に
現れる）。全カラムを文字列として読み、欠損は空文字列に統一する。比較は
**生文字列の比較**であり、型変換も正規化もしない——`36.68067987` と
`36.680679870` は別の値である。

### 2.2 行の対応は主キーの値一致だけで取る

before/after の行対応は、ファイルごとに定義された主キー（`KEY_COLUMNS`）の
値一致だけで取る。主要なもの：

| ファイル | 主キー |
|---|---|
| stops | `stop_id` |
| routes / routes_jp | `route_id` |
| trips | `trip_id` |
| stop_times | `(trip_id, stop_sequence)` |
| calendar | `service_id` |
| calendar_dates | `(service_id, date)` |
| fare_rules | `(fare_id, route_id, origin_id, destination_id, contains_id)` のうち存在するカラム |
| shapes | `(shape_id, shape_pt_sequence)` |
| feed_info | （単一行ファイルとして全フィールド比較） |

同じ主キー値の行が両世代にあれば「同じ行」とみなして共通カラムをセル単位で
比較し、片側にしかなければ行の追加・削除とする。ここに意味的判断はない。
trip_id が振り直されたフィードでは大量の row_removed / row_added が出るが、
それは L0 の正しい動作である（§2.4 の実例(e)）。

実装は2パス方式で、まず全行の「主キータプル → 行内容の SHA-1 ダイジェスト」を
両世代分作って差分の**所在**だけを求め、値の取り出しは差分行の分に限る。
メモリ消費はファイル規模ではなく差分規模に比例し、200万行級のファイルにも
耐える。

主キーが未定義のファイル（未知の独自ファイル）、主キーカラムが欠落している
ファイル、主キー値が**重複**しているファイル（実フィードに実在する）は、
行全体の多重集合比較にフォールバックする。key には行内容のダイジェスト＋
出現番号が入り、行内容そのものが old_value / new_value に保持される。
この経路ではセルの変更は「削除＋追加の対」として現れる。

### 2.3 集約（bulk）：粒度規約の例外条項

`fare_rules.txt` が200万行あるフィードが実在する（全停留所ペア×全系統の
運賃表）。行単位で列挙すると台帳が数百万件になるため、検出ルールが行粒度の
証拠を必要としないファイルに限り、行差分が閾値を超えたら件数だけを保持する
集約 RawDiff（`rows_removed_bulk` 等）各1件に落とす。stops / trips /
stop_times / calendar など行粒度の証拠を使うコアファイル群は、規模によらず
必ず行単位で列挙する。

なお、この設計の帰結として **RawDiff はパッチ形式ではない**。keyed 経路の
row_added は主キーしか持たず行の中身を記録しないし、file_added は中身を
列挙しない。「旧 GTFS + RawDiff から新 GTFS を再現する」ことは（bulk 以前に）
できない。検出ルールは新旧スナップショットを両方持っており、RawDiff は
「スナップショット内のどこが変わったか」を指す**位置指定子＋変更事実**として
機能する。ここが、修正の再適用を目的として行内容まで運ぶ MobilityData の
GTFS Diff 仕様との設計上の分岐点である。あちらは交換形式、こちらは
説明責任を閉じるための台帳であり、同じ "diff" でも目的が違う。

### 2.4 実例で読む RawDiff

地鉄改正ペアの台帳は 30,700 件。内訳は row_removed 12,380 / row_added 8,161 /
field_changed 10,159。ファイル別では stop_times.txt 18,747、shapes.txt 5,672、
fare_rules.txt 5,541、trips.txt 623、stops.txt 35 と続く。実物を見ていく。

**(a) フィード有効期間の書き換え**

```json
{"rawdiff_id": "rawdiff_000001", "file": "calendar.txt", "kind": "field_changed",
 "key": ["土日祝"], "column": "end_date",
 "old_value": "20261015", "new_value": "20270331"}
```

`service_id=土日祝` の行の `end_date` セルが変わった、とだけ言っている。
これが「フィード期限の延長」を意味するかどうかは L2 が決める。

**(b) 祝日例外の入れ替え（複合キー）**

```json
{"file": "calendar_dates.txt", "kind": "row_removed", "key": ["土日祝", "20251103"]}
{"file": "calendar_dates.txt", "kind": "row_added",   "key": ["土日祝", "20260429"]}
```

主キーが `(service_id, date)` なので日付の入れ替えは削除＋追加の対になる。
旧世代の祝日（文化の日）が消え新世代の祝日（昭和の日）が入っただけ——つまり
有効期間スライドの機械的な差分である可能性が高いが、その判定（新旧有効期間の
重なり窓の内か外か）は L2 の HOLIDAY_EXCEPTION_CHANGED ルールが行う。

**(c) 停留所の座標変化（乗り場ごとに出る）**

```json
{"file": "stops.txt", "kind": "field_changed", "key": ["3441_01"],
 "column": "stop_lat", "old_value": "36.6766371133334", "new_value": "36.6775121133334"}
```

停留所「神明」の上下2乗り場（`3441_01` / `3441_02`）の lat / lon が変わり、
計**4件**の field_changed になる。L0 にとっては座標セルが4つ変わっただけで、
これが「385 m の移設」であるという解釈はすべて上の層の仕事である。

**(d) 停留所改称**

```json
{"file": "stops.txt", "kind": "field_changed", "key": ["5311_01"],
 "column": "stop_name", "old_value": "浜黒崎小学校", "new_value": "古志はるかぜ学園前"}
```

`5311_02` にも同じ変更があり計2件。stop_id が維持されているので主キー突合が
そのまま対応を与える素直なケースである。

**(e) 便の時刻変更が「削除＋追加」に化けるケース**

このフィードの trip_id は `土日祝_06時37分_系統601_1` のように**内容（曜日・
発時刻・系統）を埋め込んだ命名**になっている。6:37発の便が6:29発に変わると
trip_id 自体が変わるため、L0 では

```json
{"file": "trips.txt", "kind": "row_removed", "key": ["土日祝_06時37分_系統601_1"]}
{"file": "trips.txt", "kind": "row_added",   "key": ["土日祝_06時29分_系統601_1"]}
```

となり、さらにこの便に属する stop_times 全行（数十件）が row_removed /
row_added になる。**時刻を8分ずらしただけで台帳に数十件の差分が載る**。
stop_times.txt が全体の6割を占める主因はこれである。この削除・追加の山から
「同じ便が8分繰り上がった」を復元するのが trip 照合（§4）である。

trip_id が維持されたまま時刻セルだけが変わる素直なケースは、こうなる：

```json
{"file": "stop_times.txt", "kind": "field_changed",
 "key": ["土日祝_06時16分_系統601_2", "4"], "column": "arrival_time",
 "old_value": "06:18:00", "new_value": "06:17:00"}
```

**(f) 行先表示の一括変更**

```json
{"file": "stop_times.txt", "kind": "field_changed",
 "key": ["平日_15時20分_系統314_4_2", "1"], "column": "stop_headsign",
 "old_value": "14富山駅前（布目・富山大学 経由）",
 "new_value": "14富山駅（布目・富山大学 経由）"}
```

同種の field_changed が**120件**並ぶ。この120件がどう扱われたかは §5.4 と §5.7 で述べる。

## 3. L1：MatchGraph — 「同じもの」を内容から復元する

### 3.1 なぜ同定の層が必要か

GTFS の各 ID はフィード内の参照整合性のためのものであり、世代間の同一性を
保証しない。実フィードで観察された ID の振る舞いはおよそ3類型ある。

1. **内容埋め込み型**：地鉄のように trip_id に時刻や系統名を埋め込む。時刻が
   変わるたびに ID が変わる。
2. **連番型**：trip_id が「系統内の発順」の連番。便が1本増減すると以降の全便の
   ID が1つずつスライドし、**同じ ID が別の便を指す**（八戸市営バスの実例）。
   ID を信じると誤対応が量産される最悪のケースである。
3. **全面張り替え型**：データ作成ツールの変更などで route_id・stop_id が
   一斉に変わる。内容はほぼ同一なのに ID は全て別物になる。

さらに ID が安定していても名称が動く。名古屋市営バスでは route_id は同一の
まま路線名が「鳴.ワイ」→「鳴.メグ」に変わり、同時に沿線の停留所名も変わった。
ID も名称も、単独では同一性の根拠にならない。そこで L1 は、名称・座標・
接続関係・停車列という**内容**から「同じもの」を再構成する。

### 3.2 MatchGraph の形式：確定表ではなく仮説の束

L1 の出力は `MatchEdge` のフラットな集合である（`model/matchgraph.py`）：

```python
MatchEdge(
    entity_type,  # "stop_cluster" | "route_family" | "pattern_cluster" | "service"
    old_id,       # 旧世代エンティティ ID
    new_id,       # 新世代エンティティ ID
    confidence,   # 0.0–1.0
    method,       # 同定方法のラベル (根拠の種類)
)
```

重要な性質が2つある。第一に、これは**1:1 の確定対応表ではない**。1つの旧
エンティティに複数の候補エッジがぶら下がってよく、実測では地鉄ペアの停留所
クラスタ 649 個に対しエッジは 1,115 本ある（最良対応649本＋次点以下466本）。
第二に、**採否の判断は L1 では行わない**。閾値スレスレの対応も floor 以上なら
保持し、L2 の各ルールが「最良エッジの confidence が 0.5 以上なら対応済みと
みなす」等の基準で使う。破棄を遅延させるのは、下流の整合性が裏づけになって
から仮説を昇格させるためで、method フィールドがあるのは「なぜ同じと判断
したか」をイベントの confidence として追跡できるようにするためである。

### 3.3 4種のエンティティとそれぞれの照合方法

MatchGraph には4種類のエンティティが載る。それぞれ独立のサブグラフだが、
構築には依存順序があり、前段の結果を後段が使う。

**(1) stop_cluster：物理停留所。** まず**世代内**で乗り場（stop_id）を物理
停留所に名寄せする。`stop_name` から「のりば」「番線」「末尾の数字・丸数字」
等を除去した**基底名**が一致し、既存クラスタの代表座標から 100 m 以内なら
同一クラスタとする（`parent_station` があれば優先）。「学校前」「教会前」の
ような一般名の停留所がフィード内の離れた場所に複数ある場合（実例：茅ヶ崎の
「教会前」は約3 km 離れて2箇所、前橋の「学校前」は約6 km 離れて2箇所）は、
距離条件により正しく別クラスタになる。

次に**世代間**でクラスタ対をスコアリングする：

```
score = 0.35 × 乗り場共有率        （共有 stop_id 数 / max(旧,新)）
      + 0.25 × 接続路線の類似度     （接続 route family 集合の Jaccard。ID でなく名称で）
      + 0.25 × 距離スコア          （300 m を上限とする線形）
      + 0.15 × 基底名の文字列類似度
```

4項がすべて異種の証拠（ID・トポロジー・幾何・名称）である点がポイントで、
どれか1つが裏切っても（改称・移設・ID張替え）他の項が支える。候補は空間
グリッドの近傍から取るが、**stop_id を共有するクラスタは距離に関係なく候補に
含める**。§2.4(c) の「神明」は 385 m 移動しており空間絞り込み（300 m）から
漏れるが、stop_id が維持されていたためこの経路で同定された。method は
基底名完全一致なら `name_exact`、それ以外は `composite`。

**(2) route_family：利用者が認識する「路線」。** route_short_name 等の表示名で
route_id 群を束ねたものが family である。世代間対応は次の2段：

- 名称が完全一致する family 同士を confidence 1.0 で結ぶ（`name_exact`）。
- 名称が一致しない対は、**翻訳済み停留所集合の Jaccard** で判定する
  （`stops_translated`）。「翻訳」とは、(1) で得た停留所クラスタの世代間対応を
  「旧基底名 → 新基底名」の写像として使い、旧路線の停留所集合を新世代の語彙に
  写してから比較することを指す。

なぜ翻訳が要るのか。路線改称は施設名変更に由来することが多く、そのとき沿線の
停留所名も**同時に**変わる。名古屋の「鳴.ワイ」→「鳴.メグ」がまさにそれで、
翻訳なしに停留所名の集合を比べると路線照合と停留所照合が**共倒れ**する。
停留所は座標や乗り場 ID など名称以外の証拠で先に対応できるので、その結果を
路線照合に注入する——これが L1 内部の依存チェーンの核心である。

受理されたエッジの連結成分は形で分類され、1:1 は改称、N:1 は統合、1:N は
分割、N:M は再編として L2 の A 群イベントの入力になる。朝日町営バスでは
命名規則の全面変更を伴う 21路線 → 9路線 の再編が、この成分分類で
改称2件＋統合7件として解決された。ただし成分分類には落とし穴があり、幹線
道路（コリドー）を共有する別路線同士が停留所集合の類似で連鎖し、巨大な成分が
できることがある（朝日町の宮崎境線↔市振線）。関与路線数が閾値を超えた成分は
各旧 family の最良エッジだけ残す **best-match 間引き**で再分解する。受理に
届かなかった弱い仮説（`stops_candidate`）はイベントには使わず、レポートの
「内容が類似する路線（対応の可能性）」という注記にだけ使われる。

**(3) pattern_cluster：停車パターン。** (family, 方向) 内の一意な停車列を、
stop_id 列ではなく**停留所クラスタ基底名列**として表す（ID が全交換されても
比較が成立する）。パターン間類似度は

```
sim = 0.6 × LCS 類似（2·LCS長 / (len_a + len_b)）
    + 0.3 × 順序整合 （共通停留所の相対順序が保存されるペアの割合）
    + 0.1 × 包含     （一方が他方の部分集合のとき min_len / max_len）
```

で、世代内では sim ≥ 0.5 の連結成分としてクラスタ化し、世代間では対応済み
family ペアの内側で代表パターン同士を結ぶ（`pattern_similarity`）。

**(4) service：運行日種別。** calendar / calendar_dates の表現の揺れを、
service ごとに day_type（weekday / saturday / sunday_holiday / weekend /
daily / `dow_XXXXXXX`（曜日指定）/ irregular（特定日）/ inactive（運行日
なし））へ正規化した上で、両世代に存在する day_type 同士を confidence 1.0 で
機械的に対応させる。曜日指定 `dow_1010000`（月・水曜）のような型を一級市民に
したのは実測に基づく：43フィードの調査で、従来「特定日運行」として扱われて
いた service の便数の**83%** が通学日・診療日等の規則的な曜日指定運行だった。

### 3.4 route_group：family の上の「路線ブランド」集約層

MatchGraph の4種に加えて、L1 にはもう1つ、**route_group** という概念がある。
以降の trip 照合（§4.3）とレポートの章立て（後編）で使われるので、ここで
位置づけをはっきりさせておく。

**動機。** family（表示名の完全一致グループ）は「同名で route_id が細分される」
型のフィードを吸収できる（富山地鉄は 172 の route_id が 44 family に畳まれる）。
しかし**枝番付きの系統名が別 route として定義される**型は吸収できない。
実例が永井運輸の「30A 前橋玉村線」「30B 前橋玉村線」「30E …」「30H …」で、
これらはそれぞれ別 family になる。利用者の認識は「前橋玉村線」1本であり、
さらに検出品質にも響く：便が 30A から 30E に付け替わると、family 単位では
「30A で減便＋30E で増便」に見えてしまう。GTFS の route の粒度は事業者・
作成ツール依存でばらばらであり、「利用者が認識する路線」をデータから再構成
する層がもう1段必要になる。そこで family を置き換えず、**上に1階層足す**：

```
route_group（前橋玉村線）            ← 路線ブランド
  ⊃ route_family（30A / 30B / 30E…） ← 表示名の完全一致グループ
    ⊃ 方向 × パターンクラスタ         ← 停車列の構造
```

**求め方。** グループ化キーは**語幹**である。family 表示名を NFKC 正規化し、
先頭の系統コードらしき連続（英数字・空白・ハイフン類・中点等）を除去した
残りを語幹とし、語幹が同じ family を1つの route_group にまとめる
（「30A 前橋玉村線」→ 語幹「前橋玉村線」）。ガードが2つあり、語幹が2文字未満、
または「系統」「線」「循環」「バス」等のストップワードに退化した場合は
グループ化しない（実例：熊本市電の「A系統」「B系統」は語幹が「系統」に
退化するため束ねない）。全 family は必ずいずれかの group に所属し、単独
family は自身だけの group になる（グループ化が不要なフィードでは無害に退化
する設計）。

注目すべきは、**停留所集合の類似をグループ化の条件に使わない**ことである。
当初案は「語幹一致 かつ 停留所 Jaccard ≥ 閾値」だったが、80フィードの横断
調査で棄却された。枝番系統は同一ブランド名の下の**別コリドー**であることが
普通にあり（30A と 30B の共通停留所は玉村町役場の1つだけ、Jaccard 0.017）、
類似ゲートを課すと動機となった当のケースを弾いてしまう。代わりに Jaccard は
**凝集度**というメタデータとして保持し、低凝集の group にはレポートが
「枝線構造」と注記する——判定には使わず、透明性の装置として使う。

**MatchGraph との関係。** route_group は MatchGraph の5番目のエンティティ
**ではない**。対応仮説のエッジも confidence も持たない、**世代内の決定的な
グループ化**である。世代間の対応は、語幹の一致と構成 family の対応
（§3.3 (2)）の上に乗るだけで、新しい対応問題は生じない。役割の違いを
対比すると：MatchGraph が「世代間で何と何が同じか」という**仮説**の束で
あるのに対し、route_group は「世代内で何と何を同じ見出しの下に置くか」
という**規約**である。

### 3.5 依存チェーンのまとめ

```
stop_cluster ──(基底名の翻訳表)──▶ route_family ──(ペア内ブロッキング)──▶ pattern_cluster
                                        │
                                        ├──(語幹で集約)──▶ route_group ──(ブロック軸)──▶ trip 照合 (§4)
                                        │
service (day_type) ────────────────────┴──────────────▶ trip 照合のブロック軸
```

停留所の対応が路線照合の語彙を与え、路線の対応がパターン照合と便照合の
探索範囲を区切り、family の語幹集約が便照合のブロック軸と章立ての単位を
与える。逆方向の依存はない。

## 4. trip 照合：TripDelta — 便という「個体」の割当問題

### 4.1 なぜ MatchGraph と別枠なのか

停留所・路線・パターン・運行日種別は「世代をまたいで存続する器」であり、
数も高々数百である。一方、便（trip）は数千〜数万の「個体」で、1つの器の中の
どの個体とどの個体が同じかという**割当問題**を解く必要がある。データ構造も
アルゴリズムも別立てになっており、結果は MatchGraph ではなく `TripDelta` に
格納される：

```python
TripInfo(trip_id, route_id, family, direction, day_type,
         base_seq,   # 停留所クラスタ基底名の列
         times)      # 全停留所の (arrival, departure) 列

TripDelta(
    exact_pairs,  # 内容署名の完全一致対（うち trip_id が違う対 = churn）
    modified,     # コスト割当で対応した対
    removed,      # 相手が見つからなかった旧世代の便
    added,        # 同・新世代の便
)
```

### 4.2 v1 の失敗：ID を「証拠」ではなく「定義」にしていた

初期実装（v1）は「同一 trip_id なら同一の便」を無条件に信頼していた。これは
連番型 ID のフィードで破綻する。八戸市営バスでは trip_id が系統内の発順で
振られており、改正で便が1本増減すると以降の全便の ID がスライドする。v1 は
「同一 ID・時刻差210分」という対を平然と組み、その巻き添えで本当の対応が
removed / added に落ちて増減便が水増しされた。

ここから得られた設計原則が「**証拠の強さの序列は、内容完全一致 ＞ 時刻・経路の
整合 ＞ ID 一致**」である。ID の一致は同一性の定義ではなく、数ある証拠の中で
最も弱いものとして扱う。v2 はこの序列をコスト関数として実装した。

### 4.3 v2 の手順

**段1：内容署名の完全一致。** 署名 = (family, 方向, day_type, 停車列, 全停留所の
発着時刻列)。これが完全に一致する対は無条件に結ぶ（exact）。このうち trip_id
だけが異なる対が churn で、「ダイヤは同一なのに ID が全部張り替わった」という
データ更新の実態を F 群イベント（TECHNICAL_ID_CHURN）として報告する材料に
なる。ID 全面張り替え型のフィード（徳島の畑線など）はこの段で丸ごと救済される。

**段2：ブロック内のコスト最小割当。** 残った便を **route_group（§3.4）×
day_type** のブロックに区切り、ブロック内で全対のコストを計算する：

```
cost(o, n) = min(Δt_shared, 60分)/60分          … 時刻の不整合
           + 1.0 × (1 − LCS率)                  … 経路の不整合
           − 0.05 × [trip_id が一致]            … ID は弱い事前
受理ゲート: LCS率 ≥ 0.5 かつ Δt_shared ≤ 60分
```

各項の設計意図：

- **Δt_shared は「共有停留所での発時刻差の中央値」**。始発時刻差ではない。
  区間短縮の実例（八戸：旧 11:30発・32停 → 新 11:30発・28停）では、共有する
  28停留所の時刻がほぼ据え置きなので Δt_shared ≈ 0 となり強く結ばれる。
  途中始発化・延長にも同じ理屈で頑健である。
- **LCS 率は順序を考慮する**ので、停留所集合がほぼ同じ逆方向の便を誤って
  結ばない（集合 Jaccard の弱点の回避）。
- **ID 一致は 0.05 の減点（ボーナス）**でしかない。八戸の「同一 ID・210分差」は
  時刻項に負けて不成立になり、「ID 違い・0分差・経路包含」が成立する。
  ほぼ互角の候補が並んだときだけ ID がタイブレークとして効く。

ブロックの軸に family でなく route_group を使うのは、まさに §3.4 の動機が
便のレベルでも現れるからである。徳島バス加茂谷線では「便を枝番違いの route
（202→204）へ移す」運用が見つかった。trip_id まで同一・停車列 LCS 0.93・
時刻差5分という明白な同一便が、family 境界のブロッキングでは候補にすら
入らなかった（実測で徳島の removed の23%がこの型）。route_group を軸に
すれば、枝番間の便の付け替えは「ブランド内の割当」として自然に解ける。
また改称・統合された路線の便も、family 対応（§3.3）を経由して新側の
route_group に写されるので候補になる。

割当はコスト昇順の決定的貪欲で行う。貪欲で足りる根拠は感度分析にあり、
正解対と非正解対のコストは大きく二極化している（正解の95パーセンタイル 0.10
に対し非正解の5パーセンタイル 0.77）。際どい閾値に依存していない。
人手で作成した正解データに対する再現率は 0.992 で、未回収の3件はいずれも
day_type を跨いで移動した便（平日→毎日など。v2 のスコープ外と明記されている
制限）だった。

**段3：残り。** どの相手も得なかった便が removed / added となる。

### 4.4 分類と下流の消費者

| 分類 | 条件 | 下流での用途 |
|---|---|---|
| exact | 署名完全一致 | ID も同一なら「変更なし」 |
| churn | exact のうち trip_id 相違 | TECHNICAL_ID_CHURN（F群） |
| modified | ブロック内割当で対応（ID は跨ぎうる） | 時刻修正（C群）・経路変更（B群）・差分時刻表の列対応 |
| removed / added | 相手なし | 時間帯ビン別の増減便（C群）、路線廃止・新設（A群） |

§2.4(e) の「6:37 → 6:29」の便はここで modified として結ばれ、その全停留所の
時刻差が一様（標準偏差 ≤ 120秒）なら TIMETABLE_SHIFTED、非一様なら区間別の
TRAVEL_TIME_CHANGED として C 群で意味づけされる。

## 5. L2：ChangeEvent — 意味の割当と説明台帳

### 5.1 ChangeEvent の構造

L2 の出力単位は **ChangeEvent** と呼ぶデータ構造である（後編で扱う
プレゼンテーション層も、このChangeEvent の消費者として書かれている）。
1件の ChangeEvent は次の形をとる：

```json
{
  "event_id": "evt_000036",
  "type": "SERVICE_REDUCED",                 // 英語のタイプ ID（下のカタログ）
  "subject": {                                // 何についての変化か
    "route_group": "新幹線市街地線",
    "route_family": "01　新幹線市街地線",
    "direction": "", "day_type": "weekday"
  },
  "old_ref": {...}, "new_ref": {...},         // 新旧の参照（型により内容が異なる）
  "quantification": {                         // 定量情報。閾値で切り捨てず数値を保持
    "time_band": "16:00-19:00", "old_count": 3, "new_count": 2
  },
  "evidence": ["rawdiff_016639", ...],        // 説明する RawDiff の ID 集合 = 台帳
  "confidence": 1.0,                          // L1 同定の確信度の伝播
  "severity": "major",                        // 利用者影響度（confidence とは独立の軸）
  "display_name_ja": "減便"
}
```

confidence と severity が独立の軸である点は設計上の含意がある。「確実に
起きた小さな変化」（confidence 1.0 / info）も「同定が仮説的だが影響の大きい
変化」（confidence 0.6 / major）もあり、両者を1つの数値に潰さない。

### 5.2 イベント体系：6群41タイプのカタログ

検出の語彙は6群41タイプ＋残差の受け皿である。全タイプを示す（※印は
体系に定義済みだが検出ルールが未実装のもの。severity の major/minor/info は
利用者影響度の既定値）。

**A. 路線（subject: route_family）**

| タイプ | 意味 | severity |
|---|---|---|
| ROUTE_ADDED / ROUTE_DISCONTINUED | 路線の新設・廃止（対応成分で説明されない出現・消滅） | major |
| ROUTE_RENAMED | 1:1 対応成分の名称変更 | minor |
| ROUTE_MERGED / ROUTE_SPLIT | N:1 / 1:N の対応成分（統合・分割） | major |
| ROUTE_RESTRUCTURED | N:M の対応成分（再編） | major |
| THROUGH_SERVICE_INTRODUCED / _DISCONTINUED ※ | 2路線の連接（直通化）とその解消 | major |

**B. 停車パターン（subject: family × 方向 × パターン）**

| タイプ | 意味 | severity |
|---|---|---|
| PATTERN_EXTENDED / PATTERN_TRUNCATED | 端点への追加・端点からの削除（延伸・短縮） | major |
| STOP_INSERTED_IN_PATTERN / STOP_REMOVED_FROM_PATTERN | 中間1停留所の挿入・削除 | minor |
| DETOUR_ADDED / DETOUR_REMOVED | 中間への連続2停留所以上の挿入・削除（迂回・経由の変化） | minor |
| TIME_BAND_VARIANT ※ | パターンの出現が特定時間帯・曜日に限定される/されなくなる | minor |
| SHAPE_CHANGED | 停車列は同一で shapes.txt の経路形状が変化（離散 Fréchet 距離で判定） | minor |

**C. 便数・時刻（subject: family × 方向 × day_type）**

| タイプ | 意味 | severity |
|---|---|---|
| SERVICE_REDUCED / SERVICE_INCREASED | 時間帯ビン別の本数増減 | 減便は通勤帯 major・他 minor |
| TRIPS_TRUNCATED ※ | 全区間便から区間便への本数移動（family 合計はほぼ保存） | major |
| FIRST_LAST_CHANGED | 始発繰り下げ・終発繰り上げ等（既定15分超） | major |
| TIMETABLE_SHIFTED | 時刻集合のほぼ一様なシフト | info |
| TRAVEL_TIME_CHANGED | 区間別の所要時間変化（ダイヤ余裕の増減を追える粒度） | minor |
| DWELL_TIME_CHANGED ※ | 同一停留所での停車時分の変化 | info |

**D. 停留所（subject: stop_cluster）**

| タイプ | 意味 | severity |
|---|---|---|
| STOP_ADDED / STOP_REMOVED | クラスタ単位の新設・廃止 | major |
| STOP_RENAMED | 対応済みクラスタの名称変化 | minor |
| STOP_RELOCATED | 対応済みクラスタの代表座標が閾値（既定300 m）超移動 | minor |
| PLATFORM_CHANGED | クラスタ内での乗り場変更（stop_id 付け替え・座標微修正） | minor |
| PLATFORM_ADDED / PLATFORM_REMOVED | クラスタ内の乗り場数の増減 | info |

**E. 運行日・カレンダー（subject: service）**

| タイプ | 意味 | severity |
|---|---|---|
| DAYTYPE_RESTRUCTURED | 曜日区分の再編（例：土曜ダイヤ廃止→休日へ統合） | major |
| HOLIDAY_EXCEPTION_CHANGED | calendar_dates の例外差分（有効期間の重なり窓で実質分と機械分を区別） | info |
| SEASONAL_SERVICE_CHANGED | 特定日運行のみの路線の出現・消滅（季節サービスの仮説、confidence 0.6） | minor |

**F. 運用形態・メタデータ・形式層（網羅性の受け皿）**

| タイプ | 意味 | severity |
|---|---|---|
| DEMAND_RESPONSIVE_CHANGE | デマンド化・予約制移行の兆候の合成（pickup_type、GTFS-Flex 系ファイル出現等。兆候数で confidence を加算） | major |
| FARE_CHANGED | 運賃改定（fare_attributes / fare_rules 全差分を1件で消費、運賃表を分解して定量化） | major |
| FEED_VALIDITY_CHANGED | フィード有効期間の書き換え | info |
| AGENCY_INFO_CHANGED / TRANSLATION_CHANGED | 事業者情報・翻訳の差分 | info |
| ACCESSIBILITY_CHANGED | 車椅子対応情報の変化 | minor |
| HEADSIGN_CHANGED | 行先表示の表記変更のみ（経路・時刻は不変） | info |
| TECHNICAL_ID_CHURN | 意味的変化を伴わない ID 張り替え（trip / route / shape） | info |
| UNEXPLAINED_RESIDUAL | どのルールにも説明されなかった残差（タイプ数には数えない受け皿） | — |

このカタログは閉じた分類ではなく、**残差から成長する**（§5.7）。また F 群の
存在が示すように、体系は「利用者向けの大きな変化」だけでなく「データ管理上の
変化」（ID 張り替え、期限書き換え）まで語彙に含む。網羅を謳う以上、意味の
薄い差分にも名前が要るからである。

### 5.3 割当の原理：イベント駆動・多対多・主説明1つ

「1つの RawDiff を41タイプのどれかに分類する」のでは**ない**。向きが逆で、
検出ルールが MatchGraph・TripDelta・スナップショットを見て ChangeEvent を
生成し、生成の瞬間に「このイベントが説明する RawDiff の ID 集合」を
**evidence** として台帳に記録する。規約は：

- 1つの ChangeEvent が数件〜数百件の RawDiff を説明する。
- 1つの RawDiff を複数の ChangeEvent が evidence にしてよい（多対多）。
  ただし**主説明（primary）は最初に触れたイベント1つ**で（先勝ち）、
  被覆率の計上はこの primary に基づく。
- 上位イベントは配下の差分を**丸ごと**消費する。路線廃止なら routes の行に
  加え、所属する全便の trips / stop_times 行差分まで。経路変更なら当該便の
  stop_times 差分全体（経路が変われば時刻も headsign も連鎖して変わるため）。
- カスケード終了後、どのイベントにも引き取られなかった RawDiff がファイル
  単位の UNEXPLAINED_RESIDUAL に集約される（残差による消費は「説明」に
  数えない）。

### 5.4 カスケード：適用順序が誤認と二重計上を防ぐ

ルールは次の固定順で適用される：

```
stops → routes → patterns → frequency → calendars → shapes → metadata
     → technical (ID churn) → 残差集約
```

この順序は恣意的ではなく、1つ1つが「先に確定させないと後段が誤る」という
依存から決まっている。代表的な3つを挙げる。

**(1) 改称を「廃止＋新設」と誤認しない。** routes 内部の判定順は
RENAMED / MERGED / SPLIT / RESTRUCTURED（対応成分の照合）→ route_id churn →
DISCONTINUED / ADDED である。名称が変わった路線は、まず対応成分として
説明を試み、それでも残った出現・消滅だけが新設・廃止として確定する。
順序が逆なら、朝日町の21→9路線再編は「21路線廃止・9路線新設」という
大惨事として報告されていた。

**(2) 廃止路線の便を減便に数えない。** C 群（増減便）の入力プールは
TripDelta の removed / added のうち「**A 群が主消費済みでない便**」に限る。
ぶりかにバス廃止の4便は A 群の ROUTE_DISCONTINUED が trips / stop_times ごと
先に消費しているので、C 群の減便集計には現れない。カスケード順（routes が
frequency より先）と primary 先勝ちの組み合わせが、この二重計上防止の実体である。

**(3) 経路変更に伴う headsign 変更を、独立の表記変更と区別する。** B 群の
経路変更イベントは当該便の stop_times 差分を「丸ごと」消費する——そこには
時刻の連鎖変更も headsign の変更も含まれる。後段の metadata ルール
（HEADSIGN_CHANGED）が拾うのは「**未消費の** headsign 差分」だけである。
地鉄の実データでは、経路変更に伴う行先表示変化は B 群の evidence に吸収され、
独立の一括変更120件（「富山駅前」→「富山駅」）だけが HEADSIGN_CHANGED として
報告された。同じ column の同じ種類の差分でも、文脈（どの物語の一部か）に
よって帰属が変わる——これが「先勝ち」の実質的な意味である。

まとめると、優先順位は2つの直交する仕組みで実現されている：
**ルールの適用順序**（大きな物語から順に語る）と、**evidence の primary
先勝ち**（一度語られた物証の帰属は動かない）。両者が組み合わさって、
30,700 件の生差分が重複も脱落もなく 282 件の ChangeEvent に配られる。

### 5.5 実例：ChangeEvent と evidence の対応

地鉄ペアでは 30,700 件の RawDiff が 282 件の ChangeEvent に説明された
（約109:1 の圧縮）。

**路線廃止（ぶりかにバス）— evidence 26件**

```json
{"type": "ROUTE_DISCONTINUED", "subject": {"route_family": "ぶりかにバス線"},
 "old_ref": {"route_ids": ["ぶりかにバス線(400_3_1)", "ぶりかにバス線(400_3_2)"],
             "trip_count": 4},
 "evidence": [...計26件], "confidence": 1.0, "severity": "major"}
```

内訳は routes.txt の row_removed 2件＋所属4便の trips row_removed 4件＋
その stop_times row_removed 20件。「丸ごと消費」の典型で、これらが別途
「減便」に二重計上されることはない。

**停留所移設（神明）— evidence 4件**

```json
{"type": "STOP_RELOCATED", "subject": {"stop_cluster": "神明"},
 "quantification": {"moved_m": 385.3},
 "evidence": ["rawdiff_030041", "rawdiff_030042", "rawdiff_030043", "rawdiff_030044"],
 "confidence": 0.75}
```

§2.4(c) の座標セル4件がそのまま evidence。confidence 0.75 は stop_cluster
対応エッジの確信度の伝播である。

**減便 — evidence 22件（便1本分の trips＋stop_times）**

```json
{"type": "SERVICE_REDUCED",
 "subject": {"route_family": "01　新幹線市街地線", "day_type": "weekday"},
 "quantification": {"time_band": "16:00-19:00", "old_count": 3, "new_count": 2},
 "evidence": [...計22件], "severity": "major"}
```

trip 照合で相手が見つからなかった removed 便1本の全行差分が evidence。
「夕方帯 3→2本」という認知単位の言明と、その物証であるセル差分が evidence を
介して接続されている。

### 5.6 台帳の集計

events.json の末尾には必ず accounting ブロックが載る：

```json
"accounting": {"rawdiff_total": 30700, "explained": 30700,
               "explained_ratio": 1.0, "residual_breakdown_by_file": {}}
```

explained_ratio が 1.0 未満のときは residual_breakdown_by_file にファイル別の
残差件数が入り、レポートの検証ビューから残差1件1件（§2.4 で見た形の
RawDiff）まで辿れる。

### 5.7 残差がカタログを育てる

説明台帳は検証の道具であると同時に、開発ループの駆動装置でもある。地鉄の
検証で残った未説明残差124件の精査から、2つの改良が生まれた。

- **120件**は §2.4(f) の行先表示の一括変更だった。イベント体系に該当タイプが
  なかったため、「行先表示変更（HEADSIGN_CHANGED）」が新タイプとして採録され、
  残差は解消した。
- **4件**は「神明」の 385 m 移設で、世代間リンクの空間絞り込み（300 m）を
  超えたため同定に漏れていた。「stop_id を共有するクラスタは距離に関係なく
  候補に含める」という L1 の修正（§3.3）はこの残差から生まれた。

「残差の精査 → ルールまたは体系の改訂」というループにより、見落としは
静かに消えるのではなく、必ず残差として可視化され、カタログの成長に転化される。

## 6. まとめと後編への接続

| 層 | 出力 | 対応付けの根拠 | 解釈 |
|---|---|---|---|
| L0 RawDiff | 生差分の台帳（分母） | ファイル別の主キー値一致（fallback：行ハッシュ） | しない |
| L1 MatchGraph | 4種エンティティの対応仮説＋route_group 集約 | 内容（名称・座標・接続・停車列）の複合スコア／語幹 | 同一性のみ |
| trip 照合 TripDelta | 便の割当（exact/churn/modified/removed/added） | 内容署名＋コスト最小割当（ID は弱い事前） | 便の同一性のみ |
| L2 イベント | ChangeEvent＋evidence＋残差 | ルールカスケードと消費規約 | 意味の割当 |

コアの出力である ChangeEvent JSON は、しかし「台帳を漏れなく紐付ける最小
単位」で切られており、人が読む単位ではない。減便は時間帯ビンごとに分裂して
いるし、1つのダイヤ改正から派生した兄弟イベントは散らばっている。これを
「路線のページ」「新旧の時刻表」という認知単位へ組み直すのがプレゼンテーション
層であり、そこでもう一度「コアが決めた事実を表示層が上書きしない」という
規律が問われることになる。後編で扱う。
