# gtfs-semantic-diff の内部構造（後編）：プレゼンテーション層 — 台帳の単位から認知の単位へ

本稿は gtfs-semantic-diff の内部構造を解説する技術記事の**後編**である。
前編ではコア——生差分の全列挙（L0・RawDiff）、世代間同定（L1・MatchGraph）、
便の照合（TripDelta）、イベント抽出と説明台帳（L2・ChangeEvent）——を扱った。
後編は、そのコアの出力から人が読める HTML レポートを合成する
プレゼンテーション層（`report/presentation.py` 約1,500行＋`report/bundle.py`＋
Svelte ビューア）を扱う。

先に全体の位置づけを確認しておく。HTML レポートには「レポートモード」と
「検証モード（網羅性ビュー）」の2つの顔があるが、**どちらも同じ
ChangeEvent JSON の消費者**である。検証モードがイベントと台帳をほぼ素の
粒度で見せるのに対し、レポートモードはその上に**プレゼンテーションモデル**
という中間表現を合成して見せる。そしてこの層の設計を貫くのは、
「表示のための導出・集約は大胆に行うが、**同定（何と何が同じか）はコアの
結果を一切上書きしない**」という規律である。この規律がなぜ必要かは、
実際に逆をやって失敗した記録とともに §5 で述べる。

```
ChangeEvent + accounting     ← 検証の背骨。プレゼン層は読み取りのみ
        ↓ 合成・集約 (report/presentation.py)
プレゼンテーションモデル      ← 認知単位に組み直した中間表現
        ↓ bundle.py で単一 HTML に同梱
viewer (Svelte)              ← レポートモード / 網羅性ビュー の描画
```

## 1. 動機：「台帳の単位」と「認知の単位」のギャップ

ChangeEvent の粒度は説明台帳のために設計されている。全 RawDiff を漏れなく
紐付けられる、検証可能な最小単位——例えば減便は（路線×方向×曜日×時間帯
ビン）ごとに1イベント——である。この粒度は監査には適するが、人が読む単位
ではない。実際、イベントを1行=1表示でそのまま描画した初期ビューアでは、
同じダイヤ改正の話が時間帯ごとに分裂し、1つのパターン変化から派生した
兄弟イベント（区間延長とそれに伴う時刻変更など）がページのあちこちに
散らばって、全体像が読み取れなかった。

「台帳として正しい単位」から「人が読む単位」への変換層が必要である。
その設計原則は5つ（docs/design/presentation.md、要件凍結 R1〜R19）：

1. **コア不変**：L0/L1/L2・JSON スキーマ・説明台帳には手を入れない。
2. **注釈主義**：表示に必要な追加分析は新イベントタイプにせず、表示用の
   導出情報としてプレゼン層で計算する。イベント体系（ontology）を表示の
   都合で動かさない。
3. **検証への導線を残す**：イベント羅列・RawDiff・explained_ratio は
   検証モードに退避するが、削除しない。
4. **決定的ルールの維持**：要約文までテンプレート＋決定的ルールで生成する。
   同じ入力からは常に同じレポートが出る。
5. **色は常に第2チャネル**：情報は太字・記号（▲▼・新/廃）・数値を第1チャネル
   とし、色はその補強に限る（色覚多様性への配慮）。

## 2. 路線ページの合成：プレゼン層が「更に」計算するもの

レポートの中心は路線（route_group＝枝番系統を束ねた「路線ブランド」）ごとの
ページで、① 路線概要 / ② 変化のサマリー / ③ 時間帯別本数 / ④ 新旧時刻表 の
4部で構成される。この合成のためにプレゼン層が行う導出を順に見る。いずれも
「新しい事実の認定」ではなく、コアが認定した事実の**組み直し**である。

### 2.1 ページ単位の統合：identity の対応成分に従う

章立ての単位は route_group だが、世代間で改称・統合・分割と判定された路線は
**ページを新世代側に統合**する。旧 family は L1 の対応成分（前編 §3.3）に
従って新側ページへ写り、ページ冒頭に「旧名称：…」が注記される。朝日町営バスの
21路線→9路線の再編では、これにより9ページ構成となり、各ページに統合前の
旧路線名が並ぶ。対応先のない廃止・新設ページには「内容が類似する路線
（対応の可能性）」が類似度付きで注記される——これは MatchGraph の
`stops_candidate` エッジ（受理閾値未満・floor 以上の弱い仮説）の唯一の
使い道であり、「仮説を破棄せず保持する」設計が表示にまで届いている例である。

### 2.2 方向グループと leg：direction_id に頼らない上下の束ね

「上り/下りを1つの見出しに束ねる」判定は GTFS の direction_id に依存しない
（実フィードでは設定が信頼できないため）。系統ペアごとに停留所集合の Jaccard が
閾値を超えたものについて**共有停留所の相対順序の整合度**を計算し、

- 整合度が逆転側 → 往復として束ねる（見出しは「A ⇄ B」）。端点の完全逆転は
  **要求しない**——上下便の端点が操車場・営業所の分だけずれるのが実フィードの
  常態である。
- 整合度が同方向側 → 区間便・経由違いとして同じ **leg**（時刻表単位）に束ねる。
- 中間域 → 束ねない（保守的）。

当初は「端点の完全逆転」を要求していたが、7フィードの実測で6フィードに
系統的な取りこぼしが出て一般化された。このとき整合度の分布は 0.00 / 1.00 に
二極化し中間域は0件——判定として際どい閾値に依存していないことの実測根拠に
なっている。leg は①の停車列・地図、③の集計行、④の時刻表の共通単位であり、
ラベル（「起点 → 終点」）も3箇所で同一語彙になる。

### 2.3 停留所軸の LCS 併合と分冊：出版時刻表を成立させる技術

④の時刻表は出版時刻表形式——1列=1便、左=朝→右=夜、行=停留所——である。
これを作るには、途中止まり・経由違いの便が混在する leg 全体で**行軸
（停留所軸）を1本に統一**しなければならない。全パターンの停車列を LCS
ベースで反復併合して1本の順序（最短共通超列に近いもの）を作り、各便は
自分が通らない停留所を「—」で埋める。プレゼン設計で最大のポイントと
された箇所である。

それでも読めない表は残る。経由が大きく異なる系統や、巡回順の違う循環変種が
同居すると、軸自体はほぼ最適でも各便の運行範囲が細切れになり、「—」だらけの
表になる（徳島バスのレビューで発覚）。対処はトップダウンの**分冊**である：
まず従来どおり1枚で作り、「飛び（経由行の途切れ）/便」がトリガー閾値を
超えたときだけ、併合しても飛びが増えない範囲でパターンを束ね直して別葉に
分け、識別停留所による「A経由・B経由」（同一停留所集合なら「◯◯先回り」）の
ラベルを付ける。重要な性質は**読める表は一切変わらない**ことで、検証済み
フィードの多くは分冊ゼロのまま通過する。分冊は④の描画上の別葉にすぎず、
便数合計は③の leg 行と常に一致する。

### 2.4 ④の差分表示は TripDelta の直接の写像

時刻表は 旧 / 新 / 差分 の3切替を持つ。差分表示では、変更されたセルが
太字＋記号、廃止便の列が取り消し線＋「廃」、新設便の列が下線＋「新」になる。
ここで新旧の便（＝列）の対応付けに使われるのは、コアの trip 照合結果
（前編 §4、TripDelta の exact / modified / removed / added）**そのもの**である。
「6:37の便が6:29に繰り上がった」は、コアで modified と判定された対が同じ
列に載り、セルの時刻が新旧で異なる、という形でそのまま可視化される。
表示層が独自に列を組むことはない。

### 2.5 便ラベル：検出と表示で分類器を共有する

対応づいた各便対は、代表1ラベルに分類される：
変更なし / 微調整 / 時刻変更 / 区間延長 / 区間短縮 / 経由変更、＋対応の
ない便に 新規 / 廃止。優先順位は**経路系＞時刻系**——経路の変化はほぼ必ず
時刻の変化を伴うので、代表を1つにしても情報が落ちず、集約時の二重計上も
避けられる。微調整と時刻変更の境界は config の閾値（既定5分）。

実装上の要点は、停車列変化の分類に B 群イベント検出と**同じ関数**
（`classify_sequence_changes`）を使っていることである。②のサマリーが言う
「経由変更」と④の列で見える「経由変更」が、定義のレベルで一致する。
この便ラベルは、路線一覧の折りたたみヘッダ（曜日別便数 旧→新＋質的チップ：
◆新設/廃止・◆系統再編・○経由変更・＊ダイヤ変更）と、②冒頭の一言
ダイジェスト（優先順：路線新廃＞系統再編＞経路系＞便数増減＞時刻系、上位
3件）の土台にもなる。ダイジェストは言語中立の構造化事実としてバンドルに
持ち、文章化は表示側で行う（国際化対応の布石）。

### 2.6 Lev.1〜5：サマリーのプレゼンテーションカスケード

②の変化サマリーは、検出カスケードと**同型の吸収規則**——上位が説明したら
下位は個別に言わない——を持つ5レベルで生成される。

| レベル | 内容 | 束ね・吸収規則 |
|---|---|---|
| Lev.1 | 路線全体の新設・廃止 | これがあるとき Lev.2〜5 は省略（①の概要が新設/廃止時点の姿の説明を兼ねる） |
| Lev.2 | 運行系統の変化（系統の出現・消滅・実質変更） | **対応するイベントは存在しない**。L1 のパターンクラスタ新旧集合とリンクから導出する——注釈主義（原則2）の代表例 |
| Lev.3 | 経由停留所の追加・削除 | 同一の変化内容なら方向・曜日を跨いで1件に束ね、**影響率（変化した便/全便）を必ず添える**。「全48便が経由変更」と「48便中12便」は別の話だからである |
| Lev.4 | 増減便 | 曜日単位で「平日: 純減2（増3・減5）」 |
| Lev.5 | それ以外 | 「時刻の微調整 N 便」に集約。詳細は④の差分表示が受ける |

サマリーが大きな変化に絞れるのは、網羅性を④（全時刻の差分表示）と検証モードが
担保しているからである。台帳の思想——「絞った表示」と「漏れのない記録」を
別のレイヤーに分けて両立する——がここでも効いている。

### 2.7 時間帯別本数と曜日タブ

③は 方向グループ → 曜日 → leg → 系統 の入れ子で時間帯別本数（旧→新）を
示す。集計行が先、内訳が後で、**変更なしの行も表示**する。セルは数値と
増減記号（▲▼）が第1チャネル、色濃度は補強である（原則5）。

曜日（day_type）は「平日→土曜→日祝→土日→毎日→曜日指定（dow_*）→特定日→
運行日なし」の固定順で常に並び、路線ページには「平日 16便→12便▼」形式の
曜日タブが付いて③④を選択曜日に絞り込める。廃止された運行日（旧>0・新=0）も
タブに残る——消えたものを見えなくしない、という一貫した態度である。

## 3. レポート全体の4部構成と「受け皿」

路線ページ群の外側で、レポート全体は4部で構成される。

1. **フィード全体の変化** — ファイルごとの新旧対応表（有無・行数・RawDiff
   内訳）、サービス期間、曜日区分ごとの便数 旧→新、運賃などフィード級の
   イベント。
2. **停留所の変化** — D 群イベントを**停留所クラスタ単位**に束ねる（同一
   停留所の改称＋移設は1項目）。地図に旧新位置を示し、影響を受ける路線ページと
   相互リンクする。路線ページの Lev.3 だけでは改称が「−1/＋1」（旧名の削除＋
   新名の追加）に見えてしまう問題を、この章へのリンク（「改称: 浜黒崎小学校→
   古志はるかぜ学園前」）で解消している。
3. **路線毎の変化** — §2 のページ群。**変更のない路線も載る**。「載っていない」
   と「変更なしと確認した」の区別は、網羅性を UI に翻訳したものである。
4. **その他の変化** — 第1〜3部のどこにも現れないイベントの種類別件数。
   **将来イベントタイプが増えても自動的にここに落ちる**受け皿であり、
   この部の存在自体が「レポートに無い＝変化が無い」という誤読を構造的に防ぐ。

## 4. 網羅性ビュー：3層トレーサビリティとレポート被覆率

検証モードは「網羅性ビュー」として再構築されている。核になる仕掛けが
**presentation_refs**——プレゼンテーションモデルが「どの表示単位がどの
event_id を消費したか」を記録する——で、これにより次の連鎖が閉じる：

```
RawDiff ──evidence──▶ ChangeEvent ──presentation_refs──▶ 表示先
（生差分）            （意味の割当）                     （画面上の位置）
```

網羅性ビューでは全イベントが「路線ページ（◯◯線の Lev.2 に反映）/ 停留所
ページ / フィード章 / **どこにも出ない（検証のみ）**」という表示先で分類され、
説明台帳の explained_ratio と対になる第2の計測値として**レポート被覆率**
（表示先が「どこにも出ない」でないイベントの割合）が算出・表示される。
また全 RawDiff がファイル・行・値の形で列挙され、各行から説明イベント→
表示先まで辿れる。

つまり網羅性は二段構えになっている：説明台帳が「差分→イベント」の網羅を
**保証**し（explained_ratio）、レポート被覆率が「イベント→画面」の網羅を
**可視化**する。前者は構造的保証、後者は計測値という違いも意図的である
——すべてのイベントを画面に出すことは目的ではなく（それは初期ビューアの
失敗に戻ることだ）、出していないものを**把握していること**が目的だからである。

## 5. 規律の由来：表示層に対応付けをさせると嘘をつく

「同定はコアに一元化し、プレゼン層は導出だけを行う」という規律は、最初から
あったものではない。逆をやって失敗した記録が2つ残っている。

**表示用ペアリングの廃止（2026-07-08）。** かつて④の時刻表は、コアで対応の
つかなかった廃止便×新設便を「発時刻の近さ（60分以内）」で表示層が後付け
ペアリングして1列に組んでいた。これは trip 照合が ID 頼みだった時代の
補修であり、照合が内容主導（v2）になった時点で廃止された。表示と検出で
対応付けが食い違うと、②のサマリーが「廃止」と数えた便が④では「時刻変更」に
見える、という嘘が生じる。修正後は④・Lev.3・Lev.5 のすべてがコアの割当
結果（delta.modified）を直接使う。

**増減便の定義改訂（2026-07-10）。** ②の Lev.4（増減便）は当初「時間帯ビン別の
本数差分の符号別合計」で数えていた。便対応が内容主導になってからこの定義を
検証すると、(a) 同一時間帯内の本当の廃止＋新設が相殺されて見えなくなる
（朝日町・大家庄：新2廃1 が「増1減0」になった）、(b) 時間帯を跨ぐ時刻変更が
増1減1に二重計上される（藤塚線：新1 が「増2減1」になった）、という2種の
歪みが見つかり、④の 新/廃 マークとも矛盾した。定義は「便ラベルの added /
removed の件数」に改められ、純増減 = 新旧便数差となって、ページヘッダ・
④の列マーク・ダイジェストが構造的に一致するようになった。時間帯ビン別の
内訳（シフトによる時間帯間の移動を含む）は③が担い続ける。

2つの教訓は同じ一文に帰着する：**事実（何と何が同じで、何が増減したか）は
コアが1回だけ決め、プレゼン層はその事実の見せ方だけを工夫する**。この分離が
あるから、レポートの表現をどれだけ作り込んでも説明台帳の検証可能性は
劣化しない。

細部にも同種の規律が現れる。路線ページの地図はかつて「基底名→座標」の表を
先勝ちで引いていたため、「教会前」のような同名別所の停留所（茅ヶ崎の実例では
約3 km 離れた2箇所）で別路線の座標が混入した。修正は (family, 基底名) を
第1キーとする引き当てへの変更で、ここでも「クラスタリング（コア）は正しく
別クラスタにしていたのに、表示層の手抜きな引き当てが事実を歪めた」という
同じ構図である。

## 6. まとめ：全層の一覧と、変化が画面に届くまで

前後編を通した全体像を1つの表にまとめる。

| 層 | 出力 | 対応付け・束ねの根拠 | 役割 |
|---|---|---|---|
| L0 RawDiff | 生差分の台帳（分母） | ファイル別主キー | 無解釈の全記録 |
| L1 MatchGraph | 4種エンティティの対応仮説 | 内容の複合スコア（名称・座標・接続・停車列） | 同一性の復元 |
| trip 照合 | TripDelta | 内容署名＋コスト最小割当（ID は弱い事前） | 便の同一性 |
| L2 イベント | ChangeEvent＋evidence＋残差 | ルールカスケードと消費規約 | 意味の割当 |
| プレゼン層 | プレゼンテーションモデル | 方向グループ・leg・LCS 軸併合・便ラベル・Lev.1〜5（すべて表示用導出。同定はコアを参照） | 認知単位への再構成 |
| viewer | 単一 HTML | presentation_refs | 描画と、イベント→表示先のトレーサビリティ |

最後に、1つの変化が全層をどう貫くかを「ある便の時刻変更」で通しておく。
stop_times のセル差分、あるいは ID ごと振り直された trips / stop_times の
行の削除＋追加として台帳に載り（L0）、内容署名とコスト割当で新旧の便が
対応づけられ（trip 照合・modified）、時刻差が一様なら TIMETABLE_SHIFTED、
非一様なら区間別の TRAVEL_TIME_CHANGED として当該セル群を evidence に取る
（L2）。プレゼン層では便ラベル「時刻変更/微調整」に分類されて②の Lev.5 に
