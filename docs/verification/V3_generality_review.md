# V3 汎用性レビュー: レビュー対応が「一般ルール」から逸脱していないかの監査

実施日: 2026-07-06。V3 のレビュー対応 (第1〜6ラウンド) は特定フィードの気付きから
出発した修正の連続であるため、(a) 特定データへの特化になっていないか、
(b) 一般ルールとして定式化されているか、(c) 他フィードを劣化させていないか、を監査した。

## 監査結果 (修正ごと)

| 修正 (出発点のフィード) | 定式化 | 判定 |
|---|---|:--|
| 短期間サービスの特定日分類 (永井・年末年始) | 「運行日数 ≤ config 閾値の calendar_dates サービスは irregular」— フィード名・日付に依存しない | ✓ 下記の横断検証で裏付け |
| 分単位の差分判定 (地鉄・秒差) | 「表示粒度で比較する」一般原則 | ✓ |
| 通過/経路外/不存在のセル書き分け (豊前) | データモデル上の区別 (null / "" / 世代別存在) に基づく | ✓ |
| Lev.3 の (追加集合, 削除集合) マージ (豊前) | 停留所集合キーの一般規則。路線名等への参照なし | ✓ |
| Lev.3 素材の trip_delta 化 (豊前・direction_id 無し) | 一般化だが**回帰を発生** (下記) | ✗→修正済み |
| 表示用ペアリング (豊前・地鉄) | 「同一バケット内で発時刻差 ≤ config 閾値」の決定的規則 | ✓ |
| key_stops (地図ラベル・概要省略) | 起終点/ハブ (通過路線数)/分岐点 — 全てトポロジ由来 | ✓ |
| 地図レーンオフセット | 進行方向右側レーンの一般手法 | ✓ |

**特定の路線名・停留所名・事業者 ID への参照はゼロ** (grep で確認可能)。
全ての閾値は config/default.toml。各修正の出発点フィードは検証ログに記録されている。

## 発見した回帰 (修正済み)

**Lev.3 素材の trip_delta.modified 限定化** (豊前対応・第2ラウンド) により、
trip_id が張り替わるフィードでは経路変更が Lev.3 の影響便数から漏れていた:
地鉄フィーダーバスの影響便数が 41便相当 → 31便に減少 (表示ペアリングで組んだ
rerouted 対が未集計)。

修正: 表示用ペアリングを時刻表構築時に一度だけ計算し、**Lev.3 / Lev.5 と共有**する
構造に統一。フィーダー 86/86便 (coverage 1.0)、豊前 66/110便 (デグレなし) を確認。
回帰テスト (test_level3_includes_display_paired_trips) を追加。

教訓: 「A フィードの問題を直す修正は、その問題が起きない B フィードで
何が変わるかを確認する」— 以後のレビュー対応では検証3フィード+豊前での
差分確認を修正ごとに行う。

## day_type 短期間ルールの横断検証

調査コーパス 60 フィードでルール有無の分類差分を計測:
irregular へ倒れたのは **4サービスのみ** (weekday 3 / saturday 1、いずれも
年末年始型の極短期)。境界例 (運行日 8〜10日) は **0件**。
正規ダイヤの誤特定日化は観測されず、ルールは経験的に安全。
既知の理論的エッジ: 有効期間が2週間未満の calendar_dates 全列挙フィードでは
正規ダイヤが特定日化しうる (調査コーパスには存在せず。検出したら閾値を
期間比に変える)。

## 構造評価とリファクタリング判断

- report/presentation.py は約950行・_Builder 集中型。責務は明確に分かれており
  (系統/方向グループ/サマリー/本数/時刻表)、**現段階での分割は見送り**
  (V3 のレビューが続いており、画面要件が動く間はファイル分割の利益が薄い)。
  V3 クローズ時に overview / summary / timetable の3モジュール分割を検討
- 「早すぎる最適化」への抵触なし (性能変更は行っていない)
