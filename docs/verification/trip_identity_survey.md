# 便の同一性判定の横断評価 (2026-07-08)

動機: ユーザーレビューでの「同一性を緩く見過ぎでは」という問題提起。
②同一 trip_id 規則 (tripdelta.modified) と ③表示用ペアリング
(発時刻差 ≤ 60分・類似ゲートなし) を定量評価した。
**この文書は計測の記録であり、閾値変更等の判断は含まない** (ユーザー指示)。

## 方法 (scripts/survey_trip_identity.py)

- コーパス: 層化サンプリング 47 + 検証フィード 7 = 54 フィード (ok 41)
- **擬似正解**: ID が安定運用されているフィード (id_overlap ≥ 0.7 かつ 同一ID対の
  停車列 Jaccard 中央値 ≥ 0.9) 11件・同一 trip_id の modified 対 1,394 組を正解とし、
  **trip_id を隠して** ③と同じ貪欲ペアリングを実行 → ID 対応と比較
- グリッド: 発時刻差 {10,15,30,60}分 × 停車列 Jaccard ゲート {なし,0.3,0.5,0.8}

## 結果

### ③ 表示用ペアリングの精度 (グリッドはほぼ平坦)

| 時間閾値 | ゲートなし | Jaccard≥0.8 |
|---|---|---|
| 10分 | P=0.910 / R=0.904 | P=0.914 / R=0.907 |
| 15分 | P=0.909 / R=0.910 | P=0.913 / R=0.914 |
| 60分 (現行) | P=0.907 / R=0.915 | P=0.912 / R=0.918 |

閾値をどう動かしても P/R は ±0.01 しか変わらない。

### 誤ペア 130 組の内訳 (60分・ゲートなし) — ここが本質

- **122 組 (94%) は「内容ほぼ同一の入れ替わり」(Jaccard ≥ 0.9)** — 例: 八戸
  旭ヶ丘線で、同停車列・同時刻帯の便同士を組んだが ID 上は隣の連番が正解、という
  もの。表示上は正解と区別がつかない (無害)。
- **実質的な誤対応は 8 組 (全予測 1,448 組の 0.6%)**。全て八戸で、例:
  - 八戸ニュータウン・高専線 平日: 旧 7:16 大杉平バスセンター→旭ヶ丘営業所 (45停)
    × 新 7:16 鮫小学校通→聖ウルスラ学院 (37停) — **Jaccard 0.17**。発時刻が
    たまたま同じ別系統の便を「時刻変更」として1列に重ねている
  - 同線 土曜: 旧 19:10 旭ヶ丘→中心街 × 新 19:15 中心街→ラピア (逆方向感のある
    別区間、Jaccard 0.71)
  - 日計線 土曜: 22停の便 × 47停の便 (Jaccard 0.31)
- この 8 組の Jaccard は 0.17〜0.88 で、**ゲート 0.8 なら 6 組、0.9 なら 8 組全てが
  除外される** (グリッド上ゲート追加で recall は下がらず微増)。統計値 (P) が
  ほぼ動かないのは無害な入れ替わり 122 組が分母を占めるためで、**人が「おかしい」と
  感じる種類の誤りはこの 8 組に集中**している。

### 正解対 (同一 trip_id で内容が変わった便) の分布

- 発時刻差: **0分が 60.8%、≤15分 92.0%、≤30分 94.5%**、p99=124分、最大605分
- 60分閾値が効いた実例: 明石 たこバス清水西ルート 10:36→11:07 (31分シフト) は正解
- 停車列 Jaccard: 中央値 1.0、p10 0.944、**0.5 未満は 0%**
- 方向グループ/leg バケットを跨いだ同一ID対 (③では原理的に組めない): 14 / 1,534

### ② 同一 trip_id 規則 (modified) の健全性

コーパス全体の同一ID対 1,534 組で **停車列 Jaccard < 0.3 は 0 組 (0.00%)**。
懸念していた「ID 使い回しで別便が modified になる」事例は今回のコーパスでは
観測されなかった。

## 限界・注意

- 擬似正解は「ID 安定フィード」11件に基づく。③が実際に働くのは ID が不安定な
  フィードであり、そこでの挙動は転移仮定 (時刻・停車列の性質は同様) に依る
- 「無害な入れ替わり」の判定は Jaccard ≥ 0.9 の便宜的定義
- ②の評価は「今回の 41 フィードで観測されず」であり、ID 使い回しフィードの
  非存在証明ではない
