# V5 検証: 網羅性ビュー (検証モードの再構築)

実施日: 2026-07-07。設計: docs/design/coverage_and_nonroute.md V5。
ユーザー指示: データ作者が微細な部分も含め変化の全体を把握できるよう、
**GTFS のデータ形式に近い表現**で、**網羅性を軸に**検証モードを再構築する。

## 実装

検証モードを3段構成に全面差し替え (旧 W1 の Summary / RouteChapters /
StopsChapter / Validation は撤去 — 路線単位の羅列はレポート第3部と重複するため):

1. **網羅性 (説明会計)** — 2つの被覆率を先頭に:
   - RawDiff 被覆 (explained_ratio、従来からの会計の背骨)
   - **レポート被覆率** (新設): イベントのうちレポート第1〜3部で個別に
     説明される割合。第4部行き (件数のみの提示) を「個別説明なし」と数える。
     表示先別の内訳と、未説明残差 (UNEXPLAINED_RESIDUAL) の一覧を併置
2. **イベント一覧 (表示先別)** — 全イベントを表示先 (第1部/第2部/第3部は
   route_group ごと/第4部) で分類して列挙。各行は既存 EventRow で、クリックで
   evidence → RawDiff 生値に到達できる (説明会計への導線は維持、V3 DoD 互換)
3. **ファイル別の生差分 (GTFS 形式)** — 全 RawDiff を txt ファイル → 差分種類
   (行追加/行削除/値変更/列変更/ファイル増減) → 行 (キー・カラム・旧値→新値)
   で列挙。**各行に「説明イベント → レポートの表示先」を添える** =
   RawDiff → Event → 表示先 の3層トレーサビリティ。大テーブルは段階表示
   (100件 + もっと見る)、「未説明のみ」フィルタ付き。何も省かない

## モデル側 (bundle.py)

- `_event_destination(e)`: イベント → 表示先 (part 1〜4) の決定的対応。
  型 (カテゴリ) と subject (route_group) による粗粒度対応 — 「全イベントに
  表示先がある」ことをまず保証する。表示単位レベルの精密な refs は将来の精緻化
- `presentation.coverage`: destinations 台帳・表示先別件数・レポート被覆率
- 第4部 (feed_overview.others) の集計を同じ対応関数に一元化 (乖離が構造上
  起きない)。テストで一致を保証
