# M9 検証ログ: family 世代間対応の内容主導化 (route identity v2)

日付: 2026-07-09 / 設計: docs/design/route_identity_review.md / roadmap M9

## 実装サマリ

- `identity/route_family.py`: 名称不一致 family の対応を「停留所クラスタ対応で
  旧基底名を翻訳した上での停留所集合 Jaccard」に置換 (旧: パターン鍵 Jaccard)。
  受理 `stops_translated` (≥0.5、route_id 共有で −0.1 緩和) / 候補
  `stops_candidate` (≥0.3、注記専用)。受理エッジの連結成分を
  改称/統合/分割/再編に分類。group 数 > 6 の成分は **2段階降格**
  (best-match 間引き → 再分解 → それでも超過なら注記に全降格)
- `events/rules/routes.py`: 成分 → ROUTE_RENAMED / MERGED / SPLIT /
  RESTRUCTURED (v0.2.2)。trip 照合ブロックは成分で束ねた group
  (`blocking_family_maps`)
- `report/presentation.py`: ページは新世代背骨 (`page_family_maps`)。
  旧名称注記 (former_names)・廃止/新設ページの類似候補注記 (similar_candidates)。
  bundle に lev1_trip_ratio (煙感知器、検証モードに表示)
- 横断監視: scripts/survey_route_identity.py

  {役場便,市振線3変種}→A2市振線 の2成分に分かれ全て受理。

  市振線側に付く。名前の示唆とは異なるが誤差レベルの内容差で、
  former_names 注記により人が確認できる (N:M グレーゾーンの想定内挙動)。

## 判断メモ

- **2段階降格 (best-match 間引き) の追加** は初期実装の実測 (朝日町の
  類似度 1.0 対の巻き添え降格) を受けた同日の設計改良。
  route_identity_review.md §3.3.1 の防御表と整合 (「有界な退化」の内側)。
- 役場便型の誤割当は追わない (内容差が誤差レベルの N:M は注記で可視化する、
  という設計判断のとおり)。

## 追記 (2026-07-10、朝日町ユーザーレビューからのフォロー修正)

ユーザーのチェックで南保線の便数不整合 (ヘッダ 8→4 / ④時刻表 5→4 /
③本数表 2→4 / Lev.4 純増2) が発覚。原因は presentation の `_systems` が
旧クラスタ→新クラスタのリンクを **1:1 前提**で処理しており、M9 の family 統合で
生じる N:1 リンク (始発便・本便・最終便の3旧クラスタ → 統合後の1新クラスタ) の
2つ目以降を落としていたこと。③は先頭クラスタの便しか数えず、④は locate 失敗で
列ごと消えていた。

- 修正: `old_system_ids` (リスト) を導入し全旧クラスタを新 system にぶら下げる
  (548ec00)。③④・地図・主要停留所の各消費者をリスト対応に
- 再発防止: **整合の不変条件** — ヘッダ (day_totals) と④の便数が day_type ごとに
  一致しなければ警告ログ。合成テスト (test_merged_families_keep_all_trips_in_sections)
- 検証: 朝日町 南保線/山崎線/市振線 でヘッダ=③=④ が一致、4フィードで
  整合警告ゼロ・explained_ratio 不変
