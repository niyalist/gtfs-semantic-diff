# M6 検証ログ: route_group 横断調査 (gtfs-data.jp 80フィード)

実施日: 2026-07-04
方法: `scripts/survey_route_groups.py` — 都道府県で層化した決定的サンプリング 80 フィード
(検証フィード 永井運輸・富山地鉄を強制包含)。各フィードの最新世代について
route_id → 同名 family → 語幹 group の集約と、語幹一致 family 対の停留所基底名集合
Jaccard を計測。GTFS-JP 固有フィールドは不使用。
生データ: `data/survey/results.jsonl` (80行) / 集計: `scripts/analyze_route_group_survey.py`

## 主要な数値

| 指標 | 値 |
|---|--:|
| 多family語幹 (30A/30B型) を持つフィード | 5 / 80 (6%) |
| 語幹一致 family 対の総数 | 60 対 |
| うち停留所 Jaccard < 0.2 | 16 対 (27%) |
| うち Jaccard ≥ 0.6 | 30 対 (50%) |
| family 内に停留所がほぼ重ならないパターンクラスタ対 (分割候補) | 19 対 / 5 フィード |

集約が大きいフィード上位: 永井運輸 (24 family → **7 語幹**、多family語幹 6)、
富山地鉄 (44 → 42)、安中市 (4 → 3)、恵那市 (23 → 22)、熊本市交通局 (2 → 1)。

## 発見 1: 30A/30B 型は少数派だが、該当フィードでは支配的

該当は 80 中 5 フィード (6%) にとどまり、コミュニティバス (深浦町・七ヶ浜町・本宮市等)
の多くは 1 route = 1 名称で問題自体が存在しない。一方、永井運輸では family の 7 割が
語幹で束なる (24 → 7)。**「まれだが、起きるフィードではレポート構成を支配する」**
という性質であり、route_group 層は「該当しないフィードでは単独 family group に
無害に退化する」設計 (route_group.md) が適切と確認できた。

## 発見 2 (設計変更): 停留所 Jaccard は AND ゲートに使えない

語幹一致対の Jaccard 分布は二峰性 (0.6 以上 50% / 0.2 未満 27%) だが、低い側の
16 対中 15 対は**永井運輸の前橋玉村線の枝番同士**だった。実データ確認:

- `30A 前橋玉村線` (39停留所、前橋駅側) と `30B 前橋玉村線` (21停留所、新町駅・上武大側)
  の共通停留所は**「玉村町役場」1つのみ** (J=0.017)。

つまり枝番系統は「同一ブランド名の下の**別コリドー**」であることが普通にあり、
当初案の「語幹一致 かつ 停留所 Jaccard ≥ 閾値」というゲートは、
**この機能の動機となった当のケースを棄却する**。よって設計を次のように変更する:

1. **グループ化は語幹一致のみで行う** (ブランドは名称の概念であり、幾何の概念ではない)。
2. **停留所 Jaccard はゲートではなく「凝集度 (cohesion)」メタデータ**として算出・保持し、
   レポートの group 章冒頭に構成 family と凝集度を明示する (低凝集 = 枝線構造の情報であり
   エラーではない)。

## 発見 3: 誤結合リスクの実体は「退化した語幹」

語幹一致による誤結合の実例は、地理的に無関係な路線同士ではなく、
**語幹が一般語に退化するケース**だった:

- 熊本市交通局: `A系統` / `B系統` → 語幹「系統」(一般語)。

対策は Jaccard ゲートではなく**語幹の品質ガード**で足りる:
最小語幹長 (2文字) + 一般語ストップワード (「系統」「線」「循環」「バス」等) に
該当する語幹はグループ化しない (元名のまま単独 group)。
その他の要注意例 (`17 藤線` → 「藤線」、kuma-bus の経路記述型長名) は正しく動作していた。

## 発見 4: 分割候補 (同名 route 内の別系統同居) — 4類型

family 内に停留所がほぼ重ならない (J < 0.1) パターンクラスタ対は
**5 フィード・6 family** (80 中 6% — 統合方向と同率)。代表パターンの起終点を
実データで確認したところ、性格の異なる4類型に分かれる:

| 類型 | 実例 | 内容 |
|---|---|---|
| 1. ブランド名下の複数コリドー | 富山地鉄「富山港線フィーダーバス」 | c0 四方神明町→蓮町 (10停) と c1/c2 水橋町↔岩瀬浜駅前 (17停)。共通停留所ゼロ。「フィーダーバス」という役割名がブランド化し別コリドーが同居 |
| 〃 | ちばレインボーバス「神崎線」 | 木下駅→船尾車庫 (25停) と 津田沼駅↔船尾車庫 (48停)。共有は船尾車庫のみ (J=0.015) |
| 2. 本線 + 乗継ぎ区間便 | 琴参バス「美合線」 | 琴平営業所↔落合橋の本線 (47停・30便) と 落合橋↔三角の山間区間便 (6停・20便)。接続点の落合橋のみ共有。1便のみの枝 (川奥・下福家) も同居 |
| 3. コミュニティバスの複数コース同居 | 鹿児島市「あいばす」④⑤⑧⑩ | 各「地域」が1 route で、中に停留所がほぼ交わらない複数コース |
| 4. 往復で経路が別 | 伊賀市「行政サービス巡回車 柘植・西柘植線」 | 上下の J=0.037。往路と復路が別道を巡回 |

**結論**: 分割の実体は既存の family ⊃ 方向 × パターンクラスタ階層で検出・保持済みで、
新たな検出層は不要。不足は**見せ方**であり、統合 (発見 2 の凝集度) と対称に扱える:
family 内クラスタ間の凝集度が低い場合、レポートの family 章を**パターンクラスタ単位の
小見出しに自動分割**する (美合線なら「本線 (琴平営業所〜落合橋)」と「区間便
(落合橋〜三角)」が別見出しになる)。M7 の report 変更に含める (route_group.md 参照)。

将来課題: C群 (便数集計) の subject にパターンクラスタを付し「区間便だけ減便」を
直接表現できるようにする (M7 の動作実績を見てから判断)。

## 閾値・規則の決定 (M6 DoD)

| 項目 | 決定値 | 根拠 |
|---|---|---|
| 語幹抽出 | NFKC 正規化 → 先頭の英数字・空白・ハイフン・中点等の連続を除去 | 調査で誤除去 4/約660 family (すべて許容範囲) |
| 最小語幹長 | 2 文字 (未満なら元名を語幹とする) | 「藤線」(2文字) は正当な路線名 |
| 語幹ストップワード | 系統・線・循環・バス (該当時はグループ化しない) | 熊本市電 A系統/B系統 の退化例 |
| 停留所 Jaccard | **ゲートに使わない**。group 凝集度として算出しレポートに明示 | 発見 2 |
| 凝集度の注記閾値 | median pair Jaccard < 0.2 で「枝線構造」注記 | 分布の谷 (0.2-0.3 が最疎) |

route_group.md の設計本文にこの決定を反映済み。M7 はこの仕様で実装する。
