# Route Group 設計 (family の上の「路線ブランド」集約層)

状態: **M6 横断調査完了・仕様確定** (実装は M7)。
2026-07-04 の議論から起こし、同日の 80 フィード調査
(docs/verification/M6_route_group_survey.md) の結果で「シグナル」節を改訂した。

## 動機

Route Family (route 表示名の完全一致グループ) は「同名で route_id が細分される」型の
フィード (例: 富山地鉄 172 route_id → 44 family) は吸収できるが、
**枝番付きの系統名が別 route として定義される**型は吸収できない。

実例 (永井運輸): `30A 前橋玉村線` `30B 前橋玉村線` `30E 前橋玉村線` `30H …` `30J …`
`30K …` がそれぞれ別 family となり、レポートで6章に分かれる。利用者の認識は
「前橋玉村線」1本であり、納得感を欠く。

さらに検出品質にも影響する:
- 便が 30A → 30E に付け替わると、現状は「30A で減便 + 30E で増便」と出る。
  group 単位なら「前橋玉村線として本数維持、区間便への振替」と読める。
- `TRIPS_TRUNCATED` (full パターン → truncated パターンへの本数移動、ontology C群・
  未実装) は group 単位の視点がないと原理的に検出できない。

GTFS の route 粒度は事業者・作成ツール依存でバラバラであり、「利用者が認識する路線」
との対応をデータから再構成する層が必要、というのが本設計の前提。

## 設計方針

**family を置き換えず、上に1階層足す。**

```
route_group (前橋玉村線)          ← 本設計で新設
  ⊃ route_family (30A / 30B / …)  ← 現行のイベント subject 単位 (変更しない)
    ⊃ 方向 × パターンクラスタ      ← 現行の下位構造 (「分ける」方向は既に対応)
```

- イベントの subject・説明会計・既存 JSON フィールドは変えない (安定インタフェース維持)。
  subject に `route_group` を**追加** (additive change)。
- 第一適用先はレポートの章立てと C群の集計補助。判定ロジックの subject 単位変更
  (group 単位の TRIPS_TRUNCATED 等) は動作実績を見てから。

## グループ化シグナル (M6 調査で確定)

**語幹一致のみでグループ化する。** 当初案の「語幹一致 かつ 停留所重なり ≥ 閾値」は
調査により棄却した — 枝番系統は同一ブランド名の下の**別コリドー**であることが普通にあり
(実例: 30A/30B 前橋玉村線の共通停留所は玉村町役場の1つのみ、J=0.017)、
Jaccard を AND ゲートにすると動機となった当のケースを棄却してしまう。

1. **語幹抽出** (グループ化キー): NFKC 正規化 → 先頭の系統コードらしき連続
   (英数字・空白・ハイフン類・中点等) を除去。
   - 最小語幹長 2 文字 (未満なら元名のまま)。
   - **語幹ストップワード**: 「系統」「線」「循環」「バス」等の一般語に退化した語幹は
     グループ化しない (実例: 熊本市電 A系統/B系統 → 語幹「系統」)。
     誤結合リスクの実体はこの退化ケースであり、幾何ゲートは不要と判明した。
   - 規則・ストップワードは config で調整可能にする。
2. **停留所集合 Jaccard = 凝集度 (cohesion) メタデータ**: ゲートには使わず、
   group ごとに family 対の Jaccard (median) を算出して保持する。
   レポートの group 章冒頭に構成 family と凝集度を明示し、
   median < 0.2 なら「枝線構造 (コリドーが分かれる)」と注記する。透明性の装置。

**採用しない (プロジェクト方針)**:

- `routes_jp` の `jp_parent_route_id` などの **GTFS-JP 固有フィールドには依存しない**。
  GTFS-JP の今後の改訂で非推奨方向にあるため、仮にデータに存在しても
  同定・分類ロジックの入力にはしない (CLAUDE.md 開発ルール参照)。
  ※ L0 diff がこれらのファイルを列挙・会計すること自体は網羅性の要請であり継続する。

## アルゴリズム (M7 実装仕様)

```
identity/route_group.py
  1. 各 family の表示名から語幹を抽出 (NFKC → 先頭コード除去。最小長・ストップワード
     ガードに落ちたら語幹 = 元名)
  2. 語幹が同じ family を1つの route_group にまとめる (全 family が必ずどこかに所属、
     単独 family は自身のみの group)
  3. group ごとに凝集度 = 構成 family 対の停留所基底名集合 Jaccard の median を算出
  4. group 名 = 語幹
```

config (M7 で追加):
- `identity.route_group.min_stem_len = 2`
- `identity.route_group.stem_stopwords = ["系統", "線", "循環", "バス"]`
- `identity.route_group.low_cohesion_note = 0.2` (レポート注記の閾値。判定には不使用)

- 世代内で独立に構築し、世代間は語幹一致 + 構成 family の対応で自然にリンクされる
  (family リンクの上に乗るだけなので新しい対応問題は生じない)。
- 決定性: 処理順・命名とも family 名ソートで固定。

## 消費側の変更 (M7)

- `context.band_profiles` に route_group を併記 → report の路線別章を group 単位に変更
  (group 内では family / 方向 / パターンごとの内訳を小見出しで維持)。
- events: subject に `route_group` を追加 (全 A/B/C 群イベント)。
- 将来 (M7 の DoD 外): group 単位の TRIPS_TRUNCATED、group 内 family 間の振替検出。

## 検証計画 (M6: gtfs-data.jp 横断調査)

実装前に、読み取り専用の調査スクリプトで全国フィードを横断計測し、
シグナルと閾値をデータで裏付ける:

1. サンプル: gtfs-data.jp の全フィード一覧から 50〜100 フィード
   (都道府県・事業者規模が偏らないよう選定)。
2. 各フィードで計測:
   - route_id 数 → 同名 family 数 → 語幹 group 数 (3段階の集約率の分布)
   - 「30A/30B 型」(語幹同一の複数 family) の出現フィード比率
   - 語幹一致 family 対の停留所集合 Jaccard 分布 (二峰性なら閾値が引ける)
   - 語幹は同じだが停留所がほぼ重ならない対 (誤結合リスク) の実例収集
   - 逆方向: 1 family 内パターンクラスタ間の重なりがほぼゼロの例 (分割候補) の頻度
3. 成果物: docs/verification/M6_route_group_survey.md に分布・実例・閾値提案を記録。
   閾値の最終値はこの調査結果で決め、config/default.toml に入れる。

## リスクと対処

| リスク | 対処 |
|---|---|
| 語幹抽出の過剰マッチ (例: 「〜線」だけ残る短い語幹) | 最小語幹長 + 停留所 Jaccard の AND 条件。M6 で実例収集 |
| 誤結合がレポートの信頼を毀損 | group 章の冒頭に構成 family と根拠 (語幹・Jaccard) を明示。confidence を保持 |
| フィードによっては group 化が不要 (地鉄型) | 単独 family group として無害に退化する設計 |
| 決定性の毀損 | あいまい一致は語幹抽出のみに限定し、規則は config で明示 |
