# W3-2 方向性検討: アカウント・データポリシー・継続利用される形

状態: **検討資料 (2026-07-10)。§6 のとおり 1,2,4,5,6 は決定済み、3 (URL 公開範囲) は
§7 の整理を踏まえて判断待ち。** 決定後に web.md と roadmap.md へ反映する。
前提: W3-0/W3-1 稼働済み (S3+CloudFront+Lambda ジョブ API、匿名利用・30日削除)。

## 0. 調査結果: gtfs-data.jp の「世代の絶対パス」は既にある

rid (`current`/`prev_N`) は世代が進むとずれるが、API は各世代に恒久 UUID
**`gtfs_file_uid`** を付与しており、`gtfs_url` も rid 非依存の
`.../files/feed.zip?uid=<uuid>` 形式で返す (2026-07-10 実測: 302 → S3 署名付き URL、
世代が進んでも同一 uid で同一 zip を取得できる)。

→ **リポジトリ側の API 変更は不要**。semdiff 側で meta.json に
`gtfs_file_uid / gtfs_url / from_date / published_at / feed_license` を保存すれば、
「同じ条件で再比較」「片側を最新に差し替えて再比較」が正確に実現できる。
現行の rid ベース記録は uid ベースに改める (W3-2 の実装項目)。

あると便利だが必須でないリポジトリ側改善 (提案余地): uid からメタデータを逆引きする
エンドポイント。現状でも feed 世代一覧から uid を引けるため回避可能。

## 1. アカウント方式

目的の整理: (a) 恒久保存の主体を特定する (b) 濫用抑止 (c) 履歴・通知など利便性
(d) フィードバックの返信先。**自前パスワードは持たない** (web.md 確定事項) は維持。
ログインは「保存・履歴・通知のため」であり、**比較機能自体は匿名で全部使える**
(匿名=30日削除) という現行構造も維持するのが良い。

| 案 | 中身 | 利点 | 難点 |
|---|---|---|---|
| A. Google のみ | Cognito + Google IdP | 実装最短。想定ユーザー(研究者・コンサル・オープンデータ界隈)の保有率が高い | 自治体・事業者の職場環境で Google が使えない場合がある |
| B. メールリンク | Cognito カスタム認証 + SES のマジックリンク | 心理障壁が最小。組織メールで使える。IdP 規約に縛られない | 実装工数中。自治体メールの迷惑判定・到達性リスク。SES 本番移行申請が必要 |
| C. 複数 IdP | A + Microsoft | 日本の自治体・企業は M365 が多く、A の穴を塞ぐ | Azure AD アプリ登録の運用が増える |
| D. GitHub / パスキー | — | — | 対象ユーザー層とずれる / リカバリ設計が重い |

**推奨: A で開始し、B/C は需要が見えたら追加。** Cognito はユーザープールを保った
まま IdP を後付けできる。ただし「同一メールで別 IdP」の重複アカウント問題だけは
最初に設計しておく: **email を正規化キーにし、ユーザーデータ (履歴・ウォッチ) は
Cognito の sub ではなく内部 user_id に紐づける** (IdP 追加時に自動リンク可能にする)。

## 2. アクセス記録の研究利用と規約

「ライセンス」は2つに分けて考える:

### 2a. 利用規約 (サービス利用条件 + ログの研究利用同意)

記録する項目と目的を規約に明示する:

- **記録**: 比較対象 (org/feed/世代 uid)、実行日時、結果 URL とその閲覧数、
  フィードバック内容、アップロード zip のメタ情報 (サイズ・feed_info)。
  IP は濫用対策目的に限定し短期保持 (例 90日) またはハッシュ化。
- **研究利用**: 匿名化・集計した形で学術研究 (論文・発表) に利用することへの同意。
  **メールアドレス等のアカウント情報は研究データから分離**し、公表は統計形式のみ。
- **機密データの逃げ道を明記**: 「アップロードしたデータはサーバに保存され得ます。
  外部に出せないデータは CLI 版 (pip install) / 単一 HTML 出力をローカルでご利用
  ください」をトップページと規約の両方に書く。これが「同意しない自由」の実体。
- 大学所属としての行動ログ研究になるため、**所属機関の倫理審査・ポリシー確認**を
  W3-2 の DoD に含める (集計統計のみなら軽い手続きで済む可能性が高いが要確認)。

同意 UX の選択肢: (i) 実行ボタン脇に短い同意文+詳細リンク (推奨。アップロード時
だけ明示同意)、(ii) 初回バナー、(iii) ログイン時のみ同意。

### 2b. 生成レポートのライセンス

- gtfs-data.jp 由来: 入力の `feed_license` (多くは CC BY 4.0) に従属。API が
  ライセンスを返すので、**レポートに出典 (事業者名・ライセンス・取得元) を自動表示**
  すれば CC BY の表示義務が構造的に満たせる。
- アップロード由来: 権利不明のため、結果 URL は**限定公開 (URL を知る人のみ)** が
  既定。公開一覧・ギャラリーに載せるのはリポジトリ由来の結果のみ。
- 削除依頼 (事業者・データ所有者から) の窓口と対応方針を規約に一文入れる。

## 3. GTFS ファイル自体の保存

| 入力 | 案 | 推奨 |
|---|---|---|
| gtfs-data.jp | 実体は保存せず **uid 参照** (§0)。消失リスクはリポジトリ側の保持に依存 | ✓ 現行方針を uid 化して維持 |
| アップ (匿名) | 現行: uploads/ 7日 + 結果30日 | ✓ 維持 |
| アップ (ログイン) | (O1) 保存しない / (O2) 検証用に短期 / (O3) **結果と同寿命で保存** | **O3**: 「同条件で再比較」「フィードバック検証」の実体。S3 コストは軽微 (zip 数MB〜50MB、Glacier IR なら 100GB でも月〜500円) |

研究利用との線引き: アップロード zip の**保存目的は再現・検証に限定**し、zip 自体を
研究データにはしない (権利不明のため)。研究に使うのはログ・メタデータ (§2a) のみ。

## 4. ログインの利便性 (ユーザー案 + 追加案)

ユーザー案 (確定でよい):
1. **履歴一覧**: 過去の比較結果を日付・事業者・都道府県・世代 (from_date)・
   イベント規模でソート/フィルタ。
2. **再比較**: 同条件 (uid ペア) で再実行 / 「新側を最新世代に差し替えて再実行」
   ボタン。uid 保存 (§0) が前提。

追加候補 (定着ドライバの見立て順):
3. **ウォッチ + 新着通知 (本命)**: 事業者/フィードをウォッチ → 日次で published_at を
   ポーリングし、新世代が出たら**自動で diff を生成してメール通知** (「○○バス
   2026-10-01 改正: 路線廃止1・減便12 — 結果を見る」)。R19 の一言ダイジェストが
   通知文面にそのまま使える。「改正を見逃さない」は時刻表趣味層・研究者・
   周辺自治体の全部に刺さり、**再訪の理由**を作る。
4. **改正カレンダー**: ウォッチ中フィードの from_date を時系列表示
   (「来月からの改正: 3件」)。将来の多世代タイムライン (detection.md §7) の入口。
5. **フィードバック追跡**: 自分の報告と対応状況の一覧 (報告には結果が恒久保存で
   紐づく、という既存設計の自然な延長)。
6. **メモ・注釈**: 結果ページへの私的メモ (改正の背景記事 URL など)。
7. **公開管理**: 自分の結果の限定公開/一覧公開の切替、恒久 URL の棚卸し。
8. **API トークン**: CLI からジョブ投入して URL を得る (研究・自動化ユーザー向け)。
9. (将来) 組織スペース: 自治体交通担当チームでの履歴・ウォッチ共有。

## 5. 構想の拡げ方: 「道具」から「観測所」へ

継続利用の核は「自分がアップロードする道具」ではなく
**「日本の GTFS 改正の定点観測サイト」**になること。

- 対象は全国 **594 フィード** (2026-07-10 実測)。新着世代の検出時に current↔prev_1 の
  diff を自動生成しても、1件数秒〜30秒 × 日に数件 → **Lambda コストは実質ゼロ**。
- トップページ = 「今週の改正」: 新着 diff の一覧を事業者・都道府県・改正日・
  変化の大きさ (major イベント数) で並べる。一言ダイジェスト (R19) が一覧の顔になる。
- 研究面と両輪: 全国の改正の類型統計 (何が・どこで・どの頻度で変わるか) は
  それ自体が成果で、§2a のログ研究とデータ基盤を共有する。
- **ストレージ注意**: 単一 HTML (山交級で 57MB) を全フィードに置くと重い。
  自動生成分は **hosted bundle 形式 (ビューア共通 + JSON fetch、web.md 配布モード2)**
  に切り替えてから始める。
- リスク: 「減便」が並ぶページへの事業者感情。中立的な事実提示の原則、
  フィードバック窓口、削除依頼ポリシー (§2b) で備える。

### 段階の切り方 (案)

- **W3-2 (基盤)**: Google ログイン (§1A) + 内部 user_id 設計 / uid ベースの入力記録
  (§0) / 利用規約・プライバシー・ライセンス表示 (§2) / 履歴一覧 + 再比較 (§4-1,2) /
  ログイン時の入力 zip 保存 (§3 O3) / フィードバック + SES / コストガード実績記録
- **W3-3 (観測所)**: ウォッチ + 自動 diff + 新着通知 (§4-3) / 改正カレンダー (§4-4) /
  hosted bundle 化 / トップの新着一覧 (§5)
- 以降: API トークン、組織スペース、多世代タイムライン

## 6. 決めること (チェックリスト) — 2026-07-10 ユーザー回答

1. アカウント: **決定 — Google のみで開始。** 需要が見えたとき IdP を発展させやすい形

2. ログ研究利用: **決定 — 簡易で同意コストの低い方式でよい** (実行ボタン脇の短い

## 7. 結果 URL の公開範囲 (論点3の深掘り)

### 現状の実装 (W3-1) — ユーザー認識との差分

| 入力 | job_id / URL | 性質 |
|---|---|---|
| リポジトリ由来 | `{org}__{feed}__{old_rid}__{new_rid}` → `r/…html` | **決定的 (正準 URL)**。誰が実行しても同一比較は同一 URL。org/feed/rid から推測可能 = 実質公開。恒久保存 |
| アップロード | `anon-{ランダム16進12桁}` → `r/anon/…html` | **実行ごとにランダム**。URL を知る人のみ。30日削除 |

つまり「操作ごとに独自 URL」は現状アップロード側だけ。リポジトリ由来を決定的に
したのは W3-1 の簡易キャッシュ (同一比較の再計算・重複保存を避ける) のため。

なお現行の rid ベース正準キーには欠陥がある: rid は世代が進むとずれるため、
同じ `prev_1__current` が**将来は別の世代ペアを指し、旧結果を上書きする**。
正準 URL を続けるなら uid ベースのキー (§0) への移行が必須。

### 選択肢

- **P1. 正準 URL (公開)**: 同一世代ペア → 同一 URL。キャッシュが効き、
  「この改正ペアの結果はここ」という安定した参照先になる (共有・引用・将来の
  観測所構想と親和)。元データはオープンデータ (CC BY 等) であり、URL から
  分かるのは「何のデータか」だけで「誰が実行したか」は含まれない。
  実質公開であることを UI に明示する必要がある。
- **P2. 実行ごとランダム URL (限定公開)**: ユーザーの当初認識。すべての結果が
  URL を知る人のみ。同一比較でも毎回計算・保存され、リンク先が人によって分散する。
- **P3. 折衷 (正準保存 + ランダム別名)**: 実体は正準キーで1つ、閲覧 URL は
  実行ごとに別名発行。計算は節約できるが、内容が同一かつオープンなものに
  別名を配る意味は薄く、複雑さに見合わない。

### 推奨

**リポジトリ由来 = P1 (uid ベース正準 URL・実質公開)、アップロード = P2 (ランダム・
限定公開) の現行ハイブリッドを維持し、規約と実行時 UI に「リポジトリ由来の結果は
誰でも到達しうる公開 URL になる」ことを明示する。** 判断のポイントは
「結果 URL の秘匿に守る価値のある情報が含まれるか」: リポジトリ由来の結果は
公開データの決定的な派生物 (誰が実行しても同じ) であり、秘匿する対象は実行者の
行動だけ — それは URL でなくログの問題 (§2a) として扱える。

## 8. 正準 URL とツール更新の関係 (論点3の続き・バージョニング)

正準 URL + lazy 生成にすると「URL は最初から決まっていて、最初のリクエスト時に
生成される」形になる。そこで比較ロジックが更新されたときの扱いを設計しておく。

### 前提となる既存原則

- web.md: 「結果が恒久保存されているため、(フィードバック) 報告は常に再現・検証可能」
  → **生成済み結果の黙った上書きはこの原則を壊す** (報告や論文が指す結果が
  別物に変わってしまう)。
- meta.json / events.json (config_snapshot) には生成ツールのバージョンが既に入って
  おり、結果は自己記述的。CalVer がそのままバージョンラベルになる。

### 設計: 結果は不変 (immutable)、入口 URL は最新を指す

```
r/{pair}/                    ← 入口 (共有・引用のデフォルト)。最新版を表示
r/{pair}/v/{2026.7.10}/...   ← バージョン固定 URL。生成後は不変
r/{pair}/index.json          ← 版の台帳: 生成済みバージョン一覧 + latest ポインタ
```

- **生成済みの結果は上書きしない**。ツール更新後の再生成は**新しい版として追記**し、
  index.json の latest を進める。
- **再生成のトリガも lazy**: ジョブ API に同じペアが投入されたとき、保存済み latest の
  ツール版が現行より古ければ再計算して版を追加、同じなら既存を即返す (キャッシュ)。
  バッチでの一斉再生成はしない (W3-2 時点。観測所構想を採る場合に再検討)。
- **閲覧側**: ビューアは index.json を見て最新版を表示しつつ、
  「この結果は v2026.7.9 で生成 (最新ツール: v2026.7.11)。再生成 / 旧版を見る」を
  ヘッダに出す。旧版を見たい人 (フィードバック検証・論文引用・回帰調査) は
  版セレクタから固定 URL に到達できる。
- **引用・フィードバックは固定 URL を使う**: フィードバック記録には
  r/{pair}/v/{ver} を保存する (対象が動かない)。共有ボタンは用途で出し分け
  (「このページを共有」= 入口 / 「この版を引用」= 固定)。

### 保持ポリシー

旧版は原則保持 (ニッチな利用規模では容量は軽微。hosted bundle + gzip 前提)。
膨らんだら「フィードバック・ユーザー履歴から参照されていない旧版に S3 ライフサイクル」
を足せる構造にしておくが、**計測前に作り込まない** (設計原則5)。

アップロード由来 (P2・ランダム URL) は実行ごとに独立した結果なので版管理は不要。
再比較 (§6-4) は単に新しい実行として新 URL を発行する。
