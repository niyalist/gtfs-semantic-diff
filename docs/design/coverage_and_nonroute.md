# 網羅性の提示と路線に紐付かない変化 (設計ノート)

起案: 2026-07-07 (V3 レビュー中の課題整理)。状態: **計画 — 優先順はユーザーレビュー待ち**。

## 背景

V1 (2026-07-06) で「路線 (route_group) に紐付く変更の表現」に意図的にスコープを
絞り、V2〜V3 で路線ページ (①〜④+曜日タブ) は成熟した。一方で当初から後回しに
していた領域が残っており、「全ての差分を網羅する」という本ツールの第一原則
(説明台帳) が **UI の上では** 疎かになっている。

## 課題感の整理 (2026-07-07 時点)

1. **路線に紐付かない変化の可視性**。停留所そのものの変化 (D群:
   改称・移設・新設・廃止・乗り場)、カレンダー再編 (E群)、運賃・メタデータ・
   翻訳・行先表示 (F群) は、検証モードのイベント羅列の中にしか存在しない。
   特に停留所の変化は利用者・データ作者の双方に伝える価値が高いのに、
   路線ページからは Lev.3 の -1/+1 (改称が追加+削除に見える問題、バックログ
   既載) としてしか見えない。
2. **網羅性の可視化**。explained_ratio は台帳上計測されている (検証3フィードで
   1.0000) が、**「レポート面に出た変化」と「全差分」の間の対応が見えない**。
   読者が「レポートに無い=変化が無い」と誤認するリスクがある。データ作者の
   用途では「何がレポートに出て、何が出ていないか」自体が情報。
3. **階層間の対応付けの欠落**。RawDiff ↔ ChangeEvent の evidence 導線は W1 の
   検証モードにあるが、**ChangeEvent ↔ プレゼン表示単位 (Lev.2〜5、時刻表の列)
   の対応はどこにも記録されていない**。presentation は events を消費して
   ビューを作るが、どのイベントがどの表示に反映されたかを持たない。
4. **機械 (AI・外部システム) 向けの提供形態**。JSON イベントストリームが安定
   インタフェース (設計原則3) であり、AI に diff を渡す場合は低レベル
   (events + rawdiffs) が適する。しかし bundle 内 JSON の仕様書がなく、
   検証モード頼みの「暗黙のスキーマ」になっている。

## 提示レベルの3層整理

| 層 | 対象読者 | 形式 | 現状 |
|---|---|---|---|
| 人向けレポート | 利用者・事業者・データ作者 | 路線ページ / **停留所ページ (新)** / **フィード章 (新)** | 路線のみ成熟 |
| 台帳ビュー | データ作者・本ツールの検証 | 検証モードの再構築 = **網羅性ビュー** | イベント羅列のまま |
| 機械向け | AI・外部システム | bundle JSON (events / rawdiffs / presentation) + スキーマ文書 | 実体はあるが文書なし |

方針: 同じデータ (ChangeEvent JSON が正) を、読者に応じてページ・形式を変えて
提供する。コア・台帳は変更しない。

## 提案タスク (roadmap V4〜V6・X1 として採録)

### V4: 停留所の変化ページ (人向け)

- レポートモードに「停留所の変化」章を新設。D群イベントを **停留所クラスタ単位**
  に束ねる (同一停留所の改称+移設は1項目)。各項目: 変化内容 (旧名→新名、
  移設距離)、地図 (旧新位置)、**影響を受ける route_group への相互リンク**
  (route_group ページからも当該停留所変化への逆リンク)
- 連動: Lev.3 の改称注釈 (バックログ1) をここで併せて解消 — 路線ページ側は
  「改称: 浜黒崎小学校→古志はるかぜ学園前」と表示し停留所ページへリンク
- 変化のない停留所は列挙しない (網羅は網羅性ビューが担う)

### V5: 網羅性ビュー (検証モードの再構築)

- 出発点: **現検証モードから路線単位の情報を消す** (路線ページと重複する
  A/B/C 群の個別イベント羅列をやめる)
- 全イベントを「**表示先**」で分類したナビゲーションに再編:
  「路線ページ (◯◯線 Lev.2 に反映) / 停留所ページ / フィード章 /
  **どこにも出ない (検証のみ)**」。最後のカテゴリの存在が網羅性の証明になる
- そのために presentation モデルに **presentation_refs (表示単位 → 消費した
  event_id 群)** を記録する (モデル拡張。コア不変・注釈主義の範囲)
- RawDiff → ChangeEvent → 表示先 の3層トレーサビリティ: explained_ratio と
  残差 (UNEXPLAINED_RESIDUAL / TECHNICAL_ID_CHURN) をビューの先頭に
- 表示だけでなく「レポート被覆率」(表示先が『どこにも出ない』でないイベントの
  割合) を新たな計測値として出す

### V6: フィード章 (その他の変化、人向け)

- E/F 群を種類別の節で人が読める形に: 運賃 (料金表の before/after)、
  フィード有効期間、事業者情報、曜日ダイヤ区分再編、行先表示、翻訳、
  バリアフリー情報
- V1 で後回しにした「fare 型・metadata 型が主のフィードでの章構成」を
  ここで解消する

### X1: 機械向けインタフェースの文書化 (小・並行可)

- docs/spec/bundle_schema.md: bundle の各キー (events / rawdiffs / presentation /
  geometry / timetables / catalog / meta) のスキーマと安定性の約束を明文化
- AI・外部システムへの推奨: 低レベルは events + rawdiffs (+ evidence)、
  表示済み解釈は presentation、の使い分け指針
- V5 の presentation_refs が入れば、3層の対応付けも機械可読になる

## レポートの4部構成 (2026-07-07 ユーザー提案により確定・実装)

上記タスクの提示先として、レポートモードを4部構成に再編した:

1. **フィード全体の変化** — txt ファイルごとの新旧対応表 (有無・行数・RawDiff
   内訳)、サービス期間、曜日区分ごとの便数 旧→新、フィード級イベント
   (期間・事業者・カレンダー・運賃等 = V6 の前倒し骨格)
2. **停留所の変化** — V4 の章。変化がある場合は地図を最初から開いた状態で表示
3. **路線毎の変化** — 既存の路線章 (3.1, 3.2, … と部内番号。変更のない路線も含む)
4. **その他の変化** — 第1〜3部のどこにも現れないイベントの種類別件数
   (SHAPE_CHANGED の座標詳細、ID 張り替え、未説明残差等)。**将来のイベント
   タイプ追加も自動的にここに落ちる** カバレッジの受け皿。V5 の
   presentation_refs 導入後は「表示先のないイベント」の機械的列挙に昇格する

第4部の存在により「レポートに無い=変化が無い」という誤読を構造的に防ぐ。

## 順序の提案と論点 (ユーザー判断待ち)

推奨順: **V3 クローズ → V4 (停留所) → V5 (網羅性ビュー) → V6 (フィード章)、
X1 は随時**。理由: 停留所は人に伝える価値が最も高く identity 資産 (stop cluster
link) で安く作れる。V5 は presentation_refs というモデル拡張を含むため
1マイルストーンとして独立させる。V6 は表現の設計が必要だが依存が少ない。

論点:
1. V4 と V5 の順序 (網羅性を先に固める考え方もある)
2. 停留所ページの粒度 — 変化のあった停留所のみ (推奨) か、全停留所一覧+変化マークか
3. 「検証モード」の呼称 — 網羅性ビューに改称するか、検証モードの中身を差し替えるか
