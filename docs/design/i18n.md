# 国際化 (I トラック): 英語対応と国際 GTFS 検証

状態: 計画 (2026-07-12 起草)。方針: **対応言語は日本語+英語の2言語で十分**。
目標は「英語環境から diff.gtfs.jp に来た人が、迷わず比較を実行して英語のレポートを
読める」こと、および「日本以外の典型的な GTFS で正しく動くと実データで言える」こと。

## 1. マイルストーン

### I1: 国際検証データセットの構築

「GTFS を探し出して確認のためのデータセットを作る」の本体。**先頭に置く**
(以降の全マイルストーンの検証材料になるため)。

- **取得元**: Mobility Database (mobilitydatabase.org — カタログ+**過去版 API** で
  世代ペアが作れる)。補助: transitfeeds.com アーカイブ (旧 OpenMobilityData)、
  transit.land。世代管理があることが gtfs-data.jp との対応物になる。
- **選定基準 (特徴マトリクス)**: 単なる「有名どころ」でなく、日本のフィードに
  ない特徴を1つ以上ずつカバーする 6〜8 フィード:
  1. 模範実装 (GTFS 発祥系): TriMet (Portland) など
  2. calendar_dates 主体の米国流 (calendar.txt をほぼ使わない): 米大都市圏
  3. frequencies.txt (ヘッドウェイ運行): 中南米系 (メキシコシティ等)
  4. 非 ASCII・アクセント文字 (仏語等): STM Montréal / IDFM (パリ)
  5. 多モード (鉄道 route_type 0/1/2 + pathways/levels): MBTA / HSL Helsinki
  6. 国家規模ストレステスト (bulk・Lambda 制約の確認): スイス全国 / DELFI (独)
  7. Fares v2 (fare_products 等 = 本ツールには未知ファイル): 採用事業者
- **成果物**:
  - `scripts/fetch_intl_feeds.py` — 選定フィードの世代ペアを再現可能に取得
    (URL・版 ID を台帳化。zip は再配布しない — ライセンス確認も台帳に記録)
  - `data/intl/` (gitignore) + `docs/verification/intl_feeds.md` (選定理由・
    特徴マトリクス・ライセンス・取得手順の台帳)
  - 全ペアで CLI compare を実行し、完走/クラッシュ・explained_ratio・所要時間・
    残差内訳を同ドキュメントに記録 (**修正はしない**。問題は I3/I4 の入力)
- **DoD**: 6ペア以上の実行記録が残り、問題点が課題番号付きで列挙されている。

### I2: Web/UI の英語対応 (G1/G2/G3/G7)

- 入力 UI: ja/en 辞書 + 言語トグル (ビューアと同じ流儀)。**初期言語は
  navigator.language** (ja 系以外は en)。英語時は「zip アップロード」タブを
  前面に (gtfs-data.jp タブは "Feeds from gtfs-data.jp (Japan)" として残す)。
- API エラー: 文字列でなく**エラーコード** (`same_generation` 等) を返し、
  クライアント側辞書で表示 (安定インタフェースの流儀に合わせる。既存文字列は
  互換のため残してよい)。
- 利用規約: en 版 (terms.en.html または同一ページ内切替)。内容は現行 §1〜7 の翻訳。
- OGP/メタ: og:locale、英語 description の併記方針を決める (トップは ja 主 + en 説明)。
- admin ページは対象外 (運用者向け)。
- **DoD**: 英語設定のブラウザで「トップ→アップロード比較→レポート閲覧→
  ログイン→マイページ→削除」が全て英語で完結する (実操作確認)。

### I3: レポートの英語品質と JSON 値の言語中立化 (G4/G7)

- JSON 値の日本語根絶: leg ラベルの「循環」→ kind="loop" からビューア側で
  組み立て (i18n)。zip 表示名は構造化 (agency/from_date/uploaded を分離保存し
  表示時に組む) か en 対応。OGP 題名はフィードの言語で切替 (feed_lang があれば
  参照、なければ ja 既定) — 検討の上決定。
- ビューア初期言語をブラウザ言語で初期化 (トグルは維持)。
- I1 データセットの実レポートを英語 UI で通読し、不自然な表現・未訳・
  日本前提の文言 (「ダイヤ改正」の英訳語彙統一等) を洗い出して修正。
- **DoD**: 国際フィード2件以上のレポートが英語 UI で違和感なく読める
  (通読チェックリストを検証ログに残す)。

### I4: 国際フィードの技術対応 (G5/G6 + I1 で見つかる課題)

- **地図タイルの国際化 (最重要)**: 停留所の bbox が日本域なら地理院、
  それ以外は国際タイル (OpenFreeMap 等、キー不要・出典表記) に自動切替。
  3コンポーネントのタイル選択を共通化してから実装。
- 停留所名正規化: 英語圏の実名での過剰剥がし検証 (合成テスト + I1 実データ)。
  問題があれば指標語の適用条件を絞る (閾値・条件は config)。
- day_type (平日/土曜/休日) と「祝日」概念、運賃表示 (通貨)、タイムゾーンの
  表示点検。fares v2 等の未知ファイルは L0 が拾う (網羅性は既に保証) —
  残差カタログとして精査し、意味のあるものは claiming ルール追加を検討
  (yamako の pass_rules と同じバックログに合流)。
- **DoD**: I1 データセット全ペアで explained_ratio・地図表示・レポート成立を
  記録した検証ログ (docs/verification/)。既存日本フィードの回帰不変。

### I5: 公開整備 (G8)

- README.en.md (または README を英日併記に再構成)。トップページの英語文言確定。
- (任意) OGP 画像の英語版・言語別出し分け。
- **DoD**: 英語話者がリポジトリ/サイトだけでツールの目的・使い方・制約に
  到達できる。

## 2. 決めごと・前提

- 翻訳は「ja/en 辞書の対管理 + ずれゼロをテストで担保」の現行方式を全面に広げる
  (ビューアで実績あり)。翻訳ファイルの外部化 (gettext 等) は規模に対して過剰。
- gtfs-data.jp 連携は日本専用のまま (国際の入口は zip アップロード。
  Mobility Database の世代 API 連携は将来構想として w3_2_directions.md の
  観測所構想と同列に置く — 今回のスコープ外)。
- レポート本文の「事実」(名称・数値) はフィード由来のまま (翻訳しない)。
  翻訳するのは UI ラベル・説明文のみ。
- 国際フィードの検証は「ルールベースの機械処理で出来ることをやる」原則の
  まま行う — 各国固有事情への過剰適応はしない (挙動が予測可能に劣化するなら可)。

## 3. リスク・未知

- 国家規模フィード (スイス/DELFI) は Lambda 15分・3008MB に収まらない可能性 —
  bulk (2026-07-12) で耐性は上がったが、I1 で実測して「対象外の規模」を明文化
  する選択肢も持つ。
- Mobility Database の過去版 API の粒度 (どこまで世代が遡れるか) は I1 冒頭で確認。
- 「休日」day_type は日本の祝日概念を含む — 海外フィードでは
  weekday/sat/sun のみで自然に動くはずだが、I1 で要確認。
