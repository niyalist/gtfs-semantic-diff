# 便対応付けの再設計 (trip matching v2)

状態: **提案 (2026-07-08)** — ユーザー承認後に実装着手。
背景: docs/verification/trip_identity_survey.md (八戸の反例と根本原因の特定)。

## 1. 問題

改正前後の便の対応付けは本ツールの根本機能である。対応付けが正しいからこそ
「この便がこう変わった」(時刻修正・経路変更・短縮) と「便が消えた/増えた」を
区別でき、C群 (増減便・時刻)・B群 (パターン変化)・ID churn・④差分時刻表の
すべてが成立する。将来の応用 (変更の統計抽出) もこの対応付けの質に依存する。

現行は**4段の縦積み**で、それぞれ別の原理に立っている:

| 段 | 原理 | 場所 | 問題 |
|---|---|---|---|
| ① exact | 内容署名の完全一致 | tripdelta | 健全 (完全一致は無条件に信頼できる) |
| ② modified | **同一 trip_id を無条件に信頼** | tripdelta | **連番型 ID 運用で破綻** (八戸: ID は「系統内の発順」でしかなく、便の増減で別の便へスライド) |
| — | 残り = removed / added | tripdelta | ②の誤対応の巻き添えで過大になる |
| ③ 表示ペアリング | 発時刻差 ≤ 60分の貪欲 (表示層のみ) | presentation | ②が誤って consume した後の残骸しか見えない。停車列を見ない |

根本原因は「**ID の一致を同一性の証拠の一つではなく定義として扱っている**」こと。
証拠の強さの正しい序列は 内容完全一致 > 時刻・経路の整合 > ID 一致 である。

## 2. 設計方針

1. **1回の大域割当に統一する**。①〜③の縦積みを、コスト最小の2部マッチング
   1段に置き換える (完全一致はコスト0の特別な場合として自然に含まれる)。
   階層は「マッチング1段 + 種別ラベル付け」に浅くなる。
2. **解く場所はコア (events/tripdelta.py)**。対応付けは表示の都合ではなく
   意味的差分の土台なので、presentation の後付けペアリング (③) は廃止して
   コアに吸収する。データモデルは1つ (CLAUDE.md 原則2) — 移行は一括で完遂する。
3. **決定的・config 閾値** (原則4)。コスト関数の重み・受理閾値はすべて
   config/default.toml。同じ入力から常に同じ対応付け。
4. **会計・JSON スキーマは不変**。ChangeEvent の構造・説明会計の仕組みは
   変えない。変わるのはイベントの「中身の質」(増減便の数え方が正確になる等)。

## 3. 定式化

### 3.1 ブロッキング (候補対の範囲)

全 old×new は無駄かつ危険なので、候補は次の範囲に限る:

- **route_group (枝番系統を束ねる路線ブランド) × 同一 day_type**
  (2026-07-09 改訂: 当初は family 単位だったが、徳島 加茂谷線で「便を枝番違いの
  route へ移す」運用 (202→204、trip_id まで同一・LCS 0.93・時刻差5分) が
  family 境界で弾かれる反例が出た。実測で徳島 removed の 23% (52便)・永井 4便が
  この型。route_group は「利用者が認識する路線」として M6/M7 で確立した既存層で、
  新しい概念は増えない)
- family の名称変更は family_links (identity の対応) を経由して新側の
  route_group に写す
- day_type 跨ぎ (平日→毎日 等) は v1 のスコープ外とし、DAYTYPE_RESTRUCTURED
  (E群) の守備範囲に残す (制限として明記)

### 3.2 コスト関数 (対 (o, n) ごと、決定的)

```
cost(o, n) = w_time  · min(Δt_shared(o,n), cap) / cap        # 時刻の不整合
           + w_route · (1 − lcs_ratio(o,n))                  # 経路の不整合
           − w_id    · [o.trip_id == n.trip_id]              # ID 一致は弱い事前
受理条件: cost(o, n) ≤ τ (超える対は組まない → removed/added)
```

- **Δt_shared = 共有停留所における発時刻差の中央値** (共有が無ければ
  始発時刻差)。これが本設計の要:
  区間短縮 (八戸 旧11:30 32停 → 新11:30 28停) では共有28停の時刻がほぼ
  据え置きなので Δt_shared ≈ 0 となり、強く結ばれる。始発時刻だけを見る
  現③より頑健で、途中始発化・延長にも対応する
- **lcs_ratio = LCS(base_seq) / max(len)**。順序を考慮するので逆方向便を
  誤って結ばない (集合 Jaccard の弱点を回避)
- **ID 一致は減点 (ボーナス) として効く**が、Δt_shared が大きければ勝てない。
  八戸の「同一 ID・210分差」は不成立になり、「ID 違い・0分差・経路包含」が
  成立する。畑線 (ID 全張替え・内容同一) は ID 項なしで cost≈0 → 従来どおり救済
- 重み・cap・τ は config `[matching]` (新設) に。初期値は評価 (§5) で決める

### 3.3 割当アルゴリズム

- **第1実装: コスト昇順の決定的貪欲** (現③と同型、tie-break を
  (cost, Δt, o.trip_id, n.trip_id) で明示)。ブロック内の便数は高々数百、
  O(n² log n) で十分速い
- 貪欲は局所最適の連鎖ミスが理論上ありうるが、Δt_shared ベースのコストでは
  正しい対がほぼ常に厳密に小さいコストを持つ (評価で確認)。評価で連鎖誤りが
  残る場合のみ最適割当 (Hungarian、依存追加なしの自前実装 ~100行) に差し替える。
  ソルバは関数1つに隔離してどちらでも入れ替え可能にする

### 3.4 出力 (TripDelta の新しい形)

```
matched:  list[(old, new, kind)]   # kind: unchanged / id_changed / retimed /
                                   #       rerouted / retimed+rerouted (内容から導出)
removed:  list[old]
added:    list[new]
```

- 現行の exact_pairs / modified / churn_pairs は matched から導出できる
  (exact = kind∈{unchanged,id_changed}、churn = id_changed)。下流の互換
  プロパティとして残すが、**意味は「割当の結果」に統一**される
- 将来の統計応用のため、対ごとの変化ベクトル (Δ発時刻・停車列差・kind) を
  matched に持たせる

## 4. 影響範囲と移行 (一括)

| 消費者 | 現在の入力 | 変更 |
|---|---|---|
| C群 frequency (増減便・始終発) | removed/added/modified | matched/removed/added に置換。**誤対応由来の水増しが消え、増減便カウントが正確になる** |
| B群 patterns (区間延長短縮・経由変更) | modified (同一ID) | matched (kind=rerouted系) に置換。**ID が張り替わる/連番ズレのフィードでもパターン変化を検出できるようになる** (現在は検出漏れ) |
| TECHNICAL_ID_CHURN | churn_pairs | 導出値なので実質不変 |
| presentation ④/Lev.3/Lev.5 | modified + ③表示ペアリング | **③ (_pair_leftovers) を削除**し matched を直接使う。表示層の特例が消え実装が簡素化 |
| 会計 | evidence 台帳 | 仕組み不変。explained_ratio 維持を検証項目に |

detection.md (§ trip 照合・C群・B群) と presentation.md (表示ペアリングの記述)
を同期更新。ontology (イベントタイプ) は不変。

## 5. 評価計画 (実装前後)

1. **クリーン正解での P/R** — survey_trip_identity の枠組みを流用するが、
   擬似正解の「ID 安定」判定に**時刻整合を追加** (同一ID対の Δt_shared p90 ≤
   数分のフィードのみ正解に使う)。前回の教訓: 停車列類似だけのフィルタは
   八戸型連番 ID を素通しした
2. **反例セット** — 八戸の当該ケースを合成 GTFS の単体テスト化
   (連番スライド + 経路変更 + 区間短縮振替)。畑線型 (ID 全張替え)・
   同時刻交差 (佐那河内型) もテストに含める
3. **8検証フィードの before/after 差分** — 対応付けが変わった組を全列挙し
   目視サンプル確認。八戸の当該表が常識どおり (11:30→11:30 短縮、
   15:00→15:00…) になることを確認
4. **回帰** — explained_ratio 不変・既存157テスト・性能 (docs/perf に計測)
5. コスト重みの感度分析 (V2.1 と同様、正解対と誤対で分布が二極化するかを確認
   してから初期値を決める)

## 6. リスク

- **重みという新しい自由度**: 感度分析で二極化を確認し、際どい閾値に依存しない
  ことを示す。閾値はすべて config
- **性能**: ブロック内 O(n²)。大規模 (八戸 1,600便) でもブロック分割後は
  数百×数百が上限。計測して docs/perf に記録
- **下流の意味変化**: C群・B群イベントの件数が変わる (正しくなる方向だが、
  検証ログで before/after を明示する)
- day_type 跨ぎの振替は v1 で扱わない (明記)。実データで問題になれば
  ブロッキングの緩和を別途検討
