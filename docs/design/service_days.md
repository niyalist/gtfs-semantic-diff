状態: **承認 (2026-07-23 ユーザー承認。実装トラック: roadmap SD1〜SD4)**。
v0.5 は**整合性監査版**: v0.2〜v0.4 で積み上げた提案を、ユーザーの危惧

### 1.5 定点観測: 日次機械更新と「同居期間」(三重交通桑名営業所、2026-07-23)

- 毎日1世代が機械発行され `to_date` が1日ずつ伸びる (約100日のローリング窓)。
  service_id は `1_1_20260601` 形式の改正日サフィックス付き。
- 7/11 改正に対し **7/4 作成分から新旧同居** (`*_20260601` 6/1〜7/10 +
  `*_20260711` 7/11〜10/12 の6 service)。7/11 作成分で旧側が消え3 service に戻る。
  → **改正のたびに約1週間、同居版が「現行」として流通** — Web 利用者が
  同居スナップショットを比較対象に選ぶ確率は高い。
- 祝日は calendar_dates で「平日 service 削除+日曜 service 追加」(7/20・8/11)
  — 祝日カレンダーなしで「日曜ダイヤへの置き換え」がデータから読める。

## 2. 中核概念 (監査後)

### 2.1 実効運行日集合 (層0) — SD1 の本体 【実装済み 2026-07-23】

各 service の実効運行日集合 = (calendar 期間 × 曜日フラグ) − calendar_dates
削除 + 追加、をフィード有効期間でクリップ。day_type 分類はこの集合に基づく:

- 実効日**ゼロ** → inactive (T6: 期限切れ残骸)
- **密度** (実効日 ÷ 期間×フラグ該当日) が `min_flag_day_ratio` (config、0.5)
  **未満** → irregular (T1: 祝日専用 service。PRT 実測 0.06)
- それ以外 → 従来のフラグ分類 (不変)

実装時の学び (**日数閾値ではなく密度**): 起草時は「実効日数 ≤
short_service_max_days → irregular」としたが、STM (四半期 period 分割) の
正規土曜ダイヤ (11土曜−祝1 = 10日) が 81/114 service 誤爆した。判別軸は
「何日走るか」でなく「**フラグ日のうち実際に走る割合**」— 祝日専用 0.06 と
期間分割正規ダイヤ 0.9〜1.0 は密度で明確に分かれる。副次効果として同居
(桑名) の残存窓 service も型を維持し、**T2 は SD2 に純粋に分離**された。
検証: docs/verification/SD1_day_types.md (日本フィード分類不変・イベント
JSON A/B 一致、prt 土曜 38→38・explained 1.0000、mbta 期限切れ 95/269件
→ inactive)。

- **ラベル語彙は不変** (weekday/saturday/sunday_holiday/weekend/daily/dow_*/
  irregular/inactive)。「平日」「土休日」のざっくりラベルが第一級 — これは
  制約条件であり、本検討のどの段階でも変えない。

### 2.2 窓内区間対比較 — T2 (同居) の解 (v0.6 改訂) 【実装済み 2026-07-23】

実装の確定形は detection.md §4.5b、実測は docs/verification/SD2_window_scope.md。
実装で加わった規則 (起草時になかった学び): 除外は週次レギュラー型のみ
(特定日世界は従来比較 — 名古屋の年次ペア)、後継判定は family の世代間対応成分
(route_group ブロック) 粒度のカバー率 ≥ `carryover_coverage_min` (0.8)
(桑名の route_id 世代サフィックス・伊勢市の「環状線→右回り/左回り」分割・
佐賀の「後継のない三瀬神埼線は除外しない」をすべて実データで検証)。
以下は起草時の設計 (有効)。

**「代表世代を1つ選んで比較する」のではなく、「日付区間ごとに正しい相手と
比較する」**。手順はすべて日付演算で決定的:

1. 両世代の有効期間の交差 = **共通窓**を求める。共通窓の外にしか実効日を
   持たない service とその便は比較対象外 (旧 v0.5 のクリップに相当)。
2. 共通窓を、**窓内にある calendar 期間の端点** (start_date/end_date。
   有限個・決定的) で区切り、日付区間の列にする。calendar_dates による
   断続 (祝日・季節の削除) は**境界にしない** — 断片化を防ぎ、T3 は
   据え置き (§8)。
3. 各区間で、新旧それぞれ「その区間に実効日を持つ service の便世界」を
   構成し、区間ごとに従来どおりの比較 (day_type ラベル・便対応 v1 規則の
   まま) を行う。
4. **内容が同値の区間は「変化なし」に畳む**。差分のある区間が「YYYY-MM-DD
   からの改正」としてレポートの本体になる。

同居が絡む比較の振る舞い (桑名の実例で):

| ケース | 共通窓と区間 | 結果 |
|---|---|---|
| 同居 (旧側) vs 改正後のみ (§4.1 実測1) | 共通窓 7/11〜 → 旧世代 A は窓外 → 区間1つ (B vs B') | 正しく「改正後同士」の比較。偽減便279件が消える。相手の窓が片世代を自動選択する — 明示的な代表選定は不要 |
| 改正前のみ vs 同居 (新側) (§4.1 実測2 — **同居週の Web 既定比較**) | 共通窓 6/1〜10/3、区間は 7/11 で2分割: [6/1〜7/10] A' vs A、[7/11〜10/3] A' vs B | 区間1 = 内容同値 → 「変化なし」に畳む。区間2 = **7/11 改正の中身そのもの**。偽増便279件が「7/11 からの改正」1本の正しい説明に変わる |
| 同居 vs 同居 (ローリングの隣接週) | 区間2つ、どちらも内容同値 | 「有効期間が N 日進んだ。内容同一」に圧縮 — 理想の挙動 |
| 単一世代 vs 単一世代 (日本の大半・国際検証フィード) | 区間1つ | **現行挙動に退化** (回帰保証の主眼) |

- 除外・畳み込みの対象行 (calendar / calendar_dates / trips / stop_times) は
  「窓外世代の除外」「区間内容同一」を表す情報イベントが evidence として
  claim — 台帳の網羅性は保ったまま説明の持ち主を正しい抽象度に移す。
- M5 の HOLIDAY_EXCEPTION_CHANGED 窓正規化と同じ思想の一般化であり、
  detection.md §7 の**多世代タイムラインの最小形** (窓内2区間) でもある —
  将来そこへ育てる土台になる。
- 区間が複数とも差分を持つ場合 (稀: 異なる改正を挟んだ同居同士等) は
  区間ごとに節を分けて報告 — 「どの日からの変化か」が常に明示される。

一般性の確認 (**両側が複数世代でも機能する** — 機構は per-interval で
世代数を数えていない。v0.6.1 追記):

| ケース | 区間の構造 | 結果 |
|---|---|---|
| 同じ改正を挟む同居同士 (ローリング隣接週) | 境界 7/11 が両側で一致 → 2区間とも内容同値 | 「窓が N 日進んだ・内容同一」に圧縮 |
| 別々の改正の同居同士 (7月の A+B vs 10月末の B'+C) | 共通窓 7/11〜10/12 に内部境界なし → 1区間 (B vs B')。次改正 C は共通窓外 | 窓内は「ほぼ変化なし」+ **「新側にのみ存在する将来ダイヤ (10/24〜)」を窓外イベントで明示**。旧側に 10/24 以降の情報はないので比較しないのが正直な答え。C の中身は同一ファイル内の区間比較 = 多世代タイムライン (将来) の守備範囲 |
| 改正日の変更 (7/11 予定 → 7/18 に延期) | 境界 7/11 と 7/18 で3分割 | [〜7/10] 同値 / **[7/11〜7/17] 「この期間は旧ダイヤ継続に変わった」** / [7/18〜] 同値 — 延期という現実をそのまま言い当てる。日付単位の意味論でないと原理的に出せない説明 |
| 3世代以上の同梱 | 境界が増えるだけ。区間数 ≤ 窓内 calendar 端点数 + 1 | 同じ機構で有界に動く |

注意点2つ: (1) 極端に短い区間 (数日) は day_type によっては実効日が
0〜1日になる (土曜を含まない区間等) — 便世界が空の day_type は比較を
スキップし、区間の日数と含まれる曜日をレポートに明示する。(2) 1ファイル内で
世代期間が重複しているデータ (端点がずれた重ね書き) でも、各区間の便世界は
「その区間に実効日を持つ service の合併」なので定義は破綻しない —
重複区間では両世代の便が並ぶが、それはデータがそう主張している姿である。

  docs/verification/SD4_calendar_view.md】: 第1部に新旧の年間グリッド。

## 4. 根拠となる実測

### 4.1 現行システムは同居スナップショットで「誤った説明」を量産する (2026-07-23、両方向を実測)

桑名の実データによる現行実装の実測2件:

- **実測1** prev_19 (同居) vs current (新のみ): イベント308件、うち
  **SERVICE_REDUCED 279件** — 全路線・全時間帯で「2→1便」「10→5便」の
  **全便半減という誤説明**。explained_ratio 0.9997。
- **実測2** prev_20 (改正前のみ) vs prev_19 (同居): イベント299件、うち
  **SERVICE_INCREASED 279件** — 鏡像の**全便倍増という誤説明**。
  explained_ratio 0.9995。**同居週には current が同居版なので、これが
  Web の既定比較で起きる形**。
- 教訓: **説明台帳は「網羅」を構造的に保証するが、「説明の意味的正しさ」
  までは保証しない** — T2 は台帳をすり抜けて嘘を言う、対応必須の観測済み
  実害である。また実測2は v0.5 の窓クリップでは解けない (共通窓 6/1〜10/3 に
  両世代とも実効日を持つ) — §2.2 の区間対比較が必要な直接の根拠。
- (参考: TECHNICAL_ID_CHURN 18/14件 — 世代サフィックスの ID 振り直しは
  既存機構が正しく吸収していた)

### 4.2 M5 の窓正規化は calendar_dates 層で既に存在する

`events/rules/calendars.py` の HOLIDAY_EXCEPTION_CHANGED は新旧有効期間の
重なり窓で `within_overlap`/`outside_overlap` を区別し `substantive` を
判定している (窓が求まれば confidence 1.0)。§2.2 はこの思想の一般化。

## 5. 撤回した提案 (v0.4 §4.6 便同一性の変更) — 記録として残す

v0.4 では「day_type を除いた内容署名の完全一致で day_type 跨ぎの便を対応
させ、RUN_DAYS_CHANGED (運行日変更) として報告する (段1.5)」を提案したが、
**撤回して v1 規則 (day_type が変わった便は追わず廃止+新設) を維持**する。

理由:

1. **全体感の哲学との整合** (ユーザーの再確認): 本システムのレポートは
   day_type 世界ごとの比較である。「毎日→平日」はその枠内では「土日の世界から
   便が消えた (廃止)」であり、正確な記述。納得感が既にある表現を、理論的
   純粋さのために置き換える必要はない。
2. **手続き違反**: 段1.5 が真に救うケース (「毎日」service を「平日+土休日」に
   書き替えるエンコーディング分割) は**実データで未観測**。イベント追加は
   「残差の精査 → ontology 採録 → 実装」の順という開発ルールに対し、
   観測なしの先回り設計だった。
3. **決定性の穴**: 「時刻完全一致」条件でも、同時刻・同経路の便が複数 day_type
   に実在する場合 (T4 の亜種) に 1:N の曖昧割当が生じ、タイブレーク規則を
   増やすことになる — 複雑さの対価に見合う観測済みの利益がない。

再検討のトリガ (将来): 実フィードの残差または churn 精査で「エンコーディング
分割による大量の廃止+新設」が実際に観測されたとき。その時も、まず
TECHNICAL_ID_CHURN 系の既存受け皿で説明できないかを先に検討する。

## 6. 設計原則との照合

| 原則 (CLAUDE.md ほか) | 本提案 (v0.5 監査後) の適合 |
|---|---|
| 説明台帳 (網羅性) | L0 不変。窓外除外分は claim イベントの evidence に (残差にしない)。§4.1 の教訓「網羅≠意味的正しさ」を台帳の限界として認識 |
| データモデルは1つ | 新モデル体系なし。実効日集合は load 層の派生値、クリップは比較時の前処理 |
| JSON 安定インタフェース | スキーマ変更は加算のみ (日付集合・窓情報・新イベント型)。既存キー・値の意味は不変 (T1/T2 該当時の day_totals は「誤値→正値」の修正) |
| 決定的ルールベース | 日付演算のみ。閾値は既存 config (`short_service_max_days`) の再利用が主 |
| 早すぎる最適化の禁止 | 計算は日数×service 数の線形で無視できる規模 |
| ラベル堅持 (ユーザー要件) | 語彙不変を制約条件として明文化 (§2.1)。パターン類は表示素材に格下げ (§3) |
| 過剰適応しない | 国・事業者固有の分岐なし (祝日表も不使用)。窓クリップ・実効日は普遍的な日付演算 |
| イベント追加の手続き | 新イベント型 (窓外除外) は桑名ペアの残差精査を経て ontology 採録 → 実装の順で確定 |
| 便同一性 (M8 v1 規則) | 不変 (§5 で変更案を撤回) |
