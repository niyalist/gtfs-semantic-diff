# 運行日モデルの精緻化 (検討メモ)

状態: **検討 (2026-07-23 起草。実装未着手 — 本メモはコードを書く前の設計検討)**。
発端: PRT (ピッツバーグ) の土曜便が見かけ上 38→76 便に倍増した問題
(docs/verification/intl_feeds.md IN-8) と、それを受けたユーザー指摘
「同種の問題は日本のデータにもある。運行日情報をもっと詳しく提示する UI が
あっていい。diff 以前に、1つのデータを正確に表現する必要がある」。

関連: docs/design/day_types.md (M10 現行仕様)、detection.md §7 (多世代タイムライン)。

## 1. 何が起きたか — service_id 解釈の実例類型

本ツールはこれまで「運行日」を意図的に単純化してきた (day_type 6分類 +
M10 の dow_XXXXXXX 一級化。分類の入力は calendar.txt の曜日フラグが優先)。
PRT の問題を受けて gtfs-data.jp 三重県全39フィード×直近4世代 (156 スナップ
ショット弱) を走査したところ、**「同一曜日パターンの service が複数ある」
スナップショットが24件**見つかった。実例で類型化する:

| 型 | 内容 | 実例 (確認済み) | 現行実装での症状 |
|---|---|---|---|
| T1 祝日専用 | 曜日フラグ+calendar_dates 大量削除で「祝日1日だけ」の service を表現 (米国流) | PRT: service 1 = 土曜フラグだが実効 7/4 のみ。service 4 = 月曜フラグだがレイバーデーのみ | 土曜 38→76 の見かけ倍増、偽の「月曜」列 (IN-8) |
| T2 世代同梱 | 改正前後のダイヤを期間で分けて1ファイルに同梱 | 伊勢市おかげバス prev_2: `7_1_20240717` (〜2025-03-31) と `7_1_20250401` (2025-04-01〜) が各50便。菰野町かもしか号 prev_2 も同型。津エアポートライン「ドッグ期間」も期間分割 | **全曜日タイプが倍増** (50→100便に見える)。ユーザーが記憶していた事例そのもの |
| T3 季節分割 | 同一曜日フラグ+期間 (または大量削除) で夏/冬ダイヤを分ける | 鳥羽市定期船「夏季/冬季」(全日フラグ+削除244/427日で季節を彫り出す)、鳥羽市かもめバス「全日・夏/全日・冬」、亀山市「春・夏/秋・冬」 | 季節限定便が年中走って見える。夏冬で時刻が違えば同時刻帯に2本並ぶ |
| T4 系統・地区別 | 同一 day_type・同一期間だが系統/地区ごとに service を分ける (データ作成都合) | 伊賀市「平日/青山平日」、亀山市「平日/加太平日」、熊野市「全日/紀和・本宮線」、四日市 Navya 実証 (車両別) | **症状なし** — 便が排他なので合算表示が正しい。「同じ平日はまとめる」が人の認識に合う |
| T5 特定日運行 | calendar_dates のみ (または実質的に少数日) の service | 伊勢鉄道「運転日注意(みえ)」(快速みえの運転日)、既存 irregular 全般 | M10 で分類は対応済みだが、**表示が「特定日」の一言で具体日が分からない** (ユーザー指摘) |
| T6 残骸 | 期限切れ期間の service や calendar_dates が残存 | PRT の W/U/S (期間が丸ごとフィード期間外)、三重交通中勢 current の 2023年 calendar_dates 例外149行 | 実効日ゼロなら inactive (M10) で概ね無害だが、期間外 service に便が載ると誤算入しうる |

要点: **T1/T2/T3 は「実効運行日が互いに素な service を同じ曜日タイプとして
合算する」ことによる同一機序の誤り**。T4 は合算が正解。つまり service_id の
字面や曜日フラグではなく、**実効運行日集合**が分類・集計の正しい基礎になる。

## 2. 統一原理: 実効運行日集合 (effective date set)

提案する中核概念はひとつだけ:

> 各 service の**実効運行日集合** = (calendar の期間 × 曜日フラグ)
> − calendar_dates 削除 + calendar_dates 追加、をフィード有効期間で切り出した
> **具体的な日付の集合**。以後の分類・集計・表示はすべてここから導出する。

これは M8 (便対応は内容主導、ID は弱い事前)・M9 (family 対応は内容主導) と
同じ設計思想の運行日版である: **service_id はデータ作成者のエンコーディング
都合であり、意味の単位は「ある日に何が走るか」**。決定的に計算でき、
閾値も既存 config (`short_service_max_days` 等) の延長で足りる。

導出される判定 (すべて決定的):

1. **day_type 分類の実効日ベース化**: 実効日数が閾値以下なら曜日フラグに
   関わらず特定日 (irregular)。T1 はこれだけで解消 (PRT service 1 は実効1日
   → 特定日。土曜は 38→38 になり、既存の「特定日の置き換え/追加」表示に乗る)。
   日本フィードの祝日 service はフラグを持たないため回帰影響なし (要実測確認)。
2. **期間 (regime) 分割の検出**: 日付軸を「アクティブな service 集合が一定な
   極大区間」に分割する (実効日集合から機械的に求まる)。同一 day_type の
   service 群が期間的に互いに素なら、そのフィードには複数の「期間」がある —
   T2 (改正前/後)・T3 (夏/冬) はこれで「期間A のダイヤ / 期間B のダイヤ」に
   正しく分かれ、倍増が消える。
3. **T4 の自然な合算**: 系統別 service は実効日集合が (ほぼ) 同一なので
   regime 分割では分かれない → 従来どおり同じ「平日」に合算される。
   「敢えて service を分けたデータも人の認識 (同じ平日) に合わせてまとめる」
   というユーザー方針がそのまま実現される。判定は日集合の一致/包含で決定的。
4. **T6 の無害化**: フィード期間と交差しない service は実効日ゼロ → inactive。
   便が載っていても便数に算入しない。
5. **特定日の具体化 (T5)**: 特定日 service は実効日集合そのものを表示に使う —
   「特定日 (7/4)」「運転日: 8/11〜8/16, 12/30〜1/3 (計11日)」のように、
   少数なら列挙・多数なら範囲要約。「特定日」という抽象語だけの表示をやめる。

## 3. 運行日カレンダービュー (単一スナップショットの検査 UI)

diff 以前に「1つのデータの運行日構造を正確に見せる」ビュー。レポート消費者
(自治体・事業者担当) のデータ検品にも、本ツール自身のデバッグにも効く。

素案 (色弱原則: 記号・線種が第1チャネル、色は補強):

- **年間カレンダーグリッド** (月×日)。各日のセルに day_type 記号
  (平/土/休/特 の1文字) と regime 番号 (①②…) を刷る。
  祝日運行の置き換えは▲、運休は×を重ねる。
- **帯 (タイムライン) 表示**: 上から (a) feed_info の有効期間、
  (b) calendar 各 service の期間 (regime 分割後は regime 単位に集約)、
  (c) calendar_dates 例外の点 (▲追加・×削除)、(d) gtfs-data.jp 世代の
  from/to (リポジトリ由来の場合)。「feed_info は 12/31 までだが calendar は
  10/23 で切れる」等のズレが一目で分かる。
- 置き場所の候補: まず**レポート第1部に新旧2枚並べて**掲載 (差分の文脈で
  最も効く。第1部は既にフィード全体の器がある)。単体検査モード
  (compare を伴わない inspect) は将来の入口として温存。

## 4. diff への波及 (regime が複数あるフィードの比較)

現行の比較は「1スナップショット = 1つの時刻表」を暗黙に仮定している。
regime 検出を入れると、T2/T3 フィードでは「どの期間同士を比べるか」という
新しい問いが生じる。方針案:

- 既定は**代表 regime 同士の比較** (代表 = 実効日数最大。または基準日を
  CLI/Web で指定)。他 regime の存在は第1部とカレンダービューに明示
  (「このフィードには期間が2つあります: 〜3/31 / 4/1〜」)。
- フィード内の期間差分 (改正前後が1ファイルに同梱されている場合の
  「その改正の中身」) は、多世代タイムライン (detection.md §7 未実装項) と
  同じ問題構造 — 将来そこに合流させる。
- 説明台帳は不変: L0 は全行を列挙し続ける。変わるのは L1/L2 の便数集計と
  レポート表示のみ。JSON スキーマへの影響は day_totals 等の集計系に限定
  (スキーマ変更が要るかは実装設計時に判断)。

## 5. 実装する場合の段階 (素案 — 着手判断は別途)

1. **SD1: 実効運行日集合の導入と day_type 分類の実効日ベース化**
   (IN-8 の修正を含む)。合成テスト: T1 (PRT型)・T6。回帰: 検証5フィード+
   国際7フィードで day_totals 不変 (T1 該当の prt のみ改善) を確認
2. **SD2: 特定日の具体日付表示** (viewer の表示強化 + bundle に日付集合)
3. **SD3: regime 分割の検出と第1部注記 + 便数集計の regime 対応** (T2/T3 解消)
4. **SD4: 運行日カレンダービュー** (第1部に新旧並置)

順序の理由: SD1 は単独で PRT の誤りを直し回帰リスクが最小。SD3 は集計の
意味が変わるため、SD4 (見える化) と同時期に入れて利用者が検証できる形にする。

## 6. 未解決・要調査

- 亀山市「平日土曜/春・夏/秋・冬」のような **day_type とアクティブ期間の
  直積が複雑なケース**で regime 分割が人の認識に合う粒度になるか (実データで
  ドライラン要)。regime が細かく割れすぎる場合の統合規則 (最小 regime 日数 =
  config 閾値) を検討。
- 学期/休校ダイヤ (スクールカレンダー連動) は T3 の亜種のはず — 国際
  フィード (北米の school tripper) で実例を探す。
- 「同じ日に同じ便が2つの service から重複して走る」真の重複 (データ不備) の
  検出は本メモの範囲外だが、実効日集合があれば「同日重複」として機械検出
  できる — 将来の検品機能の種として記録。
- 三重交通そのものは現行・過去世代とも綺麗な3 service 構成だった (ユーザーの
  記憶の「倍増」は伊勢市おかげバス型の別事業者か、リポジトリ外データの可能性)。
