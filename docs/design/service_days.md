# 運行日モデルの精緻化 (検討メモ)

状態: **検討 v0.2 (2026-07-23 起草、同日改訂。実装未着手 — コードを書く前の設計検討)**。
発端: PRT (ピッツバーグ) の土曜便が見かけ上 38→76 便に倍増した問題
(docs/verification/intl_feeds.md IN-8) と、ユーザー指摘
「同種の問題は日本のデータにもある。運行日情報をもっと詳しく提示する UI が
あっていい。diff 以前に、1つのデータを正確に表現する必要がある」。
v0.2: ユーザーの概念提起 (日付↔service_id の 1:n、service_id 組み合わせ =
運行パターン、既存ラベルの堅持) を受けて中核モデルを4層に再構成し、
三重交通桑名営業所の日次更新の定点観測を追加。

関連: docs/design/day_types.md (M10 現行仕様)、detection.md §7 (多世代タイムライン)。

## 1. 何が起きたか — service_id 解釈の実例類型

本ツールはこれまで「運行日」を意図的に単純化してきた (day_type 分類 +
M10 の dow_XXXXXXX 一級化。分類の入力は calendar.txt の曜日フラグが優先)。
PRT の問題を受けて gtfs-data.jp 三重県全39フィード×直近4世代 (約150スナップ
ショット) を走査したところ、**「同一曜日パターンの service が複数ある」
スナップショットが24件**見つかった。実例で類型化する:

| 型 | 内容 | 実例 (確認済み) | 現行実装での症状 |
|---|---|---|---|
| T1 祝日専用 | 曜日フラグ+calendar_dates 大量削除で「祝日1日だけ」の service を表現 (米国流) | PRT: service 1 = 土曜フラグだが実効 7/4 のみ。service 4 = 月曜フラグだがレイバーデーのみ | 土曜 38→76 の見かけ倍増、偽の「月曜」列 (IN-8) |
| T2 世代同梱 | 改正前後のダイヤを期間で分けて1ファイルに同梱 | 三重交通桑名 prev_19 (§1.5)、伊勢市おかげバス prev_2: `7_1_20240717` (〜2025-03-31) と `7_1_20250401` (2025-04-01〜) が各50便。菰野町かもしか号 prev_2、津エアポートライン「ドッグ期間」も同型 | **全曜日タイプが倍増** (桑名: 平日 557+553=1110便に見える) |
| T3 季節分割 | 同一曜日フラグ+期間 (または大量削除) で夏/冬ダイヤを分ける | 鳥羽市定期船「夏季/冬季」(全日フラグ+削除244/427日で季節を彫り出す)、鳥羽市かもめバス「全日・夏/全日・冬」、亀山市「春・夏/秋・冬」 | 季節限定便が年中走って見える。夏冬で時刻が違えば同時刻帯に2本並ぶ |
| T4 系統・地区別 | 同一 day_type・同一期間だが系統/地区ごとに service を分ける (データ作成都合) | 伊賀市「平日/青山平日」、亀山市「平日/加太平日」、熊野市「全日/紀和・本宮線」、四日市 Navya 実証 (車両別) | **症状なし** — 便が排他なので合算表示が正しい。「同じ平日はまとめる」が人の認識に合う |
| T5 特定日運行 | calendar_dates のみ (または実質的に少数日) の service | 伊勢鉄道「運転日注意(みえ)」(快速みえの運転日)、既存 irregular 全般 | M10 で分類は対応済みだが、**表示が「特定日」の一言で具体日が分からない** (ユーザー指摘) |
| T6 残骸 | 期限切れ期間の service や calendar_dates が残存 | PRT の W/U/S (期間が丸ごとフィード期間外)、三重交通中勢 current の 2023年 calendar_dates 例外149行 | 実効日ゼロなら inactive (M10) で概ね無害だが、期間外 service に便が載ると誤算入しうる |

要点: **T1/T2/T3 は「実効運行日が互いに素な service を同じ曜日タイプとして
合算する」ことによる同一機序の誤り**。T4 は合算が正解。つまり service_id の
字面や曜日フラグではなく、**「ある日に何が走るか」**が分類・集計の正しい
基礎になる。

## 1.5 定点観測: 日次機械更新と「同居期間」(三重交通桑名営業所、2026-07-23)

bus-vision 系フィードの世代運用をリポジトリ API で観測した
(https://gtfs-data.jp/search?pref=三重県&target_feed=mie-kotsu*sancokuwana):

- **毎日1世代**が機械的に発行され、`to_date` が1日ずつ伸びる (約100日先の
  ローリング窓)。service_id は `1_1_20260601` のように**改正日サフィックス**付き。
- 7/11 改正に対し、**7/4 作成分から新旧同居が始まる** (`*_20260601` 期間
  6/1〜7/10 + `*_20260711` 期間 7/11〜10/12 の6 service)。7/11 作成分で
  旧世代 service が消え、新版のみ (3 service) に戻る。
- つまり**改正のたびに約1週間、同居版が「現行」として流通する**。Web 版の
  利用者が同居スナップショットを比較対象に選ぶ確率は高い (T2 は稀な変則
  ではなく、日本の主要配信系の定常動作)。
- 祝日の扱い: calendar_dates で 7/20・8/11 (海の日・山の日) に
  「平日 service を削除 + 日曜 service を追加」— **祝日は「日曜ダイヤへの
  置き換え」としてデータ自身が語っている**。しかも同居版では新世代側にのみ
  例外が付く (旧世代の窓 6/1〜7/10 に祝日がないため)。

この観測は §2 のモデルにとって決定的: (a) 同居は regime 分割で自然に解ける
構造 (期間が綺麗に互いに素)、(b) 祝日カレンダーを外部知識として持たなくても
「この月曜は日曜と同じものが走る」という**振替として機械検出できる**。

## 2. 中核モデル: 日付→運行パターン→ラベルの4層

ユーザー提起の概念 (日付と service_id は 1:n。ある日にアクティブな
service_id の組み合わせの種類 = 有り得る運行パターン) を土台に、4層で定義する。
全層が GTFS の中身だけから決定的に計算できる (国の祝日等の外部知識は不使用 —
現行方針の堅持)。

- **層0 実効運行日**: 各 service の実効運行日集合 = (calendar 期間 × 曜日
  フラグ) − calendar_dates 削除 + 追加、をフィード有効期間で切る。
  service_id はエンコーディング都合であり (ID は弱い事前 — M8/M9 と同思想)、
  以後この日付集合だけを信じる。
- **層1 日→運行内容**: 各日付 d に対し、アクティブな service 集合 S(d)
  (0個以上 — 1:n の明示化) と、それが載せる**便集合 T(d)** を求める。
  意味の単位は S(d) でなく T(d) — T4 (系統別に service を分けたデータ) は
  S(d) が常に一緒に現れるため T(d) では区別が消え、自然に「同じ平日」に
  合算される。
- **層2 運行パターン**: T(d) が同一の日付をまとめた同値類 = 運行パターン。
  「毎日+平日+月水金」の calendar なら {月水金}{火木}{土日} の**3パターン**
  になる (「3か4か」問題には便集合の同値で答える — 土と日は走るものが同じ
  なら同じパターン。曜日をどう跨ぐかはラベル (層3) の仕事であって
  パターンの仕事ではない、と役割を分けるのが答え)。
  同値判定は第1に trip_id 集合、第2に**便署名 (内容シグネチャ、M8 の資産)** —
  PRT 型 (祝日用に同内容の便を別 trip_id で複製) は内容同値で「土曜ダイヤと
  同じ」と判定できる。桑名型 (祝日に日曜 service をそのまま追加) は
  trip_id 同値で足りる。
- **層3 ラベルと例外**: パターンの日付フットプリント (曜日分布) を既存の
  語彙 (weekday / saturday / sunday_holiday / weekend / daily / dow_XXXXXXX /
  irregular) に写像する。**語彙は現行のまま増やさない** — 「平日」「土休日」
  「月水金」のラベルが付くこと自体が価値、というユーザー要件をモデルの
  制約条件にする。多数決・許容率は既存 config の延長。

層3 の例外日の扱い (カレンダービューと「特定日」表示の中身):

各 regime (§2.5) 内で、曜日ごとの**基準パターン** (その曜日の過半を占める
パターン) を定め、基準から外れる日付を次の3種に分類する:

1. **振替**: その日のパターンが他の基準パターンと (内容) 同値 —
   「7/20 (月) は日祝ダイヤ」。祝日・振替休日・年末年始の「休日ダイヤで
   運行」はすべてここに落ちる。**祝日を知らずに祝日の振る舞いを説明できる**。
2. **増減**: 基準パターン±少数便 — 「8/13〜8/16 は −12便」(お盆減便等)。
   差分は便単位で提示可能 (どの便が走らないか、まで言える)。
3. **特異**: どれとも同値でない — 日付を列挙して個別ダイヤとして表示。
   0便の日は**運休日**として明示 (年末年始全休等)。

これで「特定日」という抽象語は表示から消え、「8/11: 日祝ダイヤ」
「12/29〜1/3: 運休」のような具体表示になる (ユーザー指摘の解消)。

### 2.5 期間 (regime) 分割

日付軸を「曜日→基準パターンの対応が一定な極大区間」に分割する。
T2 (改正前後の同居)・T3 (夏/冬) はここで「期間A のダイヤ / 期間B のダイヤ」に
分かれ、倍増が消える。桑名 prev_19 なら: 期間A 6/1〜7/10 (平日557便)、
期間B 7/11〜10/12 (平日553便、7/20・8/11 は日祝ダイヤ)。
regime が1つだけの綺麗なフィード (日本の大半) では、この全機構は
**現行の day_type 分類と同じ結果に退化する** — 既存レポートとの互換性は
この退化性質で保証する (回帰テストの主眼)。

閾値 (すべて config): 基準パターンの多数決率、最小 regime 日数
(細かく割れすぎた場合は例外日側に倒す)、増減と特異の境界 (便数差の比率)。

## 3. 運行日カレンダービュー (単一スナップショットの検査 UI)

diff 以前に「1つのデータの運行日構造を正確に見せる」ビュー。層0〜3 の
計算結果をそのまま描く (色弱原則: 記号・線種が第1チャネル、色は補強):

- **年間カレンダーグリッド** (月×日)。各日のセルにラベル記号
  (平/土/休/毎 等の1文字) + regime 番号 (①②…)。振替は▲+参照先
  (「▲休」= 休日ダイヤ)、増減は±、運休は×、特異は◆。
- **帯 (タイムライン)**: (a) feed_info 有効期間、(b) regime 区間、
  (c) calendar_dates 例外の点、(d) gtfs-data.jp 世代の from/to
  (リポジトリ由来の場合)。「feed_info は 12/31 まで、calendar は 10/23 で
  切れる」等のズレも一目で分かる。
- 置き場所: まず**レポート第1部に新旧2枚並べて**掲載。単体検査モード
  (compare を伴わない inspect) は将来の入口として温存。

## 4. diff への波及 (regime が複数あるフィードの比較)

- 既定は**代表 regime 同士の比較** (代表 = 実効日数最大。または基準日を
  CLI/Web で指定)。他 regime の存在は第1部とカレンダービューに明示。
- 桑名の観測から、Web 版では「同居スナップショット×通常スナップショット」の
  比較が定常的に起きる — 代表 regime 比較はこのケースの正しさに直結する。
- 発展 (将来): 同居スナップショット**1つ**から regime A vs B の差分を出せば
  「その改正の中身」になる — 多世代タイムライン (detection.md §7) と同じ
  問題構造なのでそこに合流させる。
- 説明台帳は不変: L0 は全行を列挙し続ける。変わるのは便数集計とレポート
  表示のみ。ラベル語彙も不変 (層3 の制約)。

## 5. 実装する場合の段階 (素案 — 着手判断は別途)

1. **SD1: 層0 (実効運行日) の導入と day_type 分類の実効日ベース化**
   (IN-8 の修正を含む)。合成テスト: T1 (PRT型)・T6。回帰: 検証5フィード+
   国際7フィードで day_totals 不変 (prt のみ改善) を確認
2. **SD2: 層1〜2 (パターン抽出) + 特定日の具体日付表示** (振替/増減/特異/運休)
3. **SD3: 層3 regime 分割と代表 regime 比較 + 第1部注記** (T2/T3 解消。
   桑名 prev_19・伊勢市 prev_2 を検証データに)
4. **SD4: 運行日カレンダービュー** (第1部に新旧並置)

順序の理由: SD1 は単独で PRT の誤りを直し回帰リスク最小。SD3 は集計の意味が
変わるため、SD4 (見える化) と同時期に入れて利用者が検証できる形にする。

## 6. 未解決・要調査

- 亀山市「平日土曜/春・夏/秋・冬」のような **day_type とアクティブ期間の
  直積が複雑なケース**で regime 分割が人の認識に合う粒度になるか (実データで
  ドライラン要)。
- 学校の長期休み連動ダイヤ (学休日) は「振替 or 増減」として現れるはず —
  国内実例 (どこかのコミュニティバス) と北米 school tripper で確認したい。
- 「同じ日に同じ便が2 service から重複して走る」真のデータ不備は、層1 の
  T(d) 構成時に「同日重複」として機械検出できる — 将来の検品機能の種。
- 便署名の同値をどこまで緩めるか (完全一致のみか、停車列同一+全時刻同一か)。
  M8 の署名定義をそのまま使うのが第一候補。
- 三重交通の現行・過去世代は綺麗な3 service 構成だった。ユーザーの記憶の
  「倍増」は同居期間のスナップショット (§1.5 — 毎改正で約1週間存在) を
  見たものだった可能性が高い。
