# プレゼンテーションモデル設計 (認知単位のレポート再構築)

状態: **要件凍結 (2026-07-06)** — W1 レビュー (07-05) と V1 実例ギャラリーレビュー
(07-05〜06) を経て確定。V2 はこの仕様で実装する。実装中の要件変更は原則行わず、
気づきは「未確定事項・後回し」に積んで V3 レビューで再検討する。
凍結後の改訂は本文を黙って書き換えず、末尾の「凍結後の改訂履歴」に日付・理由付きで
記録した上で該当箇所を更新する (現在: R15 改訂 2026-07-07)。

**スコープ: 路線 (route_group) に紐付く変更の表現に絞る。**
路線に紐付かない変化 (運賃・メタデータ・フィード全体のカレンダー等) の表現は
頭の整理が追いつくまで後回し (現状の検証モード側の表示を維持)。

## 背景: 「台帳の単位」と「説明の単位」のギャップ

ChangeEvent の粒度は説明台帳のため (全 RawDiff を漏れなく紐付ける検証可能な最小単位)
に設計されたものであり、人の認知単位ではない。W1 ビューアは ChangeEvent を
1行=1表示単位で描画しており、時間帯ごとに詳細が分裂する・同一パターン変化の
兄弟イベントがバラバラに並ぶ等の不一致が生じた。

## アーキテクチャ

```
ChangeEvent + accounting      ← 検証の背骨。変更しない
        ↓ 合成・集約 (report/presentation.py — V2 で新設)
プレゼンテーションモデル        ← 認知の単位で組み直した中間表現
        ↓ 描画
viewer (HTML) / markdown
```

### 設計原則

1. **コア不変**: L0/L1/L2・ChangeEvent JSON スキーマ・説明台帳は変更しない。
2. **注釈主義**: 表示に必要な追加分析 (系統の出現消滅の導出、方向グループ、
   時刻表の停留所軸併合等) は新イベントタイプにせず、表示用の導出情報として
   プレゼン層で計算する。ontology は動かさない。
3. **検証への導線は残す**: イベント羅列・RawDiff・explained_ratio は「検証モード」
   に退避するが削除しない。
4. **ルールベースの維持**: 要約文を含め全表示はテンプレート+決定的ルールで生成する
   (出力の予想可能性がルールベースの強み)。
5. **色は常に第2のチャネル** (開発者は色弱): 情報は太字・記号 (▲▼・新/廃)・線種・
   数値を第1チャネルとし、色はそれを補強する用途に限る。色だけで意味を区別しない。

## 路線ページの構成 (4部構成)

### 1. 路線全体像の概説

- 新旧の和集合としての路線像: 停車列と地図 (R2 改 2026-07-07: 単位は **leg =
  時刻表単位・曜日統合**)、便数規模。系統単位の羅列 (「14 運行系統: …」) は
  読み取れないため置かない — 系統の内訳は③本数表が受け持つ (07-07)
- **停車列**: leg ごとに1行 (全パターンの LCS 併合軸を主要停留所で間引き)。
  復路軸が往路軸の完全な鏡像なら「A ⇄ B」1行に畳む (— で結ぶ)。
  非対称往復は2行のまま (→ で結ぶ) になり非対称自体が読める。
  レイアウトはラベル列+バス停列の2カラムグリッド (ラベル列幅は全行共有、
  折り返してもバス停列の縦線が揃う)
- **地図**: leg ごとに1スタイル (色+線デザイン+レーン)。leg 内は他パターンに
  完全包含されない「極大パターン」だけを同色・同レーンで重ね描き — 幹線は
  1本に見え、分岐点から枝が分かれる。区間便は線を増やさず、その端点・分岐点は
  key_stops (tier2) の白抜き強調が受け持つ。凡例は③④と同じ leg ラベル
- 地図は「詳細」として折りたたみ、既定では表示しない (レイアウトの煩雑さ回避)
- 線の区別は色+線デザイン・ラベルで行う (原則5)

### 2. 変化のサマリー (Lev.1〜5 のプレゼンテーションカスケード)

イベントを人間が解釈しやすい話題単位に読み換える。検出カスケードと同型の
「上位が説明したら下位は個別に言わない」吸収規則を持つ:

| レベル | 内容 | 束ね・吸収規則 |
|---|---|---|
| Lev.1 | 路線全体の新設・廃止 | これがあるとき Lev.2〜5 は省略し、①の概要を新設/廃止時点の姿の説明として使う |
| Lev.2 | 運行系統の変化 (系統=パターンクラスタの出現・消滅・実質変更) | identity の新旧クラスタ集合とリンクから導出 (イベントには存在しない → 注釈主義)。地図で「全体の路線が分かった上で変化が分かる」見せ方。実質的な便数が新クラスタへ移動した「区間便化」は Lev.2 に昇格 (閾値 config) |
| Lev.3 | 経由停留所の追加・削除 | 同一の変化内容 (停留所列の差) なら方向・曜日を跨いで1件に束ねる。**影響率 (変化した便/系統の全便) を必ず添える** (「全48便」vs「48便中12便」は別の話)。Lev.2 と同一パターン対から派生するものは Lev.2 に吸収。地図表現は Lev.2 と同じ |
| Lev.4 | 増減便 | 曜日単位で集計:「平日: 純減2 (増3・減5)」。**増減の定義 = 便対応の残差 (便ラベル added / removed) の件数** (R14 改 07-10。ヘッダ・④の新/廃マーク・ダイジェストと構造的に一致。対応が成立した時刻変更 = シフトは matched なので自然に除外される。ビン別の内訳は③が担う) |
| Lev.5 | それ以外 | 所要時間の微調整・時刻シフト等を件数の一行に集約 (「時刻の微調整 N 便」)。詳細は④の差分表示が受ける |

網羅性は④ (全時刻の差分表示) と検証モードが担保するため、サマリーは
大きな変化・特徴的な変化に絞ってよい (W1 までの「全イベント列挙」からの転換)。

### 3. 時間帯別本数 (旧→新) の表

- 行の入れ子: **方向グループ → 曜日 → 系統** (集計行が先、内訳が後)
- 変更なし行も表示 (区別付き)
- セルは数値 (旧→新) + 増減記号 (▲▼) を第1チャネルとし、色濃度で補強 (原則5)

### 4. 実際の変化前後の時刻表

- 出版時刻表形式: **1列=1便** (始点から終点まで全停留所の時刻)、左=朝 → 右=夜、
  運行日 (曜日) ごとに1テーブル
- **旧 / 新 / 差分 の3表示切替**。差分表示が本命: 変更セルは太字+記号、
  廃止便の列は取り消し線+「廃」、新設便の列は下線+「新」(原則5)
- 新旧の便の対応付けは trip_delta (exact / modified / removed / added) を利用
- **行軸 (停留所軸) の統一**: 途中止まり・経由違いの便が混在するため、全パターンの
  停車列を LCS ベースで1本の順序に併合する (通過しない停留所は「—」)。
  V2 最大の設計ポイント

## 表示規約

- **方向グループ**: 単純な上下 (direction_id) に依存しない。停留所集合の Jaccard が
  閾値以上の系統ペアについて、**共有停留所の相対順序の整合度**で判定する
  (R15 改訂 2026-07-07):
  - 順序整合度 ≤ 逆転側閾値 → 往復として束ねる (端点の完全逆転は**要求しない** —
    実データでは上下便の端点が操車場・営業所分だけずれるのが普通)
  - 順序整合度 ≥ 同方向側閾値 → 同方向 (区間便・経由違い) として同じ leg に束ねる
  - 中間域は束ねない (保守的)。閾値は config `[presentation]`
  束ねの範囲は route_group ページ内。leg (forward/reverse) の割当も代表系統との
  順序整合度で行う。leg のラベルは「起点 → 終点」形式で、④時刻表の表題と
  ③本数表の方向行に共通に使う (方向グループの「A ⇄ B」との対応が読み取れる。
  「◯◯方面」形式は 07-07 に廃止)
- **曜日の固定順**: 平日 → 土曜 → 日祝 → 土日 → 毎日 → 特定日。
  同一レポート内で常に同順
- 用語: 「運行系統」をレポートの表示語とする (事業者間の用語揺れはラベルの問題と
  割り切る)

## 凍結後の改訂履歴

- **R14 改訂 (2026-07-10, 朝日町 prev_6→current レビュー)**: Lev.4 (増減便) の
  定義を「ビン別本数差分の符号別合計」から「**便ラベル (R19) の added / removed
  件数**」へ変更。旧定義は M8 以前 (便対応が同一 trip_id 頼みで不確か) の
  頑健策で、対応付けが内容主導になった後は (a) 同一時間帯の本当の廃止+新設を
  相殺し (大家庄 新2廃1 → 増1減0)、(b) 時間帯を跨ぐ時刻変更を増減として
  二重計上し (藤塚線 新1 → 増2減1)、④の 新/廃 マークと矛盾した。
  ラベル由来なら net = 新旧便数差 = added−removed でヘッダ・④・ダイジェストと
  構造的に一致する。ビン別の内訳は③時間帯別本数が引き続き担う (シフトの
  時間帯移動は③で読める)。あわせて Lev.3 (R13 影響率) の系統解決を
  ④と同じ cluster 経由に統一 (旧: 代表パターンの停車列完全一致 — 統合 family で
  解決に失敗し分母がページ総便数を超えた。南保線 3/6 の実例)。

- **R16/R18 改訂 (2026-07-10, M10 day_type 精密化)**: 曜日固定順を
  平日→土曜→日祝→土日→毎日→**曜日指定 (dow_*、ビット順)**→特定日→
  **運行日なし** に拡張。曜日指定タブは値から「月・水曜」等を生成し、
  曜日集合を包含する day_type が同一世代で共存する場合は「(平日に追加)」を
  注記 (増便型と運行日変更を区別)。特定日・運行日なし service の内訳
  (日数・期間・置き換え判定) は第1部に表示。設計: docs/design/day_types.md

- **R1 改訂 (2026-07-09, M9 route identity v2)**: 章立ての単位は route_group の
  まま、**世代間で対応した (改称・統合・分割) 路線はページを新世代側に統合**する。
  旧 family は identity の対応成分に従い新側 group のページへ写り、ページ冒頭に
  「旧名称: …」を注記。対応先の無い廃止/新設ページには「内容が類似する路線

  (新設/廃止扱いの便数比率 — 対応付け品質の煙感知器) を追加。

- **R19 追加 (2026-07-09, ダイジェスト表示のレビュー)**: **便ラベル**と
  **変化ダイジェスト**。対応付いた各便対を代表1ラベルに分類する
  (変更なし / 微調整 / 時刻変更 / 区間延長 / 区間短縮 / 経由変更 + 新規 / 廃止)。
  優先順位は「経路系 > 時刻系」— 経路の変化はほぼ必ず時刻の変化を伴うため
  代表1つで情報が落ちず、集約時の二重計上も避けられる。停車列の分類は B群
  イベントと同じ classify_sequence_changes を共有 (検出と表示の一致)。
  微調整/時刻変更の境界は config retime_minor_max_min (既定5分)。
  このラベルを土台に:
  (a) 路線の折りたたみヘッダ = **曜日別便数 (旧→新、合計は出さない)** +
  質的チップ (◆新設/廃止・◆系統再編・○経由変更・＊ダイヤ変更・・微調整)。
  (b) ④時刻表の折りたたみ行 = 旧→新便数 + ラベル別件数。
  (c) ②冒頭に**一言ダイジェスト** (優先順: 路線新廃 > 系統再編 > 経路系 >
  便数増減 > 時刻系、上位3件。bundle には言語中立の構造化事実で持ち、
  文章化は i18n)。あわせて②を圧縮: Lev.3 の折りたたみ行は停留所列挙をやめ
  規模のみ (詳細は展開時)、Lev.4 (R14) の**表示**は曜日単位に集約
  (定義・方向別内訳は③が担う)、Lev.5 は微調整と閾値超を分けて数える。
  時刻表列頭の記号は現行 (＊・新・廃) を維持 (シンプルさ優先の判断)。

- **R17 改訂 (2026-07-08, 徳島バスのレビュー)**: 時刻表の**分冊 (sheet)**。
  leg 内に経由の大きく異なる系統・巡回順の違う循環変種が同居すると、停留所軸が
  ほぼ最適 (最短共通超列) でも便の運行範囲が細切れになり読めない。対処は

  分冊ゼロ)。leg (時刻表単位) は①③の単位として不変 — 分冊は④の描画上の
  別葉で、便数合計は③の leg 行と一致する。閾値2つは config [presentation]。

  R16 固定順で並べ、既定は先頭。ラベルは「平日 16便→12便▼」形式

  「章立ての最上位は路線」の決定は覆さない — 運行日はページ内のビューフィルタ。
- **R3 改訂 (2026-07-07, V3 レビュー)**: 本数表の行の入れ子に「方向 (leg) 集計行」を
  追加 (方向グループ→曜日→**方向**→系統)。leg 行のラベルと便数は④時刻表の表題・
  列数と一致し、③と④の対応が読めるようにする。あわせて leg ラベルを
  「◯◯方面」から「起点 → 終点」形式に変更 (dg ラベル「A ⇄ B」との整合)。
  V2.1 で確立した階層 (方向グループ → leg → 系統) を表示ラベルでも一貫させる趣旨。
  方向が1つのグループ (one_way/loop) では leg 行は集計行と重複するため出さない。

- **R15 改訂 (2026-07-07, roadmap V2.1)**: 方向グループの束ね条件を
  「端点完全逆転/完全一致」から「共有停留所の順序整合度」に一般化。
  経緯: V3 レビューで発覚した地図の線重なり (豊前岩屋線ほか) をビューアの
  レーン割当で回避したが、根本原因はモデル側の完全一致要求だった。7フィード実測で
  6/7 フィードに系統的な取りこぼし (順序整合度は 0.00/1.00 に二極化・中間域 0件)
  を確認し、規則の粗さと判断。系統 (パターンクラスタ)・コア・説明台帳は不変。
