# プレゼンテーションモデル設計 (認知単位のレポート再構築)

状態: 方針確定・要件収集中 (2026-07-05 の W1 レビューから起こした文書)。
マイルストーンは roadmap.md V1〜V3。V1 で要件を追記し、V2 実装前に仕様を凍結する。

## 背景: 「会計の単位」と「説明の単位」のギャップ

W1 の HTML ビューアは ChangeEvent を1行=1表示単位として描画しており、
**内部データ表現がそのまま画面になっている**。ChangeEvent の粒度は説明会計のため
(全 RawDiff を漏れなく紐付ける検証可能な最小単位) に設計されたものであり、
人の認知単位ではない。典型的な不一致:

- 同じ (路線, 方向, 曜日) の時刻表について、時間帯ごとに別イベント → 同じ時刻表が
  複数の「詳細」に重複して現れる
- 同一の停車パターン変化から派生した兄弟イベント (区間延長 + 経由追加 + 経由削除)
  がバラバラの行になる (例: 富山港線フィーダーバス)
- RawDiff の羅列は検証には必要だが、読者への説明としては機能しない

## アーキテクチャ: 第3の層を明示的に設ける

```
ChangeEvent + accounting      ← 検証の背骨。変更しない
        ↓ 合成・集約 (report/presentation.py — V2 で新設)
プレゼンテーションモデル        ← 認知の単位で組み直した中間表現
        ↓ 描画
viewer (HTML) / markdown
```

### 設計原則

1. **コア不変**: L0/L1/L2・ChangeEvent JSON スキーマ・説明会計は変更しない。
   表示の試行錯誤はすべてプレゼン層で吸収する (手戻り防止の要)。
2. **注釈主義**: 表示に必要な追加分析 (例: 時刻シフトの「一様成分+区間残差」分解、
   系統別の便数重み) は新イベントタイプにせず、**表示用注釈**として Python 側で
   計算しバンドルに同梱する。ontology は動かさない。
3. **検証への導線は残す**: イベント羅列・RawDiff・explained_ratio は「検証モード」
   に退避するが、削除しない。読者向けの説明と、結果の検証可能性は別のニーズ。
4. プレゼンテーションモデルも JSON (バンドル内) とし、viewer / markdown の
   両レンダラが同じモデルを消費できる形を目指す (markdown の追随は任意)。

## 認知単位 (画面の設計骨格)

路線 (route_group) ページを次の4段で構成する:

### ① 路線概要 — diff の前に「この路線とは何か」

- 新旧の**和集合**としての路線像: 代表停車列、系統構成 (単純な上下のみか、
  枝分かれ・区間便があるか — family_structure の情報)
- 地図: 全停留所 + 系統別に色分けしたパターン描画。太さ=便数 (努力目標)
- データ源: L1 (identity) の成果物。diff とは独立にバンドルへ全系統分を同梱する

### ② 変化の見取り図 — 時間帯別本数表が主役

- 行順: 方向 → 曜日 → 系統 (集計行が先、内訳が後)。**変更なし行も含める** (区別付き)
- セルの塗り: 増便=暖色 / 減便=寒色、濃度=増減量 → 表だけで全体像が掴める
- **イベント行の羅列 (major/減便/シフト…) は主画面から撤去**。イベントは表を
  塗るための材料に降格する

### ③ 詳細 — 「表の1行 (方向×曜日) につき1画面」

- C群イベントを (路線, 方向, 曜日) キーで統合した時刻表比較画面。
  時間帯ごとに詳細が分裂しない
- 0本の時間帯も省略しない
- シフトの性格が掴める注釈 (例: 「朝の便のみ、区間 A→B で +3分」)。
  一様成分+区間残差の分解を表示用に事前計算する (原則2)

### ④ 停車パターンの変化 — 同じ新旧パターン対は1つの物語

- (系統, 方向, 旧パターン→新パターン) をキーに兄弟イベントを束ね、
  パターン差分は一度だけ見せる。配下に影響便数・時刻への波及をぶら下げる
