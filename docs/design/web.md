# Web 版設計方針

状態: 方針確定 (2026-07-04 の議論)。マイルストーンは roadmap.md W1〜W3。
実装着手前の設計文書であり、W1 の実装知見で改訂してよい (その場合は本書を更新する)。

## 要求 (確定事項)

- Web ベース: gtfs-data.jp から事業者・世代を選択、またはフォームで GTFS zip を
  アップロード → HTML 形式の diff レポートが表示される。
- レポートは現行 Markdown に近い全体概要 + **クリックで具体値へドリルダウン**
  (時刻表の新旧テーブル、路線・停留所の地図、メタデータの具体値)。
  最初から全部は展開しない (混雑回避)。**全ての変化がクリック可能** —
  これは説明会計の構造 (全 RawDiff がイベントの evidence に載る) をそのまま UI 化する。
- 結果には URL が発行され後から参照できる。アップロードされた GTFS も保存する
  (後からの検証のため)。
- フィードバック機能 (「この結果に問題・不十分さがある場合はこちら」)。
  対象イベント ID + 結果 URL 付きで記録し、検証可能にする。
- 日本語主・英語対応 (アーキテクチャを歪めない範囲で)。
- 公開 URL で誰でも使える。ただし想定トラフィックは小さい (GTFS diff の需要は
  ニッチ)。運用コストは開発者個人が負担できる範囲に収める。

## 基本方針: 静的ファースト

**diff 結果 = 静的ファイル一式 (結果バンドル)** とし、閲覧はすべてクライアント側で行う。
サーバ計算はジョブ実行の数秒のみ。コア (compare_snapshots) には手を入れず、
Web は ChangeEvent JSON の新しい消費者として実装する (設計原則 3)。

```
[入力] gtfs-data.jp 選択 | zip アップロード
  → ジョブ (Lambda, 数秒〜10秒): compare_snapshots + バンドル書き出し
  → S3 に結果バンドル保存。URL /r/{id} を発行
  → ビューア (静的 SPA) がバンドルを fetch して描画。閲覧時のサーバ計算ゼロ
```

### 結果バンドル (report/bundle.py が生成)

| ファイル | 内容 |
|---|---|
| events.json | ChangeEventSet (現行の正出力そのまま) |
| rawdiffs.json | RawDiff 全件 (evidence ドリルダウンの実体) |
| geometry.geojson | 停留所座標 (新旧・クラスタ対応付き)、shape ポリライン新旧 |
| timetables.json | 影響のあった family×方向×day_type の新旧時刻表 (ビューアのテーブル描画用) |
| meta.json | ツールバージョン・生成日時・入力の同定情報 |

config_snapshot は events.json に既に含まれるため、バンドルだけで後日の検証・再現
議論が完結する。

## ビューア

- **Svelte + Vite** (軽量・依存少・寿命重視。ドリルダウン中心の
  インタラクティブ UI に十分で、ランタイムが小さく単一ファイル化に向く)。
- **2つの配布モード** (同一コンポーネント):
  1. **単一 HTML ファイル**: ビューア JS/CSS とバンドル JSON を1ファイルに埋め込み
     (vite-plugin-singlefile)。ダウンロード・保存・メール添付で完結する成果物。
     地図タイルだけはオンライン時のみ表示 (それ以外はオフラインで動く)。
  2. **ホスト版**: ビューア本体は共通アセット、/r/{id} のバンドルを fetch。
- **画面構成**: W1 では md 相当の構成をイベント単位で描画したが、W1 レビューにより
  **認知単位のプレゼンテーションモデルへ再構築する** (docs/design/presentation.md、
  roadmap V1〜V3)。以下は W1 時点の記述 (V3 で置き換え):
  初期表示は概要 (サマリ + route_group 章 + 変更のない路線の一覧 + 停留所章 +
  データ検証章、各章は折りたたみ)。
  イベント行をクリック → 詳細パネル: 種別に応じて時刻表 before/after、地図
  (移設・新設廃止のプロット、shape 新旧の重ね描き)、evidence の生値テーブル
  (RawDiff の old→new)。残差 (UNEXPLAINED_RESIDUAL) も同じ流儀でクリック可能。
- **地図**: MapLibre GL JS (OSS・API キー不要)。タイルは地理院タイル (日本域) を
  既定、フォールバックに OpenFreeMap。利用規約の出典表記を守る。
- **i18n**: JSON は言語中立 (英語タイプ ID + 数値) なので言語はラベル層のみ。
  model/event_types.py に display_name_en を追加し、UI 文字列は ja/en の辞書切替。
  英語圏フィードでの動作確認は限定的でよい (要求通り)。

## インフラ (AWS サーバレス)

慣れの利点と従量課金 (低トラフィックでほぼゼロ円) を取り AWS。コアは純 Python の
ため、ジョブ実行部は薄いハンドラのみでロックインは小さい。

| 役割 | サービス | 備考 |
|---|---|---|
| ビューア + 結果配信 | S3 + CloudFront | 恒久 URL の実体は S3 オブジェクト |
| ジョブ実行 | Lambda (コンテナイメージ, 2GB) | 1ジョブ数秒。API Gateway 経由の非同期 (投入→ポーリング) |
| ジョブ状態 / フィードバック | DynamoDB オンデマンド | |
| ログイン | Cognito + Google IdP | 自前パスワードは持たない |
| 通知 | SES (フィードバック時に自分へメール) | |

**コスト試算 (月)**: S3+CloudFront 〜100円 / Lambda ほぼ0円 (無料枠) /
DynamoDB 〜10円 / Cognito 0円 / ドメイン 〜150円 → **合計 数百円〜1,500円**。
規模が読めないことへの備えは従量課金の弾力性と下記ガードで吸収する。

**コスト・濫用ガード**:
- アップロードサイズ上限 (例 100MB)、IP レート制限 (API Gateway / WAF)
- **匿名利用 = 結果30日で自動削除 (S3 ライフサイクル)、Google ログイン = 恒久 URL**
  (保存ポリシーと濫用対策を兼ねる)
- AWS Budgets で月額アラート (例 3,000円)

## 保存とフィードバック (検証可能性)

- 入力 zip (アップロード分) と結果バンドルを結果 ID 配下に保存。
  gtfs-data.jp 入力は org/feed/世代 ID と取得 URL を meta.json に記録
  (リポジトリ側に実体があるため zip は保存しない。消失リスクは許容)。
- フィードバック: 結果ページの各イベント・ページ全体に「問題を報告」→
  {結果 URL, event_id, 自由記述} を DynamoDB に記録 + SES で通知。
  結果が恒久保存されているため、報告は常に再現・検証可能。

## 段階 (詳細は roadmap.md)

- **W1 (サーバ不要)**: バンドル書き出し + ビューアを作り、
  `gtfs-semdiff compare --html` でローカル生成・ブラウザ閲覧。価値の本体
  (デザイン・ドリルダウン・地図・時刻表) はすべてここで完成させる。
- **W2**: 出来たバンドルを S3+CloudFront で手動ホスティング (URL 共有開始)。
- **W3**: ジョブ API・入力 UI・ログイン・保存ポリシー・フィードバック (公開運用)。

インフラ判断は W2 まで実質先送りでき、W1 の成果物はそのまま W2/W3 に載る。
