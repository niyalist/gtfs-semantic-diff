# day_type の語彙の精密化 (M10) — 「特定日」の解体

状態: **承認・実装 (2026-07-09)**。検証ログ: docs/verification/M10_day_types.md。
背景: ユーザーの問題提起「特定日という表示はどういう含意か。パッと全体を掴む
目的に対して分かりにくい」+ 日付↔service が 1:N である (同じ日に複数 service
が有効になりうる。平日ベース+月水だけ増便、のような GTFS が組める) との指摘。

## 1. 実態調査 (2026-07-09、キャッシュ43フィード)

「特定日 (irregular)」に分類される service の悉皆調査:

| 型 | service数 | 便数 | 実例 |
|---|---|---|---|
| **曜日指定運行** (calendar フラグが6分類外) | 65 | 1,099 (83%) | 朝日町「月・火・水・金」「木」、上松町「水曜(通学利用)」、安芸市「火木」、明知鉄道「月以外の毎日」 |
| **短期 calendar_dates** (≤10日) | 16 | 219 | 永井「年末・年始」(5日で111便のフルダイヤ)、「土日祝臨」、花巻「第1・4土曜」 |
| **運行日ゼロ** (全フラグ0・日付なし) | 名古屋4件 | 24 | 「平日深夜」「終車延長」— 休止中の定義枠 |
| 日数多数・曜日多数決落ち | **0** | 0 | (観測されず) |

さらに曜日指定 service と他 service の**関係** (曜日集合の包含):

- **増便/限定型** (より広い service に包含): 53/72 — 平日ベース + 月水の通学便など
- **単独型**: 19/72 — 有田川町「水曜日(金屋)」など地区別曜日運行のコミュバス
- **置き換え型** (短期 calendar_dates): 永井 年末年始 = 平日・土日祝が
  exception_type=2 で当該5日を運休し、専用ダイヤが走る (追加ではなく代替)

結論: 「特定日」の中身は大半が**完全に規則的な曜日指定運行**であり、
「特定日」という語は実態と逆。また day_type は週の分割ではなく**重なり**を持つ
(増便型が多数派) ので、その関係を示さないと誤読される。

## 2. 設計

### 2.1 day_type の語彙 (決定的・世代間安定)

| 値 | 判定 | 表示 (ja) |
|---|---|---|
| weekday / saturday / sunday_holiday / weekend / daily | 従来どおり | 平日 / 土曜 / 日祝 / 土日 / 毎日 |
| **`dow_XXXXXXX`** (X∈{0,1}、月→日の7ビット) | calendar フラグが上記5型のどれでもない | ビットから生成「月・水曜」等 |
| irregular | calendar_dates のみで日数 ≤ short_service_max_days、または曜日多数決 < majority | 特定日 |
| **`inactive`** | 全フラグ0かつ追加運行日なし | 運行日なし |

- 曜日フラグ由来なので `dow_*` は世代間で安定 (同じ運用なら同じ値)。
  trip 照合のブロック (group × day_type) も自動的に精密化される
  (従来は月火水金ダイヤと木曜ダイヤが同じ「特定日」プールで照合されていた)。
- 日付ベースだが規則的なもの (第1・4土曜など) は解釈しない —
  irregular のまま日付要約を見せる (過剰適応の回避)。

### 2.2 関係の注記 (プレゼンテーション層、決定的)

1. **増便/限定型**: 路線ページ内で、`dow_*` の曜日集合を包含する day_type の便が
   同居する場合、曜日タブ・集計に「(平日に追加)」を付す。包含が複数あれば
   最小の上位集合 (同点は R16 順)。閾値なし (集合包含と便の有無のみ)。
2. **置き換え型**: 特定日 (irregular) / 運行日なし (inactive) の service は
   第1部 (フィード全体) に内訳を出す: service 名・日数・期間・便数と、
   「その全日付を他 service が exception_type=2 で運休しているか」
   (真なら「期間中は通常ダイヤ運休」= 置き換え、偽なら「通常ダイヤに追加」)。
3. R16 の曜日固定順は 平日→土曜→日祝→土日→毎日→**曜日指定 (dow)**→特定日→
   **運行日なし** に拡張 (dow 同士はビット列で決定的に整列)。

### 2.3 やらないこと

- 日単位の厳密展開 (ある日の全便合成)。「パッと掴む」層は day_type 抽象 +
  関係注記が担い、厳密な差分は説明会計が保証する、という役割分担を維持
- 祝日カレンダーの参照 (既定方針の維持、detection.md §0)
- day_type 跨ぎの trip 照合 (平日→月水 への便付け替えは減便+増便に見える)。
  包含関係にある day_type のブロック統合は自然な拡張だが、実害の観測を
  待ってから (M8 以来の既知の制限として明記)
