# レポート配信の再設計 (RD トラック) — 技術検討 (2026-07-24 draft)

状態: **RD1a・RD1b 実装済み (2026-07-24)。RD2 以降は未着手**。発端: IN-3 後半
(prt の HTML 478MB)、国内でも都バス級で単一 HTML が重い、地図リッチ化の構想、
AI への出力経路。

**RD1b 実装 (2026-07-24)**: アプリ HTML とデータ JSON の分離配信。
- HTML の URL 体系は不変 (`r/{pair}.html` / `r/{pair}/v/{版}.html`)。HTML は
  アプリ (~1MB) + `{"$data_url": "/r/{pair}/v/{版}.json"}` の埋め込みのみ。
  ビューア (main.js) が起動時に fetch (読み込み中表示つき)。単一ファイル版・
  旧版 HTML は埋め込み JSON をそのまま使う (後方互換)
- data URL は**サイト相対の絶対パス** — 入口 (最新版コピー) と版ページの
  両方から同じ HTML が参照されるため。入口は常に最新版のデータを指す
- データ JSON は生成時 gzip + `Content-Encoding: gzip` で S3 格納
  (CloudFront の 10MB 自動圧縮制限の回避)。版データは immutable キャッシュ
- アップロード由来 (`r/anon|r/u/{job}.html`) も同様に `{job}.json` を並置。
  履歴削除は .html と .json を対で削除
- CLI `--html-dir DIR` (index.html + data.json、ローカル http 配信用) を追加。
  テスト: test_write_html_split。本番の転送量実測はデプロイ後に記録

**RD1a 実装結果 (2026-07-24)**: core バンドル (`build_bundle(core=True)`) +
CLI `--html-lite` + Lambda worker の core 化 + ビューア両対応 (coreMode 検出)。
HTML 実測: 臨港 65→**12MB**、名古屋 137→**40MB**、prt 478→**45MB**。
名古屋 core の内訳: presentation 22.3MB (表示本体) / events 約10MB (うち
evidence サンプル。上限は config `evidence_sample_max=20` — 50 だと
サンプルだけで 21MB になり 20 に調整) / file_diffs 0.3MB。検証モード行 →
レポート項目のクリックジャンプも実装 (§3.2)。合成テスト
test_core_bundle (explainer 先勝ち規則・件数保存・accounting 不変)。

## 1. 現状の仕組み (実測)

### 1.1 単一 HTML の構造

```
report.html (自己完結・単一ファイル)
├─ ビューア App (Svelte 4 コンパイル済み + MapLibre GL) …… 約 1.0MB
├─ <script type="application/json"> に バンドル JSON 全量を埋め込み
└─ OGP メタ (題名・説明は生成時に焼き込み)
```

- サーバーは HTML を組み立てない。**ブラウザが起動時に JSON を丸ごと
  `JSON.parse` し、以後は全てクライアント側で動的描画** (SPA)。
- 起動時に `buildIndex()` (viewer/src/lib/data.js) が rawdiff 全件の
  Map (`rawdiffById`・`rawdiffsByFile`) 等を構築 — ブラウザ RAM は
  「JSON テキスト + パース済みオブジェクト + 索引 Map」の合算で、
  HTML サイズの数倍に達する。
- 地図: MapLibre GL (ベクター地図エンジン、同梱済み) + 地理院ラスタタイル。

### 1.2 バンドル内訳の実測 (2026-07-24、2026.7.24.2 相当)

| キー | 臨港 | 名古屋市バス | prt (ピッツバーグ) |
|---|---|---|---|
| rawdiffs | 52.2MB (77%) | 94.4MB (69%) | 403.3MB (81%) |
| events | 7.6MB | 13.5MB | 62.7MB |
| presentation | 6.6MB | 22.3MB | 27.9MB |
| timetables | 1.0MB | 5.8MB | 2.7MB |
| geometry | 0.1MB | 0.5MB | 3.1MB |
| **合計 (JSON)** | **67.5MB** | **136.5MB** | **499.7MB** |
| 参考: gzip 後合計 | ~4.5MB | ~8.0MB | ~35MB |

観測:
1. **rawdiffs が 7〜8割**。ビューアでの用途は2箇所のみ —
   (a) イベント evidence クリック → 生値表示、(b) 検証モードのファイル別
   生差分ブラウザ。どちらもドリルダウン時にしか要らない。
2. **events の大半は evidence ID 配列** (prt: 4,221 イベントで 62.7MB —
   数百万個の "rawdiff_XXXXXX" 文字列)。evidence を抜くと
   名古屋 13.5→2.0MB (gzip 0.08MB)、prt 62.7→3.1MB (gzip 0.12MB) (実測)。
3. **gzip が 10〜20 倍効く** (ID・構造の反復)。ところが CloudFront の
   自動圧縮は **10MB 超のオブジェクトに効かない**ため、現状の 55MB 級
   HTML は非圧縮のまま配信されている。
4. presentation (認知単位の表示データ) は名古屋で 22.3MB — 初期表示に
   必要な本体だが gzip で 1.9MB。

## 2. 要求 (2026-07-24 ユーザー)

- R-a: 現状の View の情報量は維持する
- R-b: 地図をリッチにする余地 (ブラウザ負担はむしろ軽く)
- R-c: 生データそのものは別途ダウンロードできれば良い
- R-d: 複数ページ分割も選択肢
- R-e: (先の話) レポート相当の情報を AI に渡す経路 —
  告知文生成・インパクト質的評価・可読性の高い形式化差分文書

## 3. 設計案: 「アプリ + 階層化データ」分離 (単一ファイルモードは残す)

### 3.1 前提の発見: ビューアは rawdiffs を数百行しか表示していない

- evidence 表 (EvidenceTable) は **表示上限 300 行** (`LIMIT = 300`、
  超過は「残り N 件」表記のみ)
- 検証モードのファイル別ブラウザ (FileDiffBrowser) も種類別に表示上限つき

つまり現 UI が rawdiffs から実際に見せている情報は高々数百〜数千行で、
数百万件の全量はブラウザ内で一度も描画されない dead weight。
**「Web 上での行レベル表示」は少量サンプルで実質完全に代替できる。**

### 3.2 データの3階層

| 階層 | 内容 | 名古屋実測 (gzip) | 取得タイミング |
|---|---|---|---|
| T1 core | presentation + events (evidence 抜き) + **evidence 要約** (イベント毎: ファイル別件数 + 表示用サンプル行) + **検証モード用サンプル** ((file, kind) 毎の先頭 N 行 + 件数) + timetables + geometry + catalog + meta | 約 3〜4MB | 初期ロード (これで完結) |
| T3 raw | rawdiffs 全件 / events JSON (安定 IF そのまま) を .json.gz で並置 | 名古屋 4.2MB / prt 22MB | ダウンロードボタンのみ (Web では閲覧しない) |

- **rawdiffs の行レベル全量は Web ビューアに持たせない** (2026-07-24 方針)。
  evidence 表示はサンプル行 (上限は現行の LIMIT と整合させる) + 「全 N 件中
  M 件を表示」に置き換える。台帳の信頼性は数値 (explained_ratio・ファイル別
  残差件数 = accounting、これは T1 で不変) が担い、行レベルの完全監査は
  T3 ダウンロード + ローカル (CLI 単一ファイル HTML は従来どおり全量同梱)
  で行う。
- **ダウンロード導線は検証モード側にのみ置く** (2026-07-24 ユーザー指示)。
  レポートモード (第1〜4部) は分かりやすさ優先で現状の見た目を保ち、
  DL ボタン等は足さない。
- **検証モード行 → レポート項目の対応付け** (現状の弱点と RD1 での改善):
  現ビューアは各生差分行に「説明イベント名 → 表示先 (例: 3. 系統52)」の
  テキストラベルを出しているが (FileDiffBrowser destLabel)、これは
  events.evidence 全量からのクライアント側逆引きで、クリックジャンプも
  できない。RD1 で evidence を T1 から抜くとこの逆引きは成立しなくなるため、
  **サンプル行に生成時に主説明イベント (台帳 primary_event_of — クライアント
  の先勝ち近似より正確) と表示先を焼き込む**。同時にラベルをクリックで
  当該ページへ飛べるリンクにし、導線の「甘さ」をここで解消する。
- 当初案にあった「T2: シャード分割の遅延取得」(クリック時に該当 ID 範囲の
  シャードだけ fetch — rawdiff ID はファイル順の連番なので O(1) で
  シャード特定でき、全量ダウンロードは不要) は**採らない**。ネットワーク
  機構が増える割に、得られる体験がサンプル同梱と実質同じため。将来
  「Web で全行ブラウズしたい」需要が出たら追加できる (ID 連続性により
  後付けで設計が劣化しない)。
- evidence の分離・要約は**ビューア用バンドルの中の話**であり、コアの
  ChangeEvent JSON (安定インタフェース、設計原則3) は一切変えない。
- 配信: S3 に `.json.gz` を `Content-Encoding: gzip` 付きで置き、
  CloudFront の 10MB 制限を回避。版管理は現行の
  `r/{pair}/v/{版}.html` を `r/{pair}/v/{版}/` (app.html + データ一式)
  に拡張 — 版の不変性・lazy 再生成の仕組みはそのまま。
- CLI は両対応: `--html` (従来どおり自己完結1ファイル、オフライン閲覧・
  メール添付用) / `--html-dir` (分割一式)。

初期ロードの見込み (App 1MB + T1)。rawdiffs 全量を Web から外すため
S3 の版毎ストレージも激減する (prt 478MB → 十数 MB):

| | 現状 (転送) | 分割後 初期転送 | 初期パース量 |
|---|---|---|---|
| 名古屋 | 137MB (非圧縮) | **約 4MB** | 約 30MB (旧 137MB) |
| prt | 500MB | **約 6MB** | 約 35MB (旧 500MB) |

### 3.2 複数ページ分割 (R-d) の評価 → 採らない (deep link で代替)

- 路線毎に HTML を分けると、共通データ (停留所・地図・第1部) が各ページに
  重複し総量は増える。SPA の現行遷移体験 (タブ・曜日切替) も損なう。
- 欲しいのは実質「URL で特定路線を共有できること」— これはハッシュ
  ルーティング (`#/route/系統52` 等) で SPA のまま得られる。
  データ分割 (3.1) + deep link がページ分割の上位互換。

### 3.3 地図リッチ化 (R-b) の余地

- エンジンは既に MapLibre GL。データ分離後は:
  - geometry を T1 から外し表示時 fetch も可 (今は小さいので急がない)
  - 全系統の shape・停留所を出す「路線網モード」は PMTiles (単一ファイルの
    タイルアーカイブを S3 静的配信、Range リクエストで部分取得) が相性良い
  - ベースマップも地理院ベクトルタイル or 自前 PMTiles でスタイル制御可
- いずれも「初期ロードを重くしない」原則をデータ階層化が担保する。

## 4. AI への出力経路 (R-e、将来トラック)

### 4.1 既にあるもの

- **ChangeEvent JSON = 安定インタフェース** (事実・数値の正)。
  設計原則4「LLM の使用はレポートの自然言語化に限定 (事実は JSON から)」
  と当初から整合する構造。
- render_markdown (Markdown レポート)、presentation (認知単位の構造化)。

### 4.2 提案: 「diff digest」出力 (機械可読の要約層)

そのままの events JSON (名古屋 13.5MB) や presentation (22.3MB) は
コンテキスト窓に大きすぎる。**第1部〜第3部の認知単位を保った、
事実だけの要約層** (数十〜数百 KB) を export する:

```
gtfs-semantic-diff compare ... --digest digest.md   (or --digest-json)
```

- 内容: 比較の概要 (データ/条件/結果の3段)・部別サマリ・路線毎の要約行
  (severity、便数増減、切替日、特定日)・重大イベント明細・被覆率。
- 形式: (a) 構造化 Markdown (人間可読兼用・LLM に直接貼れる)、
  (b) JSON (プログラム連携)。両方を同じ素材から生成。
- ユースケース: 告知文生成 / 質的インパクト評価 / 可読差分文書 — いずれも
  「事実は digest、文章化は外部 LLM」の役割分担。ツール本体に LLM は入れない。
- 前提作業: X1 (bundle スキーマ文書化) を digest スキーマの文書化と併合。
- 将来形: MCP サーバー化 (対話的に「この路線の詳細は?」と掘れる)。
  digest → MCP の順に育てる。

## 5. 段階案 (未承認)

- **RD1**: 配信分割 — app + T1 core.json.gz (evidence 要約・サンプル同梱)。
  CLI は --html (単一・全量同梱、従来どおり) / --html-dir (分割) 両対応。
  Web の版管理をディレクトリ版に拡張。
- **RD2**: 生データダウンロード導線 (T3: events.json / rawdiffs.json.gz を
  版と並置)。**DL ボタンは検証モードにのみ置く** — レポートモードは
  現状の見た目を保つ。国際級も Web で閲覧可能に。
- **RD3**: 地図リッチ化 (路線網モード、PMTiles 検討、deep link)。
- **RD4**: AI digest (--digest md/json、X1 と併せてスキーマ文書化)。

依存: RD1 が全ての土台。RD2〜RD4 は RD1 後なら独立に進められる。
