# 運用管理 (admin) の検討 (2026-07-12)

状態: **提案 (判断待ち)**。オープンソースであること・利用規約 (プライバシー) との
整合を制約条件として、「admin ページを作るべきか、AWS 標準機能で足りるか」を整理する。

## 0. 知りたいこと → 既にあるデータ

| 知りたいこと | データの所在 | 現状の欠け |
|---|---|---|
| アカウント作成状況 | UserData テーブル (profile 行: email, created_at, last_seen_at) | 一覧性がない (コンソールで Scan) |
| 利用状況 (実行数・閲覧) | Jobs テーブル (30日 TTL) / CloudFront・Lambda 標準メトリクス / S3 の r/ 一覧 (恒久) | ジョブの**所要時間を記録していない** (running_since のみ) |
| 計算時間・暴走 | Lambda 標準メトリクス (worker の Duration / Errors / メモリは REPORT ログ) | ダッシュボードが未整備 (毎回コンソールで見る) |
| どの差分が求められているか | S3 `r/{pair}/index.json` (feed 情報+版+生成時刻が恒久記録) / Jobs テーブル | 集計ビューがない |
| アップロードされた GTFS | UserData の zip# 行 (表示名・サイズ) / S3 uploads・userzips | 匿名アップロードはメタ記録なし (S3 キーのみ) |
| フィードバック | Feedback テーブル (恒久) | SES メールで届く (一覧は未整備) |

→ **データはほぼ揃っており、欠けは「所要時間の記録」と「見る手段」だけ**。

## 1. 選択肢

### A. AWS 標準機能のみ (CloudWatch ダッシュボード + コンソール)

- CDK で CloudWatch Dashboard を定義 (worker Duration p50/p95・Errors・
  Invocations、API 5xx、CloudFront リクエスト数)。暴走・障害の監視はこれで十分
  (Lambda の Duration/Errors は計測済みの標準メトリクス)。
- 利点: 新しい攻撃面ゼロ、認証を作らない、ダッシュボード定義もコードとして
  公開できる。欠点: 「どのフィードが人気か」「アップロード一覧」など**意味的な
  ビューが作れない** (テーブルを Scan して目で見ることになる)。

### B. 管理 CLI / 定期レポート (サーバレスバッチ)

- `scripts/admin_report.py` (管理者の AWS 認証で DynamoDB/S3 を集計し Markdown
  を出す) や、週次で SES 送信する Lambda。
- 利点: Web 面を増やさない・実装が薄い。欠点: リアルタイム性がない、
  スマホで見られない。

### C. admin Web ページ (/admin + /api/admin/*)

- **認証は既存の Cognito ログインを再利用**し、`ADMIN_EMAILS` (CDK context →
  Lambda 環境変数) の許可リストに載る email だけ許可する。新しい IdP・
  パスワード・別ログイン機構は作らない。Google アカウント紐付けは
  「あなたの Google の email を許可リストに書く」だけで実現できる。
- 読み取り専用。表示するもの:
  1. アカウント: 登録数の推移、一覧 (email・作成日・最終利用)
  2. ジョブ: 直近 N 件 (対象ペア・種別・状態・**所要時間**・エラー)、
     失敗/長時間ジョブの強調 (▲記号)
  3. 人気の比較: 生成済みペアの一覧 (S3 index.json 集計。事業者・版数・生成日)
  4. アップロード: zip の表示名・サイズ・所有者 (**中身は見せない** — §3)
  5. フィードバック一覧
- 欠点: 実装コスト (中)、管理 API という攻撃面の追加 (許可リスト+JWT で防御)。

## 2. オープンソースとの整合

admin のコードが公開リポジトリにあること自体は問題ない (secrets を置かず、
設計で守る — 現に Cognito/S3 の構成はすべて公開済み)。許可リストの email は
cdk.json の context に置く (公開されるのは「管理者のメールアドレス」のみ。
気になるなら SSM パラメータ参照にもできる)。**「隠すことで守る」要素を作らない**
ことが条件で、C 案の設計はそれを満たす。

## 3. プライバシー規約との整合

- 管理者が email・利用記録を閲覧するのは「サービスの運営 (障害対応・不正利用
  対策)」目的であり、研究利用 (匿名化・集計) とは別枠。透明性のため規約 §3 に
  一文追加を推奨: 「運営者は、障害対応・不正利用対策のためこれらの記録を
  閲覧することがあります」。
- **アップロード zip の中身は admin ページに出さない** (メタ情報のみ)。規約の
  「中身は研究に使わない」の精神と揃え、運営上必要な場合 (障害調査) も
  AWS コンソール経由の例外操作に留める。

## 4. 提案: 段階ハイブリッド

**S1 (すぐ・小): 計測の穴埋め + AWS 標準ダッシュボード**
- worker が Jobs テーブルに finished_at / duration_s を記録 (どの案でも必要な下地)
- CDK で CloudWatch Dashboard (worker Duration/Errors、API 5xx、CloudFront)。
  暴走の検知はこれで完成 (アラームを Budgets と同じメールに飛ばすのも一行)

**S2 (本命・中): 読み取り専用の admin ページ (C案)**
- 既存 Cognito + ADMIN_EMAILS 許可リスト。/api/admin/summary 等の GET のみ
- 上記 1〜5 のビュー。実装は web/admin.html (素の JS、入力 UI と同じ流儀) +
  handler に admin 系ハンドラ追加
- 規約 §3 に運営閲覧の一文を追加

**やらないこと**: QuickSight/Athena 等の BI 導入 (規模に対して過剰)、
書き込み系 admin 操作 (削除・BAN 等は当面 AWS コンソールで)。
