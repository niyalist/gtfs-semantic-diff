# 鹿沼市リーバス: fare_rules 200万行と行差分の集約 (2026-07-12)

## 症状

鹿沼市リーバス (kanumacity/ri-bus) prev_2→prev_1 の Web 比較が「終わらない」
(ユーザー報告)。前回の山交バス (docs/perf/diff0_hashed.md) は計算量バグだったが、
今回は**データ規模そのもの**が原因。

## 原因 (ローカル実測で特定)

prev_2 の fare_rules.txt が **2,003,786 行 (128MB)**、fare_attributes.txt が
98,419 行 (prev_1 はそれぞれ 24,552 行 / 5 行)。全面書き換えのため:

| 段階 | 修正前の実測 |
|---|---|
| RawDiff 総数 | **2,155,115 件** |
| diff0 | 18.1s / ピーク RSS 1.79GB |
| パイプライン全体 | 34.5s / **ピーク 3.24GB** ← Lambda 3008MB を超え OOM |
| render_html | **490MB** の HTML (ブラウザで開けない) / ピーク 5.37GB |

Web では worker が OOM 死 → 非同期 Lambda の**既定2回リトライ**で死→再実行を
繰り返し、status API の16分判定を挟んでも「暴走」に見えた。

## 本質対応 (場当たりの上限でなく、粒度の規約の拡張)

1. **行差分の集約 (bulk)** — detection.md §1 に採録。1ファイルの行差分が
   `[diff0].bulk_row_threshold` (既定 100,000) を超えたら、行単位の列挙をやめ
   `rows_removed_bulk` / `rows_added_bulk` / `rows_changed_bulk` **各1件**
   (行数を値に保持) にする。`file_added` を1件と数える既存の粒度の規約と同型で、
   台帳の網羅性 (全差分が evidence か残差に載る) は不変。検出が行粒度の
   evidence を使うコアファイル (`ROW_EVIDENCE_TABLES`) は規模によらず集約しない。
2. **主キー突合のダイジェスト2パス化** — キー→sha1 ダイジェストで差分の所在を
   求め、値は差分行の分だけ取り出す。diff0 のメモリが**ファイル規模でなく
   差分規模に比例**するようになった (集約が発動しない大規模ファイルにも効く)。
3. **worker の非同期リトライ停止** (`retry_attempts=0`) — poison job の
   自動再実行ループを構造的に断つ。今後の未知の失敗モードにも効く。

## 修正後の実測 (同ペア)

| 指標 | 修正前 | 修正後 |
|---|---|---|
| RawDiff 総数 | 2,155,115 | **126,943** (fare_rules は集約2件に) |
| 全パイプライン | 63s | **17.6s** |
| ピーク RSS | 5.37GB | **0.94GB** (Lambda 3008MB に余裕) |
| HTML | 490MB | **26MB** |
| explained_ratio | 1.0000 (60/215万で丸め) | 0.9995 (同じ残差60件が可視化) |

**残差60件 (calendar 3 / stop_times 33 / pass_attributes 18 / pass_rules 6) は
修正前から存在**していたもので、巨大分母に埋もれて 1.0000 に見えていた。
集約による新たな取りこぼしはゼロ (むしろ台帳が正直になった)。
pass_* は既知バックログ (未知ファイルの claiming ルール)、calendar/stop_times の
残差はこのフィード固有の検出ギャップとして残差カタログ育成の入力にする。

回帰: pytest 199件 (+6: bulk 発火・非対象・変更行・ハッシュ系・台帳消費)。
永井・徳島の explained_ratio / イベント数は不変 (bulk は発動しない規模)。
